## Context

pkit's operator promise includes adopters standing up a complete AI working environment **without deep knowledge of the permission subsystem's internals**. Today the flagship composite goal — autonomous agents — requires four expert gestures in the right order: `permissions profile activate autonomous` (intent), `permissions enable` (hook enforcement), `permissions sandbox enable` (OS confinement), a session restart (`sandbox.enabled` is not hot-reloaded), then `permissions probe --live` to prove the floor actually bites. Each gesture is individually principled — [ADR-002](ADR-002-permission-realizer-ownership.md) §64 makes enforcement an explicit opt-in, [ADR-004](ADR-004-autonomy-intent-confinement.md) splits intent from confinement, [ADR-005](ADR-005-permission-profile-surface.md) rejects a profile activation that silently auto-enables — but their *composition* is operator-hostile for the audience pkit exists to serve.

A `critic` + `architect` consult on a one-off composing verb (`engage`) established three things this record builds on. First, **the orthogonality gate's substance is "no silent fusion," not "separate gestures forever"**: ADR-005's rejected alternative fused enforcement into a verb (`activate`) whose stated purpose was narrower, so the operator got more than they asked for; a command whose *named purpose is the composition* is itself the explicit opt-in ADR-002 §64 requires. Second, **a plain verb that returns before its postcondition holds is dishonest**: the mid-sequence restart means no single in-process invocation can complete the engagement, and a command that claims otherwise commits the COR-028 overstatement displaced into the CLI contract. Third, a one-off alias would relitigate this gate from scratch at every future convenience verb. The product owner sanctioned resolving this as a **class** with a recorded contract — the structural-recurrence argument (every adopter reaching any composite goal walks an identical fixed sequence, by construction) standing in for COR-007's recurrence test — rather than as an unprincipled alias. The proposed status is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md).

## Decision

**pkit gains a setup-command class: goal-oriented orchestrators (`pkit <domain> setup <goal>`) that stand up a named composite goal by composing existing accepted verbs — stepwise, resumable, and honest about boundaries — declaring the goal reached only when its verification passes.** Placement: a goal lives in the domain that owns its vocabulary (`pkit permissions setup autonomy`); bare `pkit <domain> setup` lists that domain's goals with one-line glosses. The top-level `pkit setup` spelling is **reserved** for a future genuinely cross-domain composer — "stand up the whole environment" — which, if it lands, composes the domain setups under this same contract (the class recurses; the top level is not squatted by the first domain that needed a goal). The class contract, binding on every instance:

1. **Composition is the named purpose.** A setup command's documented meaning *is* "stand up this whole goal." Typing it is the operator's explicit opt-in to every step (the ADR-002 §64 argument). Silent fusion stays forbidden everywhere else: `profile activate` keeps its nudge-only behaviour; no existing primitive gains side-effects.
2. **Owns nothing.** A setup command composes accepted primitives and introduces no new mutation engine, no new ownership tier, no second writer for any surface (the ADR-005 god-object stays dead). Every effect is some primitive's effect; every switch remains individually inspectable and reversible by its own verb.
3. **Stepwise, resumable, idempotent.** Each run inspects live state, performs the steps that can proceed, reports per-step status (already done / done now / blocked), and converges on re-run. There is no state file; the live system is the checkpoint.
4. **Honest boundaries.** When a step cannot complete in-process (e.g. the session restart for `sandbox.enabled`), the command stops, names the exact manual action, and instructs the operator to re-run after it. The non-hot-reload fact is communicated, never papered over.
5. **Verification is part of the goal.** A goal is *reached* only when its proof passes (for autonomy: `permissions probe --live` showing guardrails REJECTED and the confinement floor holding). The command never declares completion on configuration writes alone; the final resumed run executes the verification and only then reports the goal reached.
6. **Dangerous flags never ride the orchestrator.** Escape hatches (`--dangerously-allow-unconfined`, `--bypass`, …) stay on the primitive that owns them; a setup command is fail-closed-only and refuses to forward them.
7. **Teardown reports residual state loudly.** A goal's down-form (`pkit <domain> setup <goal> down` — a subcommand, not a flag: tearing down is the *opposite* mutation set, and a flag that inverts a command's direction is the mode-flag smell; the subcommand spelling parallels the project's existing `enable`/`disable` pairs) reverses the live switches the goal stood up and enumerates what it deliberately leaves (e.g. "profile 'autonomous' still active in the model but unenforced") — never a bare success that lets the operator believe more was torn down than was (the ADR-004 §49 trap in reverse).

**First instance: `pkit permissions setup autonomy [--profile <name>]`** — composes `profile activate` (default `autonomous`), `enable`, `sandbox enable`, pauses at the restart boundary, and on the post-restart re-run executes `probe --live` and declares the goal. `pkit permissions setup autonomy down` runs `disable` + `sandbox disable` and reports the still-active model per rule 7.

## Rationale

**Why a recorded class instead of a one-off alias.** An alias answers one complaint; the gate-tension it navigates (explicit composition vs. silent fusion) recurs at every future composite goal. Recording the contract once means future instances inherit the settled argument instead of relitigating ADR-002 §64 — the same record-so-it-cannot-be-relitigated move ADR-005 made for the profile surface. The class is also the honest home for the constraints the consult extracted (rules 4–7), which an alias would carry only as folklore.

**Why domain placement with a reserved top level.** Two placements were live: top-level `pkit setup <goal>` (one front door, maximal discoverability) and domain-scoped `pkit <domain> setup <goal>`. Domain placement wins because the top-level spelling describes a *different, larger thing*: a cross-domain "stand up the whole environment" composer. Spending it on the first domain's goal would squat the name the complete-environment story needs, and would also sit confusably beside `pkit init`. Domain placement keeps each goal next to the primitives it composes and the status surfaces that explain it (`permissions overview`, `probe`), while the reserved top level preserves the single-front-door property at the altitude where it is true — the future whole-environment setup composes domain setups under this same contract, so discoverability is layered, not lost. The cost — an operator must know autonomy is a permissions-domain goal — is mitigated by bare `pkit <domain> setup` listing goals and by the eventual top-level composer being the actual innocent-user entry point.

**Why explicit-opt-in-via-verb survives the orthogonality gate.** The gate (ADR-002 §64; ADR-005's rejected auto-enable) protects the operator from getting *more than the typed gesture promised*. A setup command inverts the failure mode: it promises exactly the composition it performs, in its name and its per-step report. The opt-in is not a side-effect — it is the command. ADR-005's `activate` stays nudge-only precisely so the distinction stays sharp.

**Why stepwise-resumable answers the restart objection.** The consult's strongest carrier objection was that a verb returning before its postcondition holds lies about completion. The class dissolves it by contract: a setup command is a *convergent orchestrator*, not a one-shot verb — it reports progress, stops honestly at the boundary, and reserves "goal reached" for the verification pass (rule 5). The restart is clearly communicated as the operator's one manual step, which is the correct division: pkit cannot restart the operator's session, and pretending otherwise would be the overstatement COR-028 forbids.

### Alternatives considered

- **A one-off composing verb (`permissions engage`).** Rejected — relitigates the gate with no contract for the next instance; the consult also showed its naive teardown shape walks into the §49 residual-state trap rule 7 now forecloses.
- **Chaining flags on `profile activate` (`--then-enable --then-confine`).** Rejected — re-fuses enforcement into the activation verb ADR-005 explicitly de-fused; mutating side-effects behind flags is the silent-fusion shape regardless of spelling.
- **A skill/runbook only (no CLI surface).** Rejected as the primary carrier — the audience is operators who should not need to read a runbook to stand up the environment; the CLI is the surface they touch. Retained as complement: the recipe documentation remains, and the skill/doc explains what `setup` automates.
- **Do nothing (document the four steps).** Rejected — defeats the operator-experience purpose the product owner named; the four-step sequence is exactly the deep-knowledge requirement pkit promises to remove.

## Implications

- **ADR-002 §64, ADR-004, ADR-005 are unchanged.** The class operates at the invocation layer; the ownership partitions, the fail-closed invariant, and the activation/enforcement orthogonality all stand. This record is the new-principle ADR the consult prescribed (not a reconciliation note on ADR-005).
- **Build sanction on acceptance.** Accepting this record sanctions building `pkit permissions setup autonomy` (+ `down`) per the seven-rule contract. All five composed primitives (`profile activate`, `enable`, `sandbox enable`, `sandbox disable`, `probe --live`) are built; the sandbox writer (#274 / PR #275) and the probe verifier (#276 / PR #277) must merge before the first instance lands. New CLI surface → version bump per PRJ-002; pure addition, no migration expected (COR-010 check still runs).
- **Primitives remain canonical.** Experts and scripts keep composing `profile activate` / `enable` / `sandbox enable` directly; `setup` adds no capability they lack, only sequence knowledge and verification.
- **`probe` becomes the class's verification substrate.** Rule 5 makes goal-completion claims ride the existing probe command (#276), keeping "configured" and "proven" distinct vocabularies.
- **Future goals ride the same contract.** Candidate later instances (e.g. a project-management bootstrap, an agents-deployment goal) implement the same seven rules; any instance that cannot satisfy them (e.g. needs a new mutation engine) is evidence the goal belongs elsewhere, not grounds to relax the contract.
