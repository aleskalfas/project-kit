## Summary

[ADR-016](ADR-016-capability-contributed-intent-grants.md) let an installed capability ship its own permission *policy* — a grants fragment that denies a privilege to one of its agents — but deliberately kept *"the privilege definition stays central for now"*, recording the move of the definition itself into a capability as **deferred seam CA-1**. A privilege you want to deny still had to be *defined* in the backbone catalog, so a capability that recognises a brand-new class of action couldn't be self-contained: it needed a core-catalog edit an adopter can't make without `pkit sync` overwriting it.

**Decision:** un-defer CA-1. An installed capability may ship its own **privilege-catalog fragment** (`.pkit/capabilities/<cap>/permissions/privilege-catalog.yaml`) that the catalog loader (`load_catalog` in `.pkit/permissions/decide.py`) merges into the central catalog. A capability now ships a privilege **definition and** its deny policy together, self-contained, with no core-catalog edit.

Three constraints make the merge safe — and the first two are load-bearing, not decoration:

- **Additive-only, collision-rejecting.** A fragment may only *add* privilege ids; the loader refuses a fragment whose id collides with a backbone privilege or another capability's. A capability can never overwrite, shadow, or strip an existing definition. To make collisions structurally impossible and attribution intrinsic, a capability-contributed id is **capability-scoped** (`<cap>:<name>`) — which gives the privilege-id half internal structure the COR-019 token grammar must be widened to admit (a clarification, detailed in the Decision).
- **Never a guardrail.** A fragment privilege carrying `guardrail: true` is **rejected** (not silently stripped). A capability may extend the *vocabulary* of recognised actions but may never install a deny that applies to every adopter by default; it shapes only its own agents' reach, through its own deny grant (ADR-016's channel, unchanged).
- **Visible.** `pkit permissions overview` attributes each merged privilege to the capability that contributed it; nothing enters the catalog silently.

Only *installed* capabilities count — the merge walks the component manifest, so an orphan directory contributes nothing. First use: the trip-planning capability ships a scoped privilege recognising raw shell scrapers an agent reaches for (`curl`, `wget`, `python`, `sed`, `awk`) plus a deny grant for its own researcher subagents. The privilege ships *with that capability*, in the adopter tree — not in project-kit.

## Context

The forcing case is the trip-planning capability's parallel-research rollout. Its researcher subagents read external sources as the core of their work, but `WebFetch`/`WebSearch` are unavailable to spawned subagents in the Claude Code harness, so a subagent's reflex is to fall back to `curl` + `python`/`sed` — commands not on the allowlist that, on macOS, can't be sandbox-auto-allowed ([ADR-014](ADR-014-macos-sandbox-platform-stance.md)), so each one prompts the operator (one observed run produced ~170 prompts). The capability already ships the right fix — a prompt-free read command — but *instruction* to use it proved unreliable across parallel runs. The dependable fix is to make ad-hoc scraping *denied* for that agent, not merely discouraged.

A deny references a privilege, and **that privilege has to exist in the catalog the hook loads**. `load_catalog` reads exactly one path — the backbone `.pkit/schemas/privilege-catalog.yaml`, core-owned and refreshed by `pkit sync` — and no existing privilege recognises these raw shell commands. So an adopter can *grant or deny* freely (ADR-016) but cannot *define* the privilege they'd want to deny without editing core, which sync overwrites. That definition gap is exactly the boundary ADR-016 named as CA-1 and deferred until a second consumer appeared. It has appeared.

This decision was reached in an architect review that weighed three rejected landings against this one (below). As project-kit's own architecture record, harness and path specifics are in scope here. The `proposed` status is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md); accept before any implementation cites it.

## Decision

**An installed capability may contribute privilege *definitions* by shipping a `privilege-catalog.yaml` fragment in its own kit-owned `permissions/` directory, which `load_catalog` merges into the central catalog under an additive-only, collision-rejecting, guardrail-forbidding rule, discovered by walking the installed-component manifest, and attributed on the model's read surface.** ADR-016's grants mechanism is untouched; this adds the missing definition half it deferred.

**1. The merge lives in `load_catalog`, which gains a manifest walk it does not have today.** This is *not* a free symmetry with the grants fragment, and the ADR does not pretend it is. The grants walk (`_capability_fragment_grants`) lives in the *model* loader; `load_catalog` today reads a single file and never touches the manifest. The merge must live in `load_catalog` itself — not in `load_model` — because both the hook and the CLI call `load_catalog` directly to learn which commands are recognised, and because `guardrail_denies` runs on whatever catalog `load_catalog` returns. So `load_catalog` grows a manifest-scoped walk (reusing the grants walk's *pattern*, not its code path) and returns the merged catalog. Both readers get the identical merged catalog (the [ADR-002](ADR-002-permission-realizer-ownership.md) same-code invariant is preserved precisely because the merge is on the shared loader, not duplicated downstream).

**2. Additive-only and collision-rejecting — the catalog is a map, so the merge rule is load-bearing.** Grants are an append-only *list*; the catalog is a *map keyed by id*, where a naive merge is last-writer-wins and a colliding id would let a fragment overwrite a backbone privilege's recogniser (or strip a backbone guardrail). The loader therefore **refuses** a fragment whose id already exists — in the backbone catalog or in another capability's fragment — rather than overwriting. To make collision structurally impossible (not merely caught), a capability-contributed id is **capability-scoped** as `<cap>:<name>`. This is **not free against the existing token grammar, and this ADR does not pretend it is**: a grant references a privilege by the [COR-019](../../../.pkit/decisions/core/COR-019-schema-reference-form.md) token `[privilege-catalog:<id>]`, whose id half (the `_TOKEN` matcher in `decide.py` and COR-019's companion `reference_token` pattern) admits a single segment with no embedded colon — so `[privilege-catalog:trip-planning:ad-hoc-scraping]` would not match today. The grammar must be **widened to admit a structured id half**. This is a **COR-019 *clarification*, not an amendment**: the token namespace stays `privilege-catalog`; only the id half gains permitted internal structure to carry the capability scope. The token normaliser (`_privilege_ids`) MUST round-trip the scoped id back to the catalog key exactly — a mis-resolved token would empty the grant's privilege match and the deny would silently not bind (a **fail-open** hazard), so exact resolution is load-bearing. A capability can only ever *add* vocabulary in its own namespace; it can never touch a definition it does not own.

**3. A capability-contributed privilege may never be a guardrail — reject, do not strip.** A backbone privilege may carry `guardrail: true` (a deny that applies to everyone). A fragment privilege may not: the loader **rejects the fragment** if any entry carries `guardrail: true`, rather than silently dropping the flag (silent-strip would, under a hypothetical collision, turn off a backbone safety deny — the reject-and-the-collision-rule together close that). The guardrail check runs *inside* the merge, before `guardrail_denies` ever sees the result, so a forbidden guardrail can never become a global deny. A capability's reach is over its own agents via its own deny grant; it can extend recognised vocabulary but never impose a standing deny on every adopter.

**4. Recogniser overlap is permitted and bounded by ADR-016's grants channel.** A fragment recogniser may match a command another privilege also matches (e.g. `python`). When it does, both privilege ids "hit" and the existing deny-wins composition applies per the *subject's* grants. This does not let a capability narrow a third party's reach: a capability fences behaviour only by *also* shipping a deny grant, and that grant (ADR-016's channel) is what binds — the new definition is inert until a grant references it. The honest bound is therefore: a capability can deny recognised commands **to the subjects its own grants name**; the definition merely supplies the vocabulary, and the grants channel's existing layering (deny-wins, operator-overridable) governs the effect. (See the narrowing-footing note in Implications.)

**5. Same-code invariant (ADR-002 / ADR-003).** `load_catalog`'s new walk and merge run in the propagated `decide.py`, which the PreToolUse hook executes with bare `python3` where the global `pkit` is not importable. The walk MUST use the existing stdlib-safe YAML fallback and the existing manifest read — no `src/project_kit` import, no third-party-only dependency. The fragment reuses the existing privilege-catalog document shape (a map of `<id> → {description, recognize, guardrail?}`), so no new schema and no new parser path are introduced; the loader gains a walk, a merge, a collision check, and a guardrail check — not a new format.

**6. Attribution is mandatory (narrowing-but-reported).** `pkit permissions overview` / `explain` MUST show that a privilege came from a capability and which one. Because the merge is collision-rejecting and ids are capability-scoped, the attributed source and the effective definition can never diverge — the honesty posture [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) requires holds even at the edges.

**7. A denylist is a speed-bump, not a boundary.** A capability-contributed deny shapes a *non-adversarial* agent's reflexive behaviour — it stops a researcher subagent from habitually reaching for `curl`. It is **not** a security boundary: per [ADR-004](ADR-004-autonomy-intent-confinement.md) a tool-call denylist is porous (a `bash -c '…'` wrapper evades the recogniser). The decision is justified on behaviour-shaping value, and claims nothing stronger.

## Rationale

**Why un-defer CA-1 now — on lifetime-coupling, not vocabulary-neutrality.** ADR-016 deferred the definition relocation on the recurrence test ([COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)): one consumer, so a capability-owned catalog fragment would have been speculative generality. A second consumer now exists. The decisive property is **not** that the trip privilege is "non-neutral vocabulary" — `curl` is about as neutral as `git`, and operators legitimately use both. It is **lifetime-coupling**: this privilege exists only to serve one capability's researcher discipline, and it should *live and die with that capability* — present when installed, gone when uninstalled — which a backbone catalog entry structurally cannot provide (a backbone entry persists in every adopter's catalog forever, including adopters who never install the capability). That is the same ownership/lifetime correction ADR-016 made for *grants*, now demonstrated for *definitions* by a real second consumer. The recurrence is met by an instance whose recognition vocabulary genuinely belongs to the capability, not to the core.

**Why additive-only collision-rejection is the real safety crux, not the guardrail rule alone.** The guardrail prohibition is necessary but not sufficient. The catalog is a map; the dangerous vector is a fragment id colliding with a backbone (or peer) id and overwriting its recogniser — which could re-scope or disable an existing privilege (including stripping a backbone guardrail) without any grant. Capability-scoped ids plus a collision-rejecting merge close that structurally: a fragment can only introduce ids in its own namespace and can never overwrite. With that in place, an unused capability privilege changes *no* decision (it is allow-by-default vocabulary referenced by nothing until the capability's own grant names it), so the worst a capability can do is fence its own agents.

**Why the merge belongs in `load_catalog` and why that is not free.** The hook and the CLI both call `load_catalog` to learn recognised commands, and `guardrail_denies` consumes its output; putting the merge anywhere downstream (e.g. in `load_model`) would leave the hook deciding against the *unmerged* catalog and a slipped guardrail unscreened. Living in `load_catalog` keeps both readers byte-identical and lets the guardrail check gate the result before any deny is derived. The cost — `load_catalog` gains a manifest read it lacked — is real and is named here rather than hidden under a "just symmetric to grants" gloss.

### Alternatives considered

- **Author a *neutral* backbone privilege (e.g. `shell-net` for `curl`/`wget`, `shell-scripting` for `python`/`sed`/`awk`) and deny it via ADR-016's existing grants channel — no new mechanism.** This is the strongest alternative: if these commands are neutral backbone vocabulary (and `curl` arguably is), the grants channel already suffices once the backbone recognises them, and CA-1 need not be un-deferred. Rejected on **lifetime-coupling and accretion**: a backbone privilege persists in *every* adopter's catalog forever, including those who never install trip-planning, and sets the precedent that any capability's recognition need is pushed into core — re-importing the homeless-need and orphan-persistence problems ADR-016 solved for grants. It also bakes one capability's carving judgment (the open `shell-web-fetch` vs `shell-scripting` split) into the neutral catalog. The mechanism is justified not because *this* privilege *cannot* be neutral, but because capability-owned recognition vocabulary should live and die with its capability — a property no backbone entry has.
- **Add `ad-hoc-scraping` to the backbone catalog as a capability-specific privilege (the hand-off's original proposal).** Rejected, and more weakly than the above: it has the accretion problem *and* puts an overtly capability-specific id in a neutral core file. The hand-off itself demoted this to a fallback after architect review.
- **A full adopter-side privilege-extension mechanism** — let an *adopter* (not just a capability) define privileges in a project-owned file the loader merges. Deferred, not rejected: a larger design (a new adopter-owned file role with its own precedence and override story against capability and backbone definitions) than the forcing case needs. The capability-fragment channel is the minimal extraction that serves the recurrence; the adopter channel un-defers when an adopter needs a privilege no capability and no backbone ships.
- **Leave it as instruction (no permission change).** Rejected by the forcing case: instructing the subagent to avoid `curl`/`python` proved unreliable across parallel runs.

## Implications

- **Amends ADR-016's definition-locus rule; supersedes none of its mechanism.** ADR-016's grants channel, layering, and safety properties stand unchanged. But ADR-016 stated a general rule — *"the privilege definition stays central"* (Summary) / *"the privilege stays in the backbone catalog"* (point 6) — and this record *changes that rule* to "central unless a capability ships its own, additively." That is a partial amendment of ADR-016's definition-locus principle, not merely consuming a deferred seam. ADR-016 gets a *"definition-locus rule amended / CA-1 consumed by ADR-021"* pointer on point 6 and its CA-1 Implications bullet.
- **New kit-owned file role: `.pkit/capabilities/<cap>/permissions/privilege-catalog.yaml`.** Core-owned (propagated on capability install/sync; adopter edits are overwritten — the [ADR-003](ADR-003-permission-core-code-home.md) propagated-tree property). It reuses the existing privilege-catalog document shape, so it introduces **no new schema** and rides the existing companion — but the companion may need a note that fragment ids are capability-scoped and `guardrail` is forbidden in a fragment.
- **`load_catalog`'s walk + merge + collision check + guardrail reject, and the attribution surface, are implementation work (the Feature under the tracking EPIC), after this ADR is accepted, in the same change.** This ADR records only the decision. The implementation owns: the manifest-scoped catalog-fragment walk; the additive/collision-reject merge; the capability-scoped-id convention; the **COR-019 token-grammar widening** (`_TOKEN` and the companion `reference_token` pattern admit a structured id half) plus the exact-round-trip token resolution in `_privilege_ids`; the `guardrail: true`-in-fragment rejection (before `guardrail_denies`); and the `pkit permissions overview` attribution — with tests covering collision-reject, guardrail-reject, orphan-directory-skip, recogniser-overlap-deny-wins, and scoped-token round-trip (deny binds, not fail-open).
- **Narrowing-footing: a definition is not itself a narrowing.** Unlike ADR-016's deny (which removes a privilege and is auto-applyable per [ADR-008](ADR-008-confinement-allowances.md)'s narrowing class), *adding* an allow-by-default privilege narrows nothing — it changes no decision until a grant references it. Auto-applying a merged definition on install is safe on *that* footing (inert vocabulary), not on borrowed narrowing; the narrowing-but-reported posture applies to the *deny grant* the capability also ships, exactly as ADR-016 already governs.
- **The privilege itself is adopter-side, out of project-kit's scope.** The `ad-hoc-scraping` privilege (and the open question of splitting it into a fetch privilege for `curl`/`wget` vs a scripting privilege for `python`/`sed`/`awk`, and the deny grant naming the researcher subagent) ships *with the trip-planning capability* in the adopter tree. project-kit ships only the mechanism that lets a capability do this.
- **COR-010 / versioning.** A **pure addition** for [COR-010](../../../.pkit/decisions/core/COR-010-resource-lifecycle.md): no rename/removal in a kit-owned tree, no `schema_version` bump (the fragment reuses the existing catalog shape), no breaking CLI change — so **no migration**; the implementation runs `pkit migrations check-diff --include-working-tree` to confirm COVERED. It **is** a surface change for versioning per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md) (a new contribution mechanism plus a new convention adopters follow), so `.pkit/VERSION` bumps in the implementing PR — that bump rides with the implementation, **not** this ADR.
