## Context

[COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md) established how pkit delivers content into *fixed-path adopter-owned* config files: a merge primitive that contributes, never subtracts. The complementary rule is the **no-shared-files invariant** ([COR-001](../../../.pkit/decisions/core/COR-001-content-mechanisms.md) / [COR-017](../../../.pkit/decisions/core/COR-017-capability-pattern.md)): every file has exactly one owner — pkit owns its files outright and never co-edits a file the adopter owns. Today pkit owns no part of any `.gitignore`; `install.py` merely *prints* a recommended snippet, and project-kit's own root `.gitignore` is hand-curated. pkit's standing disposition toward adopter git state is **print, don't mutate**.

A real adoption scenario is unserved: a developer wants to use pkit on a repo whose **other members have not adopted it** — so pkit's footprint (`.pkit/`, the deployed `.claude/skills` + `.claude/agents`) must stay **git-private**: working locally, invisible to the shared tree, forcing nothing on teammates who never chose it. The default — what project-kit itself is — commits pkit so a whole team shares one methodology.

A `critic` + `architect` consult and an audit of the first draft settled the shape. Decisive findings: (1) **markers inside a committed `.gitignore` are out** — COR-002 explicitly *rejected* in-file markers under the no-shared-files invariant; pkit must not co-edit the adopter's shared `.gitignore`. (2) A `.gitignore` cannot hide *its own* directory, so hiding the whole `.pkit/` folder requires a *parent*-level rule; **`.git/info/exclude`** — a per-clone, never-committed ignore file — is the only native channel that can hide `.pkit/` while leaving zero committed trace. (3) `git rm --cached` is a **new git-index-mutation posture** for the backbone and must be deliberate. (4) The footprint spans tiers (core `.pkit/`, adapter `.claude/` deploys, capability state), so it must be **declared per component**, never hardcoded in core. (5) `.claude/settings.json` is co-owned and out of scope. (6) The narrow set of *inherently-local runtime files* (e.g. `sandbox-provenance.yaml`) that one might ignore even in shared mode is small and not the stated need — **deferred** per [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md). Scoping v1 to the private case means **pkit touches no committed `.gitignore` whatsoever** — it writes only the per-clone `.git/info/exclude` — which sidesteps the marker / shared-file question entirely. The product owner directed that going private **always** runs the cleanup, gated by an explanatory confirmation. Proposed status is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md).

## Decision

**pkit's git footprint is adopter-controlled through two visibility modes over a per-component-declared footprint, realized entirely through the per-clone `.git/info/exclude` — no committed `.gitignore` is ever written. `shared` (default) commits pkit and keeps `info/exclude` clear of pkit. `private` writes the whole footprint to `info/exclude` so pkit is invisible to the shared repo, and — because going private means "stop sharing pkit" — always runs a confirm-gated untrack of any footprint files already tracked. No file the adopter owns is ever co-edited.**

1. **The footprint is one declared list per component.** Each installed component (backbone, adapter, capability) declares its footprint in its package metadata: `footprint: [<glob>…]` — the paths that constitute pkit's presence (e.g. backbone `.pkit/`; the claude-code adapter its `.claude/skills`, `.claude/agents`). The mechanism aggregates declarations across installed components. Core owns the mechanism; components contribute their paths (the registration seam the deploy primitives already use). Core never names adapter- or capability-specific paths. No `committed`/`local_only` split — v1 hides the footprint wholesale or not at all.

2. **`shared` (default) — pkit is committed; `info/exclude` is kept clear.** `pkit visibility shared` ensures none of the aggregated footprint is present in `.git/info/exclude` (removing pkit's lines if a prior `private` added them), returning pkit to normal tracked/committable state. It writes **no** `.gitignore` — pkit's files are committed the ordinary way, exactly as project-kit self-hosts today.

3. **`private` — make the whole footprint git-invisible via `.git/info/exclude`.** `pkit visibility private` writes the aggregated footprint (including `.pkit/` itself and the `.claude/` deploys) to `.git/info/exclude`. This is the only native channel that (a) sits in a *parent* context, so it can hide the `.pkit/` directory a nested `.gitignore` cannot hide itself, and (b) is never committed, so privacy leaves no trace in the shared repo. A fresh private install therefore never tracks pkit and needs no cleanup. (Its usual downside — per-clone, lost on re-clone — is a feature here: pkit re-writes it on install, and you *want* it invisible and unshared.) pkit owns a delimited region *within* `info/exclude` (a file git itself owns, not the adopter — markers here co-edit nothing the adopter authored), so it can add and later remove its own lines idempotently without disturbing any hand-added exclude entries.

4. **Going private always runs a confirm-gated untrack.** Because switching to private *means* "remove pkit from the shared repo," `visibility private` always invokes the untrack step: it computes `git ls-files ∩ footprint`, and if any footprint files are already tracked, presents an **explanatory confirmation** — naming that these files will be removed from the shared tree on the next commit, while local working copies are preserved (`git rm --cached` keeps the file on disk) — and on confirmation runs `git rm --cached` on exactly those paths. On a fresh install the set is empty and the step is a silent no-op. `--dry-run` previews both the `info/exclude` write and the untrack set, mutating nothing. Declining leaves the exclude rules written and the files tracked (re-runnable later).

5. **`untrack` is also a standalone, footprint-restricted, precondition-guarded verb.** `pkit visibility untrack [--dry-run]` runs the same confirm-gated action independently. (Implemented as a subcommand under `visibility` rather than a bare top-level verb — it lives in the footprint domain and reads consistently with the noun-first command grammar, while remaining its *own* verb, never silently folded into a `shared`/`private` mode flip; the quarantine the consult asked for is preserved.) It **only ever** touches paths in the declared footprint, never stages a non-pkit file, and preserves working copies. It **refuses when the repo is mid-operation** (merge / rebase / cherry-pick in progress) or when target footprint paths have staged changes, rather than entangle its removal with a pending index. This is the deliberate new git-index-mutation posture, bounded by: footprint-only, confirmation-required, `--dry-run`-previewable, working-copy-preserving, clean-precondition.

6. **`.claude/settings.json` and inherently-local runtime files are out of v1 scope.** `settings.json` is co-owned (harness baseline + pkit's merged additions + the operator's own settings) and cannot be ignored or untracked as a unit; pkit's per-machine realized enforcement state belongs in the harness's already-gitignored `.claude/settings.local.json` (a related locality fix, tracked separately). The handful of runtime-local files under `.pkit/` (e.g. `sandbox-provenance.yaml`, caches) that a *shared*-mode install might want ignored are **deferred** (COR-007): until then they simply show as untracked, which is harmless.

## Rationale

**Why `.git/info/exclude` and nothing committed.** The privacy requirement is that the shared repo show *no sign* of pkit. The only git-native ignore channel that is itself uncommitted is `.git/info/exclude`; using it means pkit hides its whole footprint — including the `.pkit/` directory, which a nested `.gitignore` cannot hide itself — while writing nothing into any file the team shares. Because v1 does not attempt shared-mode ignoring at all, pkit never co-edits the adopter's root `.gitignore`, so COR-002's marker rejection and the no-shared-files invariant are honored trivially rather than navigated. The delimited region pkit owns lives only inside `info/exclude`, a file git owns — co-editing nothing the adopter authored.

**Why the shared/private asymmetry is forced, not arbitrary.** A `.gitignore` can never hide its own directory — only a parent can. Shared mode needs no ignore at all (pkit is committed); private mode must hide `.pkit/` *itself*, which requires a parent-level, per-clone, uncommitted rule. `info/exclude` is the one channel that satisfies all three, and the intent is *personal* (only this clone), which is exactly what `info/exclude` is for in git's own design.

**Why going-private bundles untrack, against the consult's default to split.** The consult recommended quarantining the destructive untrack into its own verb so a policy toggle never silently stages deletions. The product-owner refinement keeps the safety property the consult cared about — *no silent destruction* — via the explanatory confirmation, while honoring that "go private" *means* "stop sharing pkit," of which removing already-shared files is an inseparable part. `info/exclude`-for-private makes this safe in the common case: a fresh install never tracks pkit, so untrack is a genuine no-op, firing only when files were already committed — precisely when cleanup is wanted. The standalone `untrack` verb still exists (rule 5).

**Why a declared footprint rather than a hardcoded list.** Hardcoding in core would make core depend on the claude-code adapter's deploy paths (a layering inversion) and silently miss a second adapter's or a capability's footprint. Declaration-and-aggregation is the same registration seam the deploy primitives use, and keeps the mechanism harness-neutral ([COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md)) while each tier owns the truth about its own files.

**Why an ADR and not a new COR.** This mechanism composes only existing principles — the native `.git/info/exclude` channel, the registration-seam pattern, the no-shared-files invariant honored by writing nothing committed, and a bounded new git gesture. It introduces no new universal principle; it is project-kit's architectural realization, matching the ADR-002…008 hybrid. Should a second visibility surface appear later (e.g. shared-mode local-only ignoring, or controlling whether decision records sync to a remote), the broader *adoption-visibility* idea would have earned its own COR; one mechanism is not yet a principle (COR-007).

### Alternatives considered

- **A marker-delimited managed region in the committed root `.gitignore`.** Rejected — co-owns a file the adopter owns, which COR-002 rejected under the no-shared-files invariant. Scoping v1 to private (which uses only the uncommitted `info/exclude`) removes the need to write any committed `.gitignore`.
- **A pkit-owned `.pkit/.gitignore` for shared-mode local-only files now.** Deferred (COR-007) — the set is tiny (`sandbox-provenance.yaml`, caches), it is not the stated need, and keeping it out lets v1 touch no committed file at all. Revisit when the local-only set proves worth managing.
- **A `.gitignore` (or marker block) for private mode instead of `info/exclude`.** Rejected — a committed file can't hide `.pkit/` without a committed trace that leaks pkit's presence; and a nested `.pkit/.gitignore` cannot hide its own parent. `info/exclude` is the only parent-level *and* uncommitted channel.
- **A single `visibility` verb that silently untracks.** Rejected — silent destruction behind a policy flip. The confirmation (rule 4) is what makes bundling acceptable.
- **Hardcoded footprint list in core.** Rejected — layering inversion; misses second-adapter / capability footprints.
- **Print-only untrack.** Rejected by product-owner direction in favor of pkit executing it, gated by confirmation — `--dry-run` output *is* the inspectable printed-commands form.
- **`git update-index --skip-worktree`.** Rejected — files stay tracked-in-tree, visible to teammates; fails the privacy need.
- **A new COR for adoption-visibility.** Deferred — see rationale.

## Implications

- **New package-metadata `footprint` block (registration seam).** Backbone, adapters, and capabilities gain an optional `footprint: [<glob>…]` declaration. Minimal schema growth — visibility *modes* are policy the mechanism applies, not component data. A pure addition; confirm via `pkit migrations check-diff` it trips no COR-010 migration.
- **New CLI surface (PRJ-002 bump).** `pkit visibility` (no subcommand = status) with `shared` / `private` / `untrack` subcommands, each `--dry-run`-able. New commands are a surface change → version bump.
- **New, bounded git-index-mutation posture.** `visibility untrack` is the first backbone gesture that mutates adopter git-index state. Bounded by rule 5. Recorded here so the boundary cannot later widen by accident.
- **Self-hosting + migration — trivially clean.** pkit writes no committed `.gitignore`; project-kit self-hosts in `shared` mode with its hand-curated root `.gitignore` entirely untouched. No migration question arises.
- **Docs.** `install.py`'s print-a-suggestion gitignore guidance + the adapter README should note the new `visibility` control. The architecture docs gain this ADR.
- **Build sanction on acceptance.** Accepting this record sanctions building the footprint declaration + aggregation, the `private` / `shared` `.git/info/exclude` writer (delimited pkit region, idempotent add/remove), the confirm-gated `untrack` (bundled into `visibility private` and standalone, with the clean-precondition guard), and `--dry-run` — with tests, README rows, and a version bump.
