## Summary

Claude Code carries a per-command escape hatch — `dangerouslyDisableSandbox` — that lets a single Bash call run with the OS box turned off for that call. Under autonomy, where the whole safety claim is "agents run confined, so prompts are suppressed," that escape is a hole: an agent could reach for it and run a command fully unconfined while the operator believes the box still holds. [core.md rule 14](../../../.pkit/rules/core.md) already forbids reaching for that escape on routine work; this record makes the prohibition *mechanical* under the autonomy posture instead of relying on the rule alone.

The decision: **`pkit permissions setup autonomy` makes strict fail-over the autonomy default — it asserts `allowUnsandboxedCommands: false` as part of standing up the posture, so the per-command escape is inert and an agent cannot silently disable the sandbox.** This lands at the **operator-opted posture tier** — it takes effect *because the operator ran `setup autonomy`*, exactly where [ADR-004](ADR-004-autonomy-intent-confinement.md) decision point 4 says the strict lockdown belongs (it tightens that record's SHOULD into the posture's default). It is **not** a kit baseline default applied to every adopter before they opt in — that would reverse the deliberate code-level reconciliation that rejected baseline-`false` for breaking legitimate fail-over, and would override the operator's posture choice that [ADR-008](ADR-008-confinement-allowances.md) keeps the operator owning.

`setup autonomy` reaches the seal through the **existing `--strict` mechanism** (which already writes `allowUnsandboxedCommands: false` via `sandbox enable --strict`) — no new mechanism, no new writer, "owns nothing" intact ([ADR-007](ADR-007-setup-command-class.md) rule 2). The seal is **orthogonal to** the macOS `excludedCommands` exclusion path ([ADR-014](ADR-014-macos-sandbox-platform-stance.md)): different keys, different layers, so strict does not break the pkit/uv exclusion that makes macOS autonomy work. The escape stays a blessed *operator* stopgap ([ADR-017](ADR-017-confinement-allowance-cannot-override-harness-deny.md)) outside the posture — strict is opt-in and reversible by turning it off, so sealing it under autonomy does not revoke the operator's ability to use it elsewhere.

This **tightens [ADR-004](ADR-004-autonomy-intent-confinement.md) decision point 4's SHOULD into the autonomy posture's default — no supersession, no foundational re-authorisation.** The record lands `status: proposed`; operator sign-off flips it to `accepted` per the [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md) acceptance gate.

## Context

[ADR-004](ADR-004-autonomy-intent-confinement.md) split agent autonomy into intent × confinement and pinned a fail-closed invariant on any mechanism that suppresses prompts *because* execution is confined (the sandbox's auto-allow mode): such a mechanism MUST fail closed, so a missing or bypassed box blocks rather than silently degrades. Its decision point 4 made the floor concrete — a confinement preset MUST assert `sandbox.failIfUnavailable: true`, and **SHOULD** assert the strict escape-hatch lockdown (`allowUnsandboxedCommands: false`) "so a command cannot silently fall back to running unconfined-and-unprompted." That SHOULD is the seam this record acts on: under the *autonomy posture* specifically, the SHOULD becomes the default, because the `dangerouslyDisableSandbox` per-command escape is exactly the "silently fall back to unconfined" path ADR-004 named.

[core.md rule 14](../../../.pkit/rules/core.md) ("Don't escalate past the permission model for commands it already permits") is the standing instruction that forbids reaching for the sandbox-disable escape reflexively on routine work. It is loaded into every agent. But a teaching rule is advice, not a mechanism — an agent under pressure can still reach for the escape. This record makes the rule's intent enforceable for the posture that needs it most: under autonomy, strict fail-over renders the escape inert, so the rule is backed by configuration rather than by good behaviour alone.

[ADR-008](ADR-008-confinement-allowances.md) records the operator-owns-posture stance and the `--strict` honesty bullet: under `--strict` (`allowUnsandboxedCommands: false`) `excludedCommands` becomes the only out-of-box path, and the resolution is *visibility, not prohibition* — an excluded command is a deliberate, reported reduction. That bullet is the design fact this record stands on: strict and the exclusion path coexist by construction. ADR-008 also establishes that confinement posture is the operator's to choose, which is why this record places the seal at the posture tier (the operator opted in by running `setup autonomy`) rather than as a baseline floor applied pre-opt-in.

[ADR-007](ADR-007-setup-command-class.md) fixes the contract `setup autonomy` instantiates: rule 2 "owns nothing" (a setup command composes accepted primitives, introduces no new mutation engine or second writer — every effect is some primitive's effect, individually reversible by its own verb) and rule 5 "verification is part of the goal." Reaching the seal through `sandbox enable --strict` keeps rule 2 intact: the strict write is the existing `sandbox enable` primitive's effect, reversible by turning strict off.

[ADR-017](ADR-017-confinement-allowance-cannot-override-harness-deny.md) names `dangerouslyDisableSandbox` as a sanctioned *interim operator stopgap* (the JS-install workaround reframed around running the riskiest op unconfined via a loud, explicit operator gesture). This record must not revoke that — and it does not: strict is opt-in and reversible, so the operator's ability to use the escape *outside* the autonomy posture survives unchanged.

The forcing question (its producing issue, #249) is whether the unsandboxed-command escape should remain reachable under autonomy or be sealed by the posture itself. The producing session ran the `critic` and `architect` reviewers; the `architect` consult endorsed the minimal design — seal at the posture tier via `setup autonomy` / `--strict`, NOT a baseline default; record a small ADR; treat it as tightening ADR-004 decision point 4's SHOULD into a default with no supersession. As project-kit's own architecture-decision record, harness and platform specifics are in scope here, unlike the harness-neutral [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md). The record lands `status: proposed`; operator sign-off flips it to `accepted` per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md).

## Decision

**`pkit permissions setup autonomy` makes strict fail-over the autonomy default: standing up the posture asserts `allowUnsandboxedCommands: false`, so the per-command `dangerouslyDisableSandbox` escape is inert and no agent can silently disable the sandbox under autonomy.** Each condition below is load-bearing.

1. **Posture-tier, not baseline.** The seal takes effect *because the operator opted into autonomy* by running `setup autonomy` — it is the autonomy posture's default, exactly where ADR-004 decision point 4 places the strict lockdown. It is **not** a kit baseline default applied to every adopter before opt-in. Baseline-`false` was already rejected at the code level for breaking legitimate fail-over (a project that deliberately runs commands unsandboxed), and a baseline seal would override the operator's posture choice that ADR-008 keeps the operator owning. The seal lives at the posture the operator chose, and nowhere upstream of that choice.

2. **Reuses the existing `--strict` mechanism — no new mechanism.** `sandbox enable --strict` already writes `allowUnsandboxedCommands: false`; `setup autonomy` reaches the seal by composing that existing primitive (default-on for the autonomy goal), not by introducing a new writer or a new settings key. "Owns nothing" (ADR-007 rule 2) holds: the strict write is the `sandbox enable` primitive's effect, individually reversible by turning strict off.

3. **Orthogonal to the macOS exclusion path.** The seal (`allowUnsandboxedCommands: false`) and the macOS pkit/uv `excludedCommands` exclusion ([ADR-014](ADR-014-macos-sandbox-platform-stance.md)) are different keys on different layers and do not collide: strict closes the *per-command disable-the-box* escape; the exclusion carves *one named command* out of the box while keeping it hook-gated. The macOS exclusion still resolves under strict (ADR-008's `--strict` honesty bullet: under strict, `excludedCommands` is the only out-of-box path and stays a deliberate, reported reduction). **The implementing work MUST carry a probe assertion that the macOS pkit/uv exclusion still resolves with strict on** — so the seal can never silently break the path that makes macOS autonomy work.

4. **rule-14 teaching stays; the seal is its mechanical backing, never a fail-open pseudo-deny.** [core.md rule 14](../../../.pkit/rules/core.md) stays in core.md, loaded into every agent — the seal does not replace the teaching, it enforces its intent under the posture. The seal MAY be echoed as *reported state* in `sandbox status` ("unsandboxed-command escape: sealed under strict"), but only as honest reporting of the live `allowUnsandboxedCommands: false` setting — never as a fail-open pseudo-deny that claims a boundary the configuration does not actually hold.

5. **The escape survives outside the posture — strict is opt-in and reversible.** Sealing the escape under autonomy does **not** revoke the operator's ability to use `dangerouslyDisableSandbox` outside that posture. The escape remains the blessed operator stopgap ADR-017 sanctioned; the seal is the autonomy posture's choice, reversible by the operator turning strict off (`sandbox enable` without `--strict`, or disabling autonomy). What the posture removes is the *agent's silent* reach for the escape under autonomy — not the *operator's deliberate* use of it elsewhere.

6. **decide.py untouched.** This is settings/posture configuration, not a change to the decision core. The hook's `decide()` and the same-code invariant ([ADR-003](ADR-003-permission-core-code-home.md)) are unaffected — the seal lives entirely in the `sandbox` block written by the existing primitive.

## Rationale

**Why the posture tier and not a baseline default.** The escape is only a hole *under autonomy*, where prompt-suppression rests on the box holding. Outside autonomy, running a command unsandboxed is a legitimate, sometimes-necessary operator action — which is exactly why baseline-`false` was rejected in the code-level reconciliation: it would break legitimate fail-over for adopters who never asked for confinement. Placing the seal at the posture the operator opted into satisfies ADR-004 decision point 4 (which scopes the strict lockdown to a *confinement preset/posture*) without imposing it on adopters who have made no such choice. It also honours ADR-008's operator-owns-posture stance: the operator chose autonomy, and autonomy's honest default is "the escape is sealed."

**Why tightening the SHOULD is not a supersession.** ADR-004 decision point 4 already prescribes the strict lockdown — as a SHOULD on confinement presets. This record does not contradict, reverse, or replace that; it selects the SHOULD's stronger reading *for the autonomy posture specifically*, turning "should assert `allowUnsandboxedCommands: false`" into "the autonomy posture asserts it by default." That is the kind of in-bounds tightening ADR-004 anticipated when it scoped the lockdown to a preset/posture. No accepted decision changes meaning; no `superseded_by` frontmatter is warranted; no foundational re-authorisation is required. (Contrast [ADR-027](ADR-027-required-sandbox-exclusion-auto-apply.md), which narrowed a *rejected alternative* and a rule clause for one subclass — also in-place, also no supersession; this record is the lighter-weight sibling, merely promoting a SHOULD the same posture's decision already named.)

**Why reuse `--strict` rather than a new lever.** A new mechanism would mean a second writer for the sandbox block and a new key to reconcile — both forbidden by ADR-007 rule 2 and ADR-008's single-sandbox-block-writer discipline. `--strict` already does exactly the write the seal needs; `setup autonomy` composing it is the orthodox path, keeps every effect individually reversible, and inherits ADR-008's already-recorded `--strict` honesty (visibility, not prohibition; the exclusion path stays open and reported).

**Why orthogonality to the exclusion must be probe-asserted, not assumed.** The seal and the macOS exclusion are *designed* to be orthogonal (different keys, different layers), but the macOS exclusion is the single thing that makes macOS autonomy work (ADR-014) — so an undetected interaction would silently break the primary developer platform. The cheapest insurance is a probe assertion that the exclusion still resolves with strict on; making it a recorded requirement (decision point 3) means the implementing work cannot ship the seal without proving the path it must not break still holds. This is the spike-pinned-by-evidence discipline ADR-014 and ADR-017 set, applied prospectively.

**Why the escape stays a stopgap, not a banned primitive.** ADR-017 sanctioned `dangerouslyDisableSandbox` as the honest interim operator workaround for harness over-fires the toolkit model cannot fix. Banning it outright would forfeit that escape valve. The seal is narrower and correct: it removes the *agent's silent* reach under autonomy (where it is a hole) while leaving the *operator's deliberate* use intact everywhere else (where it is a sanctioned stopgap). Reversibility — strict off restores the escape — is what keeps the seal a posture choice rather than a revocation.

### Alternatives considered

- **Make `allowUnsandboxedCommands: false` a kit baseline default for every adopter.** Rejected — reverses the code-level reconciliation that rejected baseline-`false` for breaking legitimate fail-over (running a command unsandboxed is a valid operator action outside autonomy), and overrides the operator's posture choice ADR-008 keeps the operator owning. The seal belongs at the posture the operator opted into, not upstream of that opt-in.

- **Ban `dangerouslyDisableSandbox` outright (refuse it everywhere).** Rejected — revokes the sanctioned interim operator stopgap ADR-017 recorded for harness over-fires. The escape is a hole only under autonomy; sealing it there (reversibly) keeps the operator's deliberate use available elsewhere.

- **Rely on core.md rule 14 alone (no mechanical seal).** Rejected as insufficient under autonomy — a teaching rule is advice an agent under pressure can still override, and autonomy is precisely where a silent unsandboxed fallback is most dangerous (prompts are suppressed because the box is believed to hold). The rule stays; the seal gives it mechanical backing for the posture that needs it.

- **Introduce a new settings key / new writer for the seal.** Rejected — `--strict` already writes `allowUnsandboxedCommands: false`; a second mechanism would add a second writer to the sandbox block (forbidden by ADR-007 rule 2 and ADR-008's single-writer discipline) with no capability the existing lever lacks.

- **Surface the seal as a hook-level deny (a fail-open pseudo-deny in `decide()`).** Rejected — the seal is a settings/posture fact, not a decision-core change; representing it as a hook deny would touch `decide.py` (decision point 6 forbids), risk the same-code invariant (ADR-003), and tempt a fail-open pseudo-deny that claims a boundary the configuration must actually hold. `sandbox status` reports the live setting honestly; that is the correct surface.

## Implications

- **ADR-004 decision point 4's SHOULD is tightened into the autonomy posture's default; no supersession.** "A confinement preset SHOULD assert `allowUnsandboxedCommands: false`" becomes "the autonomy posture asserts it by default." ADR-004's two-axis model, fail-closed invariant, and delegate-confinement stance all stand unchanged. No `superseded_by` frontmatter on ADR-004; no foundational re-authorisation. ADR-004 decision point 4 gains a one-line forward-pointer to this record naming the posture default.

- **ADR-008's `--strict` honesty bullet is the path this rides; it stands unchanged.** Under strict, `excludedCommands` remains the only out-of-box path and stays a deliberate, reported reduction; the seal does not change that. The single-sandbox-block-writer discipline (rule 2) is honoured — the seal is the existing `sandbox enable --strict` write, not a new writer.

- **ADR-017's operator stopgap survives outside the posture.** Sealing the escape under autonomy does not revoke the operator's deliberate use of `dangerouslyDisableSandbox` elsewhere — strict is opt-in and reversible. No change to ADR-017.

- **ADR-007's setup-command contract holds.** "Owns nothing" (rule 2) is preserved: the seal is the existing strict primitive's effect, individually reversible. This is not flag-forwarding of a dangerous escape (rule 6) — it is the orchestrator *closing* an escape by default, the opposite direction.

- **Orthogonality is a recorded probe requirement.** The implementing work MUST assert (via `probe` / test) that the macOS pkit/uv `excludedCommands` exclusion still resolves with `allowUnsandboxedCommands: false` set — the seal must not silently break the path that makes macOS autonomy work (ADR-014).

- **decide.py is untouched.** The seal lives in the `sandbox` block; the hook's `decide()` and the same-code invariant (ADR-003) are unaffected. The seal MAY be echoed as reported state in `sandbox status`, never as a fail-open pseudo-deny.

- **Relationship to ADR-027.** Both records adjust `setup autonomy`'s behaviour and land for acceptance together; they are independent and compatible. ADR-027 admits a necessity-verified auto-apply of the platform-*required exclusion* (a widening, carefully fenced); this record makes strict fail-over the posture *default* (a narrowing — it closes the unsandboxed escape). They touch different sandbox keys (`excludedCommands` vs `allowUnsandboxedCommands`) and reinforce rather than conflict: under autonomy the box is sealed against silent disable *and* the one forced exclusion is stood up honestly.

- **Implementation is separate work, not this ADR.** Defaulting `setup autonomy` to compose `sandbox enable --strict`, the `sandbox status` reported-state line, and the orthogonality probe assertion land in the implementing issue. This record sanctions building them on acceptance.

- **Versioning.** Behaviour-observable surface change (`setup autonomy` now seals the escape by default) → the implementing change bumps `.pkit/VERSION` per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md). This decision record alone does not bump.

- **No migration.** No file/directory rename or removal in a kit-owned tree, no `schema_version` bump, no breaking CLI signature change, no capability subtree restructure — `setup autonomy` gains a default, it does not change its signature. `pkit migrations check-diff --include-working-tree --base main` is expected clean for the ADR file; the implementing change runs the same check.

- **Cross-platform and cross-harness.** The seal is the harness-neutral "no silent unconfined fallback under a confinement posture" principle realised through Claude Code's `allowUnsandboxedCommands` key. Each future adapter realises the same posture default through its own equivalent lever (or declares the residual gap if its harness has none), per COR-028's honesty discipline.
