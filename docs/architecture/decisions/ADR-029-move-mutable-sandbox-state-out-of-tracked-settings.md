## Summary

[ADR-008](ADR-008-confinement-allowances.md) rule 4 promised that a *widening* sandbox allowance ‚Äî one that carves a command out of the box to run unconfined ‚Äî is **never written to a committed file**, so one machine's boundary reduction can never silently lower the floor for every checkout. In a conventional adopter layout that holds for free, because `.claude/settings.json` is per-machine. But in a repo that **tracks** `settings.json` ‚Äî project-kit's own self-host, and any adopter who commits it ‚Äî the single sandbox-block writer puts *both* the committed baseline and the mutable widenings into the same tracked file, so rule 4 holds only by the operator's vigilance in not committing them. The single writer routes a widening (`excludedCommands: ["uv"]`) into the tracked `settings.json`'s **working copy** ‚Äî drift the operator must keep unstaged; on `main` it stays uncommitted only by that vigilance. (Verified: `main`'s committed `settings.json` is clean; the `excludedCommands` line is working-tree drift from a prior local `setup autonomy`, not a committed line. The gap is real ‚Äî the writer *does* place a widening into the tracked file ‚Äî but vigilance has so far kept it out of a commit, which is precisely the "holds only if everyone remembers" failure this record makes unnecessary.) Flagged as the open follow-up in [ADR-027](ADR-027-required-sandbox-exclusion-auto-apply.md) condition 4 (its producing issue, #259).

The decision: **the single sandbox-block writer routes each entry to its home by ADR-008's narrowing/widening classification. The narrowing floor + baseline (`enabled`, `failIfUnavailable`, the credential `denyRead` floor, narrowing `allowWrite` / `allowedHosts`) stays in the committed `.claude/settings.json`; the mutable widening + per-machine deviations (`excludedCommands`, the `allowUnsandboxedCommands` strict seal, operator `denyRead` deviations) go to the gitignored `.claude/settings.local.json`.** This makes ADR-008 rule 4's never-written-to-a-committed-file guarantee hold **by construction** for *every* adopter ‚Äî not by the accident of whether they track `settings.json`, and not by operator discipline.

The routing is sound because the only cross-file merge it depends on is the committed `denyRead` floor surviving alongside local entries ‚Äî a documented filesystem-array merge ‚Äî plus local-wins scalar precedence for the seal (`excludedCommands` is routed all-to-local, so its merge semantics never matter; see Context for the honest evidence breakdown). Splitting entries across the two files changes *where each entry lives*, not *what the harness enforces*. The single-writer discipline ([ADR-008](ADR-008-confinement-allowances.md) rule 2) is preserved ‚Äî there remains exactly **one** writer that *routes* each entry to its classified destination; this is emphatically **not** two independent writers, which would be a rule-2 regression.

This **realizes ADR-008 rule 4 by construction and clarifies rule 2's single-writer as single-writer-with-routing ‚Äî no supersession, no foundational re-authorisation.** It maps cleanly onto [COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md)'s baseline / project-additions / local layering and onto [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md)'s universal-applicability bar (one code path, correct everywhere; a no-op for adopters who do not track `settings.json`). The record lands `status: proposed`; operator sign-off flips it to `accepted` per the [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md) acceptance gate.

## Context

[ADR-008](ADR-008-confinement-allowances.md) split confinement allowances by their effect on the boundary: a *narrowing* allowance makes the box usable without enlarging the agent's reach (a build-cache `allowWrite`, a needed socket, the credential `denyRead` floor); a *widening* allowance lowers the boundary (carving a command out via `excludedCommands`, weakening TLS). Its rule 4 governs the widening class with four guarantees ‚Äî applied only by explicit gesture, loud, **never written to a committed file**, always reported ‚Äî and its rule 2 mandates a *single* sandbox-block writer so exactly one place knows the merge rule. The never-persisted guarantee is the one that stops a machine-local boundary reduction from becoming a silent floor-lowering for every checkout.

That guarantee is **environment-conditional today**. [ADR-027](ADR-027-required-sandbox-exclusion-auto-apply.md) condition 4 named the gap precisely: the committed-file prohibition "is unconditional in a conventional adopter layout where `.claude/settings.json` is per-machine. In a repo that *tracks* that file ‚Äî e.g. project-kit's own self-host ‚Äî the exclusion lands in a tracked file and the operator must keep it uncommitted until the mutable sandbox state moves to an untracked file; tracked as #259." This record is that move. The forcing observation: the writer places every sandbox entry ‚Äî committed floor and mutable widening alike ‚Äî into `settings.json`, so when that file is tracked, rule 4 holds only by the operator remembering not to stage the widening lines. The kit's own `settings.json` working copy carries `excludedCommands: ["uv"]` right now (working-tree drift from a local `setup autonomy`, kept out of a commit only by vigilance) ‚Äî proving the gap is live, not theoretical.

The harness-merge facts the routing relies on, with their evidence strength stated honestly:

- **`denyRead` (filesystem array) merges across scopes** ‚Äî this is the load-bearing one, and it is the documented case: the Claude Code sandboxing docs state filesystem arrays are "combined, not replaced" across settings scopes, so a committed `denyRead` floor *survives* when `settings.local.json` is present. The credential floor cannot be silently dropped by the split.
- **Scalars (`allowUnsandboxedCommands`, `failIfUnavailable`, `enabled`) take higher-precedence (local) value, others preserved** ‚Äî standard settings-precedence, documented.
- **`settings.local.json` is the harness's own per-machine sandbox file** (the `/sandbox` panel writes there) and is gitignored by convention ‚Äî documented.

A scoping note that *narrows* the dependency: **`excludedCommands` is routed all-to-local and never to the committed file**, so the design never relies on `excludedCommands` merging across scopes ‚Äî only one file ever holds it. (The general "do `sandbox.excludedCommands` arrays union across scopes?" question is *not* explicitly documented for the `sandbox` block and is **not** load-bearing here precisely because we never split that key.) So the routing's correctness rests on exactly two documented facts: `denyRead` filesystem-array merge (floor survives) and scalar local-precedence (seal). The merge is what lets the routing be a pure relocation: the runtime config is identical whether an entry sits in the committed or the local file. Two layers of confidence on the floor-survives property: pkit unit-tests its **own** `_merge_sandbox_blocks` mirror (a committed `denyRead` floor unions with a local `excludedCommands` so the floor is preserved in pkit's merged view), and the **harness-side** runtime merge rests on the documented filesystem-array "combined, not replaced" behavior (the harness owns that merge; pkit cannot unit-test the harness's runtime, only mirror it). If a future harness change made `sandbox` filesystem arrays *replace* rather than merge, pkit's mirror and the harness would diverge ‚Äî the residual risk this records honestly rather than claiming away.

The forcing question (its producing issue, #259) is whether to *document* the tracked-file gap or *close* it structurally. The architect consult recommended closing it: route mutable state to the untracked file so the invariant is restored unconditionally rather than left to operator vigilance. As project-kit's own architecture-decision record, harness and platform specifics are in scope here, unlike the harness-neutral [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md). The record lands `status: proposed`; operator sign-off is the acceptance gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md).

## Decision

**The single sandbox-block writer routes each entry to its destination file by ADR-008's narrowing/widening classification: the narrowing floor + baseline stays in the committed `.claude/settings.json`; the mutable widening + per-machine deviations go to the gitignored `.claude/settings.local.json`. This makes ADR-008 rule 4's never-committed guarantee hold by construction for every adopter.** Each condition below is load-bearing.

1. **Routing by boundary effect, keyed to the existing classification.** The destination is chosen by the same narrowing/widening discriminator ADR-008 already encodes in the toolkit data ‚Äî no new axis. Narrowing-and-baseline entries (`enabled`, `failIfUnavailable`, the credential `denyRead` floor, narrowing `filesystem.allowWrite`, narrowing-but-reported `network.allowedHosts` per [ADR-015](ADR-015-command-declared-network-egress.md)) route to the committed `settings.json`. Widening-and-deviation entries (`excludedCommands`, the `allowUnsandboxedCommands` strict seal per [ADR-028](ADR-028-seal-unsandboxed-escape-under-autonomy-posture.md), operator `denyRead` deviations off the floor) route to the gitignored `settings.local.json`. The classification is the routing key; the writer does not invent a second decision.

2. **One writer, with routing ‚Äî not two writers.** ADR-008 rule 2's single-sandbox-block-writer discipline is preserved and clarified: there remains exactly **one** internal helper that mutates the sandbox block, and it now *routes* each entry to its classified destination file. This is the design constraint, stated explicitly: **two independent writers (one per file) would be a rule-2 regression** ‚Äî it would reintroduce the "two places know the merge rule" failure rule 2 exists to prevent. The writer knows both files because it owns the routing; nothing else writes the sandbox block.

3. **Rule 4 realized by construction, not by vigilance.** With widenings routed to the gitignored file, ADR-008 rule 4's "never written to a committed file" guarantee holds *mechanically* for every adopter ‚Äî the operator cannot accidentally commit a widening because the writer never puts one in a tracked file. The guarantee no longer depends on whether the adopter tracks `settings.json` (it is a no-op relocation for those who do not) nor on operator discipline (it is structural for those who do).

4. **The `denyRead` split-brain is resolved by the array-merge.** Credentials are protected by a committed baseline `denyRead` floor; an operator's per-machine `denyRead` deviations land in `settings.local.json`. Because Claude Code merges `denyRead` arrays across scopes (verified fact above), the effective deny-set at runtime is the union of the committed floor and the local deviations ‚Äî the floor cannot be weakened by the local file's presence, and the deviation is honoured. There is no split-brain: the floor is committed and survives; deviations are local and add.

5. **Provenance stays consistent across the two files.** The `sandbox-provenance.yaml` sidecar (already gitignored) records which entries pkit authored. Moving widenings to `settings.local.json` co-locates the effect (the widening) with its ledger (the gitignored provenance) ‚Äî a consistency win. **Implementation constraint:** the disable / seal provenance read must take the *union* across both settings files so teardown (ADR-008 rule 2's provenance-tagged removal; ADR-027 condition 6's self-heal) stays correct ‚Äî the writer must read both files to know what it authored, in whichever file it landed.

6. **COR-002 layering is honoured; the merge baseline is untouched.** The committed/gitignored split maps onto [COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md)'s baseline / project-additions / local tiers: `settings.local.json` is the local tier the layering already names. The kit's merge baseline (`core/settings.json`) carries **no** sandbox block ‚Äî the sandbox writer authors the block in the project-side files, not the baseline ‚Äî so the merge primitive's path is entirely untouched by this change. This is a sandbox-writer routing change, not a merge-primitive change.

## Rationale

**Why routing is the right shape, and why the harness merge is what makes it free.** The naive fix ‚Äî document the tracked-file gap and ask operators to keep widenings unstaged ‚Äî leaves rule 4 holding by vigilance, the exact "boundary holds only if everyone remembers" failure the permission subsystem exists to remove. Routing closes the gap structurally. It is *only* a relocation (not a behaviour change) because Claude Code deep-merges the sandbox block across the two files: the committed floor's arrays survive, the local widenings add, and the effective runtime config is identical to today's single-file config. Verifying that merge semantics against the official docs is what turns "move mutable state out" from a hopeful refactor into a provably effect-preserving one ‚Äî the floor cannot be silently dropped by the split.

**Why one-writer-with-routing, not two writers.** The tempting implementation is two writers ‚Äî one owning `settings.json`, one owning `settings.local.json`. That is a rule-2 regression: rule 2 exists so exactly one place knows the sandbox-block merge rule, and two writers would split that knowledge across two code paths that must be kept consistent by hand (the same failure mode rule 2 named). The correct shape keeps the single writer and gives it a routing responsibility: it classifies each entry and writes it to the entry's home. One place still knows the merge rule; it now also knows the routing rule. The single-writer invariant is preserved in substance and merely *clarified* to "single writer that routes."

**Why this is an ADR, not a PRJ.** The fix is universal, not project-kit-specific. It is one code path in the sandbox writer, correct for every adopter: structural for those who track `settings.json`, a transparent no-op for those who do not (their `settings.json` was already per-machine, so routing widenings to `settings.local.json` changes nothing observable). The decision being recorded is a universal architectural boundary ‚Äî *where confinement state lives, by its effect on the boundary* ‚Äî which is exactly the ADR-008 decision space this record extends, and it clears COR-014's universal-applicability bar. A PRJ would wrongly frame a universal invariant as a project quirk. (A thin companion PRJ could optionally pin the orthogonal project fact that "project-kit tracks `settings.json` on purpose," but that is project policy, not the architectural decision; the architectural decision is this ADR.)

**Why rule 4 stays in substance and gains only a forward-pointer.** This record does not weaken, reverse, or replace rule 4 ‚Äî it makes rule 4's existing guarantee hold by construction where it previously held conditionally. That is the same tighten-without-supersession shape as [ADR-028](ADR-028-seal-unsandboxed-escape-under-autonomy-posture.md) tightening [ADR-004](ADR-004-autonomy-intent-confinement.md)'s SHOULD, and ADR-028/ADR-027 forward-pointing into the records they realize. No accepted decision changes meaning; rule 2 is clarified (single-writer ‚Üí single-writer-with-routing), not contradicted; no `superseded_by` frontmatter is warranted; no foundational re-authorisation is required.

### Alternatives considered

- **Document the tracked-file gap; rely on the operator not to commit widenings.** Rejected ‚Äî leaves ADR-008 rule 4 holding by vigilance, the precise "boundary holds only if everyone remembers" failure the subsystem exists to eliminate. The kit's own live `excludedCommands: ["uv"]` violation is the evidence that vigilance fails. Routing makes the guarantee structural.

- **Two independent writers, one per settings file.** Rejected ‚Äî a regression on ADR-008 rule 2. Splitting the sandbox-block merge knowledge across two code paths reintroduces the "two places know the merge rule" failure rule 2 forbids. The single writer with a routing responsibility keeps one place authoritative.

- **Put everything (floor + widenings) in `settings.local.json`; commit nothing.** Rejected ‚Äî the narrowing floor (the credential `denyRead`, baseline `failIfUnavailable`) *should* be committed so it ships to every checkout; that is the floor's whole purpose. Only the *widening* and per-machine deviations must stay uncommitted. Routing by boundary effect is what keeps the floor shared and the widenings local.

- **Record this as a PRJ scoped to project-kit's self-host.** Rejected ‚Äî the fix is a universal code path correct for every adopter (a no-op for those who do not track `settings.json`), and the decision is a universal "where confinement state lives" boundary extending ADR-008. A PRJ would mis-scope a universal invariant as a project quirk (COR-014).

## Implications

- **ADR-008 rule 4 is realized by construction; rule 2 is clarified, not changed; no supersession.** Rule 4's never-committed guarantee now holds mechanically for every adopter. Rule 2's single-writer discipline is preserved and read as single-writer-with-routing. ADR-008's boundary-effect organising principle (rules 1‚Äì6) stands unchanged. No `superseded_by` frontmatter on ADR-008; no foundational re-authorisation. **ADR-008 rule 4 gains a one-line forward-pointer to this record.**

- **The rule-4 gap is closed by construction, not by a strip step.** `main`'s committed `.claude/settings.json` is already clean (the `excludedCommands: ["uv"]` is working-tree drift, not a committed line ‚Äî see Context). So there is nothing to strip from the committed file; the fix is that the writer can no longer *route* a widening there ‚Äî `excludedCommands` and the seal land in the gitignored `settings.local.json` by construction, so the working copy of the tracked file stays clean too. The macOS required exclusion of ADR-027 lands in the gitignored file on the next `setup autonomy` / `sandbox` run. The committed file carries the narrowing floor only. **Already-installed residual (closed by #275):** an adopter whose *pre-ADR-029* `setup autonomy` already wrote a `_required` exclusion into the tracked `settings.json` keeps that drift until it is relocated ‚Äî the new-widening routing above never moves an existing entry. `setup autonomy` now runs a provenance-scoped relocation pass that moves such a `_required` tracked-file exclusion to `settings.local.json` at runtime (writing local before stripping committed, so the runtime union never un-excludes the command mid-move); operator `_manual` and untagged/foreign entries are left untouched. The convergence on the next run *is* the migration mechanism ‚Äî no one-shot lifecycle script (the entry is uncommitted working-copy drift, not a live floor-lowering).

- **`denyRead` floor + deviations resolved by the array-merge.** The committed baseline floor and local operator deviations merge to the union at runtime (verified harness fact); the floor cannot be weakened by the local file. No split-brain.

- **Provenance read must union both files.** The disable / seal / self-heal provenance read (ADR-008 rule 2; ADR-027 condition 6) must take the union across `settings.json` and `settings.local.json`, since the writer now lands authored entries in whichever file the entry's classification dictates. This is a recorded implementation constraint, not a behaviour change to teardown semantics.

- **COR-002 layering honoured; merge primitive untouched.** The split maps onto COR-002's baseline / project-additions / local tiers; the merge baseline (`core/settings.json`) carries no sandbox block, so the merge path is not touched. This is a sandbox-writer change, not a merge-primitive change.

- **Implementation is separate work, not this ADR.** The routing logic in the single sandbox-block writer, the provenance union-read, the removal of the committed `excludedCommands: ["uv"]`, the ADR-008 forward-pointer, README surface updates clarifying per-machine vs committed state, and tests land in the implementing issue (#259, branch `fix/259-move-mutable-sandbox-state-out`). This record sanctions building them on acceptance.

- **Versioning.** Behaviour-observable surface change (the sandbox writer now routes widenings to `settings.local.json`; committed `settings.json` no longer carries widenings) ‚Üí the implementing change bumps `.pkit/VERSION` per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md). This decision record alone does not bump.

- **Migration.** No file/directory rename or removal in a kit-owned tree, no `schema_version` bump, no breaking CLI signature change, no capability subtree restructure ‚Äî the writer gains a routing step and a target file. Whether the move of an *existing* committed widening to the local file warrants a one-shot idempotent migration (relocate-and-strip on already-installed adopters who committed a widening) is a COR-010 judgement for the implementing change; `pkit migrations check-diff --include-working-tree --base main` gates it. The ADR file itself is migration-clean.

- **Cross-platform and cross-harness.** The decision is the harness-neutral principle "mutable boundary-lowering state lives where it cannot be committed, the shared floor lives where it ships" realised through Claude Code's `settings.json` / `settings.local.json` deep-merge. Each future adapter realises the same split through its own committed/local settings layering (or declares the residual gap if its harness has no local tier), per COR-028's honesty discipline.

- **A second routing axis composes with this one ([ADR-032](ADR-032-per-machine-activation-routing-axis.md)).** This record routes by boundary effect alone. ADR-032 adds a value-provenance + per-machine-activation axis, OR-composed at the routing point (route local if *either* axis says local), to keep host-derived values (the SSH-agent socket) and operator activations (`sandbox.enabled`, `active_profile`) out of committed state ‚Äî the socket case being a live regression this effect-only routing introduced. This record's effect axis is unchanged; ADR-032 is a sibling, not an amendment.
