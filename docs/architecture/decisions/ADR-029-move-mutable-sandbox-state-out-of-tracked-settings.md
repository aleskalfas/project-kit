## Summary

[ADR-008](ADR-008-confinement-allowances.md) rule 4 promised that a *widening* sandbox allowance — one that carves a command out of the box to run unconfined — is **never written to a committed file**, so one machine's boundary reduction can never silently lower the floor for every checkout. In a conventional adopter layout that holds for free, because `.claude/settings.json` is per-machine. But in a repo that **tracks** `settings.json` — project-kit's own self-host, and any adopter who commits it — the single sandbox-block writer puts *both* the committed baseline and the mutable widenings into the same tracked file, so rule 4 holds only by the operator's vigilance in not committing them. The kit's own tracked `settings.json` currently carries `excludedCommands: ["uv"]`: a live rule-4 violation, flagged as the open follow-up in [ADR-027](ADR-027-required-sandbox-exclusion-auto-apply.md) condition 4 (its producing issue, #259).

The decision: **the single sandbox-block writer routes each entry to its home by ADR-008's narrowing/widening classification. The narrowing floor + baseline (`enabled`, `failIfUnavailable`, the credential `denyRead` floor, narrowing `allowWrite` / `allowedHosts`) stays in the committed `.claude/settings.json`; the mutable widening + per-machine deviations (`excludedCommands`, the `allowUnsandboxedCommands` strict seal, operator `denyRead` deviations) go to the gitignored `.claude/settings.local.json`.** This makes ADR-008 rule 4's never-written-to-a-committed-file guarantee hold **by construction** for *every* adopter — not by the accident of whether they track `settings.json`, and not by operator discipline.

The routing is sound because Claude Code **deep-merges** the `sandbox` block across the two files at runtime (verified against the official sandboxing docs, below): the committed narrowing floor and the local widenings combine into one effective config, so splitting them across the two files changes *where each entry lives*, not *what the harness enforces*. The single-writer discipline ([ADR-008](ADR-008-confinement-allowances.md) rule 2) is preserved — there remains exactly **one** writer that *routes* each entry to its classified destination; this is emphatically **not** two independent writers, which would be a rule-2 regression.

This **realizes ADR-008 rule 4 by construction and clarifies rule 2's single-writer as single-writer-with-routing — no supersession, no foundational re-authorisation.** It maps cleanly onto [COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md)'s baseline / project-additions / local layering and onto [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md)'s universal-applicability bar (one code path, correct everywhere; a no-op for adopters who do not track `settings.json`). The record lands `status: proposed`; operator sign-off flips it to `accepted` per the [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md) acceptance gate.

## Context

[ADR-008](ADR-008-confinement-allowances.md) split confinement allowances by their effect on the boundary: a *narrowing* allowance makes the box usable without enlarging the agent's reach (a build-cache `allowWrite`, a needed socket, the credential `denyRead` floor); a *widening* allowance lowers the boundary (carving a command out via `excludedCommands`, weakening TLS). Its rule 4 governs the widening class with four guarantees — applied only by explicit gesture, loud, **never written to a committed file**, always reported — and its rule 2 mandates a *single* sandbox-block writer so exactly one place knows the merge rule. The never-persisted guarantee is the one that stops a machine-local boundary reduction from becoming a silent floor-lowering for every checkout.

That guarantee is **environment-conditional today**. [ADR-027](ADR-027-required-sandbox-exclusion-auto-apply.md) condition 4 named the gap precisely: the committed-file prohibition "is unconditional in a conventional adopter layout where `.claude/settings.json` is per-machine. In a repo that *tracks* that file — e.g. project-kit's own self-host — the exclusion lands in a tracked file and the operator must keep it uncommitted until the mutable sandbox state moves to an untracked file; tracked as #259." This record is that move. The forcing observation: the writer places every sandbox entry — committed floor and mutable widening alike — into `settings.json`, so when that file is tracked, rule 4 holds only by the operator remembering not to stage the widening lines. The kit's own `settings.json` carries `excludedCommands: ["uv"]` right now, proving the gap is live, not theoretical.

The load-bearing harness fact, **verified against the official Claude Code sandboxing documentation** (not assumed): Claude Code **deep-merges** the `sandbox` block across `.claude/settings.json` (project scope) and `.claude/settings.local.json` (local scope). Array-valued keys — `sandbox.excludedCommands`, `sandbox.filesystem.denyRead` / `allowRead`, `sandbox.network.allowedHosts` — are **merged** across scopes ("paths from every scope are combined, not replaced"), so a committed `denyRead` floor in `settings.json` *survives* when `settings.local.json` adds `excludedCommands`. Scalar keys — `allowUnsandboxedCommands`, `failIfUnavailable`, `enabled` — take the higher-precedence (local) value, with non-overridden keys preserved. And `settings.local.json` is where the harness's own `/sandbox` panel writes per-machine sandbox state; it is gitignored by convention. The merge is what lets the routing be a pure relocation: the runtime config is identical whether an entry sits in the committed or the local file.

The forcing question (its producing issue, #259) is whether to *document* the tracked-file gap or *close* it structurally. The architect consult recommended closing it: route mutable state to the untracked file so the invariant is restored unconditionally rather than left to operator vigilance. As project-kit's own architecture-decision record, harness and platform specifics are in scope here, unlike the harness-neutral [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md). The record lands `status: proposed`; operator sign-off is the acceptance gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md).

## Decision

**The single sandbox-block writer routes each entry to its destination file by ADR-008's narrowing/widening classification: the narrowing floor + baseline stays in the committed `.claude/settings.json`; the mutable widening + per-machine deviations go to the gitignored `.claude/settings.local.json`. This makes ADR-008 rule 4's never-committed guarantee hold by construction for every adopter.** Each condition below is load-bearing.

1. **Routing by boundary effect, keyed to the existing classification.** The destination is chosen by the same narrowing/widening discriminator ADR-008 already encodes in the toolkit data — no new axis. Narrowing-and-baseline entries (`enabled`, `failIfUnavailable`, the credential `denyRead` floor, narrowing `filesystem.allowWrite`, narrowing-but-reported `network.allowedHosts` per [ADR-015](ADR-015-command-declared-network-egress.md)) route to the committed `settings.json`. Widening-and-deviation entries (`excludedCommands`, the `allowUnsandboxedCommands` strict seal per [ADR-028](ADR-028-seal-unsandboxed-escape-under-autonomy-posture.md), operator `denyRead` deviations off the floor) route to the gitignored `settings.local.json`. The classification is the routing key; the writer does not invent a second decision.

2. **One writer, with routing — not two writers.** ADR-008 rule 2's single-sandbox-block-writer discipline is preserved and clarified: there remains exactly **one** internal helper that mutates the sandbox block, and it now *routes* each entry to its classified destination file. This is the design constraint, stated explicitly: **two independent writers (one per file) would be a rule-2 regression** — it would reintroduce the "two places know the merge rule" failure rule 2 exists to prevent. The writer knows both files because it owns the routing; nothing else writes the sandbox block.

3. **Rule 4 realized by construction, not by vigilance.** With widenings routed to the gitignored file, ADR-008 rule 4's "never written to a committed file" guarantee holds *mechanically* for every adopter — the operator cannot accidentally commit a widening because the writer never puts one in a tracked file. The guarantee no longer depends on whether the adopter tracks `settings.json` (it is a no-op relocation for those who do not) nor on operator discipline (it is structural for those who do).

4. **The `denyRead` split-brain is resolved by the array-merge.** Credentials are protected by a committed baseline `denyRead` floor; an operator's per-machine `denyRead` deviations land in `settings.local.json`. Because Claude Code merges `denyRead` arrays across scopes (verified fact above), the effective deny-set at runtime is the union of the committed floor and the local deviations — the floor cannot be weakened by the local file's presence, and the deviation is honoured. There is no split-brain: the floor is committed and survives; deviations are local and add.

5. **Provenance stays consistent across the two files.** The `sandbox-provenance.yaml` sidecar (already gitignored) records which entries pkit authored. Moving widenings to `settings.local.json` co-locates the effect (the widening) with its ledger (the gitignored provenance) — a consistency win. **Implementation constraint:** the disable / seal provenance read must take the *union* across both settings files so teardown (ADR-008 rule 2's provenance-tagged removal; ADR-027 condition 6's self-heal) stays correct — the writer must read both files to know what it authored, in whichever file it landed.

6. **COR-002 layering is honoured; the merge baseline is untouched.** The committed/gitignored split maps onto [COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md)'s baseline / project-additions / local tiers: `settings.local.json` is the local tier the layering already names. The kit's merge baseline (`core/settings.json`) carries **no** sandbox block — the sandbox writer authors the block in the project-side files, not the baseline — so the merge primitive's path is entirely untouched by this change. This is a sandbox-writer routing change, not a merge-primitive change.

## Rationale

**Why routing is the right shape, and why the harness merge is what makes it free.** The naive fix — document the tracked-file gap and ask operators to keep widenings unstaged — leaves rule 4 holding by vigilance, the exact "boundary holds only if everyone remembers" failure the permission subsystem exists to remove. Routing closes the gap structurally. It is *only* a relocation (not a behaviour change) because Claude Code deep-merges the sandbox block across the two files: the committed floor's arrays survive, the local widenings add, and the effective runtime config is identical to today's single-file config. Verifying that merge semantics against the official docs is what turns "move mutable state out" from a hopeful refactor into a provably effect-preserving one — the floor cannot be silently dropped by the split.

**Why one-writer-with-routing, not two writers.** The tempting implementation is two writers — one owning `settings.json`, one owning `settings.local.json`. That is a rule-2 regression: rule 2 exists so exactly one place knows the sandbox-block merge rule, and two writers would split that knowledge across two code paths that must be kept consistent by hand (the same failure mode rule 2 named). The correct shape keeps the single writer and gives it a routing responsibility: it classifies each entry and writes it to the entry's home. One place still knows the merge rule; it now also knows the routing rule. The single-writer invariant is preserved in substance and merely *clarified* to "single writer that routes."

**Why this is an ADR, not a PRJ.** The fix is universal, not project-kit-specific. It is one code path in the sandbox writer, correct for every adopter: structural for those who track `settings.json`, a transparent no-op for those who do not (their `settings.json` was already per-machine, so routing widenings to `settings.local.json` changes nothing observable). The decision being recorded is a universal architectural boundary — *where confinement state lives, by its effect on the boundary* — which is exactly the ADR-008 decision space this record extends, and it clears COR-014's universal-applicability bar. A PRJ would wrongly frame a universal invariant as a project quirk. (A thin companion PRJ could optionally pin the orthogonal project fact that "project-kit tracks `settings.json` on purpose," but that is project policy, not the architectural decision; the architectural decision is this ADR.)

**Why rule 4 stays in substance and gains only a forward-pointer.** This record does not weaken, reverse, or replace rule 4 — it makes rule 4's existing guarantee hold by construction where it previously held conditionally. That is the same tighten-without-supersession shape as [ADR-028](ADR-028-seal-unsandboxed-escape-under-autonomy-posture.md) tightening [ADR-004](ADR-004-autonomy-intent-confinement.md)'s SHOULD, and ADR-028/ADR-027 forward-pointing into the records they realize. No accepted decision changes meaning; rule 2 is clarified (single-writer → single-writer-with-routing), not contradicted; no `superseded_by` frontmatter is warranted; no foundational re-authorisation is required.

### Alternatives considered

- **Document the tracked-file gap; rely on the operator not to commit widenings.** Rejected — leaves ADR-008 rule 4 holding by vigilance, the precise "boundary holds only if everyone remembers" failure the subsystem exists to eliminate. The kit's own live `excludedCommands: ["uv"]` violation is the evidence that vigilance fails. Routing makes the guarantee structural.

- **Two independent writers, one per settings file.** Rejected — a regression on ADR-008 rule 2. Splitting the sandbox-block merge knowledge across two code paths reintroduces the "two places know the merge rule" failure rule 2 forbids. The single writer with a routing responsibility keeps one place authoritative.

- **Put everything (floor + widenings) in `settings.local.json`; commit nothing.** Rejected — the narrowing floor (the credential `denyRead`, baseline `failIfUnavailable`) *should* be committed so it ships to every checkout; that is the floor's whole purpose. Only the *widening* and per-machine deviations must stay uncommitted. Routing by boundary effect is what keeps the floor shared and the widenings local.

- **Record this as a PRJ scoped to project-kit's self-host.** Rejected — the fix is a universal code path correct for every adopter (a no-op for those who do not track `settings.json`), and the decision is a universal "where confinement state lives" boundary extending ADR-008. A PRJ would mis-scope a universal invariant as a project quirk (COR-014).

## Implications

- **ADR-008 rule 4 is realized by construction; rule 2 is clarified, not changed; no supersession.** Rule 4's never-committed guarantee now holds mechanically for every adopter. Rule 2's single-writer discipline is preserved and read as single-writer-with-routing. ADR-008's boundary-effect organising principle (rules 1–6) stands unchanged. No `superseded_by` frontmatter on ADR-008; no foundational re-authorisation. **ADR-008 rule 4 gains a one-line forward-pointer to this record.**

- **The live rule-4 violation is removed as part of the implementing work.** The kit's tracked `.claude/settings.json` currently carries `excludedCommands: ["uv"]` — the live violation #259 names. The implementing change removes it from the committed file; the routing re-applies it to `.claude/settings.local.json` on the next `setup autonomy` / `sandbox` run (the macOS required exclusion of ADR-027 lands in the gitignored file by construction). The committed file ends with the narrowing floor only.

- **`denyRead` floor + deviations resolved by the array-merge.** The committed baseline floor and local operator deviations merge to the union at runtime (verified harness fact); the floor cannot be weakened by the local file. No split-brain.

- **Provenance read must union both files.** The disable / seal / self-heal provenance read (ADR-008 rule 2; ADR-027 condition 6) must take the union across `settings.json` and `settings.local.json`, since the writer now lands authored entries in whichever file the entry's classification dictates. This is a recorded implementation constraint, not a behaviour change to teardown semantics.

- **COR-002 layering honoured; merge primitive untouched.** The split maps onto COR-002's baseline / project-additions / local tiers; the merge baseline (`core/settings.json`) carries no sandbox block, so the merge path is not touched. This is a sandbox-writer change, not a merge-primitive change.

- **Implementation is separate work, not this ADR.** The routing logic in the single sandbox-block writer, the provenance union-read, the removal of the committed `excludedCommands: ["uv"]`, the ADR-008 forward-pointer, README surface updates clarifying per-machine vs committed state, and tests land in the implementing issue (#259, branch `fix/259-move-mutable-sandbox-state-out`). This record sanctions building them on acceptance.

- **Versioning.** Behaviour-observable surface change (the sandbox writer now routes widenings to `settings.local.json`; committed `settings.json` no longer carries widenings) → the implementing change bumps `.pkit/VERSION` per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md). This decision record alone does not bump.

- **Migration.** No file/directory rename or removal in a kit-owned tree, no `schema_version` bump, no breaking CLI signature change, no capability subtree restructure — the writer gains a routing step and a target file. Whether the move of an *existing* committed widening to the local file warrants a one-shot idempotent migration (relocate-and-strip on already-installed adopters who committed a widening) is a COR-010 judgement for the implementing change; `pkit migrations check-diff --include-working-tree --base main` gates it. The ADR file itself is migration-clean.

- **Cross-platform and cross-harness.** The decision is the harness-neutral principle "mutable boundary-lowering state lives where it cannot be committed, the shared floor lives where it ships" realised through Claude Code's `settings.json` / `settings.local.json` deep-merge. Each future adapter realises the same split through its own committed/local settings layering (or declares the residual gap if its harness has no local tier), per COR-028's honesty discipline.
