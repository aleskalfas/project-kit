## Summary

The two existing permission axes govern *what an agent may do* (intent — [ADR-016](ADR-016-capability-contributed-intent-grants.md)) and *what hosts a session may reach* (confinement — [ADR-015](ADR-015-command-declared-network-egress.md)). Neither governs a third thing: a capability *deliberately sending sensitive local data off the machine, under its own outbound credential, to a destination it controls*. That is a distinct trust decision — it crosses a trust domain — and it needs its own record.

**Decision:** a capability **may** introduce a designed data-upload path with a second outbound credential, **provided** six invariants hold — the credential is isolated and never shipped, the user is shown-and-acknowledges what leaves the machine before any outbound call, sensitive payloads can only reach a *private* destination, the egress path is structurally separated from the tool's offline core, an offline/unconfigured run degrades to a local file rather than silently dropping or silently sending data, and every surface carries the same "this is a data-upload path, not a security boundary" honesty gloss [ADR-015](ADR-015-command-declared-network-egress.md) already mandates for egress.

This completes a deliberately-symmetric trio of capability-contribution axes: **intent** (ADR-016) / **confinement** (ADR-015) / **data-egress** (this). It supersedes nothing. The motivating instance is the reframed `user-feedback` design — a *deterministic* `tn feedback` command that bundles diagnostics (carrying sensitive copyrighted document excerpts) and files them into a private feedback repo under a scoped PAT. The detailed decision, the honest residual, and the rejected alternatives follow.

## Context

The forcing question (task #94, under EPIC #92) is whether project-kit's capability contract should permit a capability to introduce a path that **deliberately sends sensitive local data off the machine under a second credential** — and if so, under what invariants. This is not covered by any accepted record:

- [ADR-004](ADR-004-autonomy-intent-confinement.md) splits agent autonomy into **intent** (what an agent may invoke) and **confinement** (what a session may reach); both are *agent-action* axes. Its network-egress residual — `permission-enforcement.yaml` keeps `network-egress: enforcement: none`, the honest gap-truth [ADR-015](ADR-015-command-declared-network-egress.md) restates — records that pkit *does not confine* egress as a security boundary. But ADR-004 governs what an *agent* does within a session; it says nothing about a tool's own deliberate data-upload to a destination *it* owns under a credential *it* carries.
- [ADR-015](ADR-015-command-declared-network-egress.md) lets a capability declare which hosts a session may *reach* (the confinement axis). Reachability is not the same as a designed upload: ADR-015 governs the box's network posture, not a tool deliberately moving a user's private documents to a private repo.
- [ADR-016](ADR-016-capability-contributed-intent-grants.md) lets a capability contribute *intent* grants (what an agent may invoke). A deterministic upload command is not an agent action and carries no intent grant.

So a capability that wants to ship a deliberate data-egress path sits in an **unrecorded trust region**: it moves sensitive data off the machine under a second outbound credential, which none of ADR-004/015/016 sanctions or bounds. Introducing such a path is a *trust-domain change* — it expands what leaves the user's machine and under whose credential — and a trust-domain change needs an explicit, recorded decision, not an implicit ride on an existing axis.

**Motivating instance.** The reframed `user-feedback` design surfaced the need. Concretely, the `tn feedback` command in an adopter tool (trustworthy-notes / the summarizer workstream):

- is a **deterministic CLI command, not an autonomous agent** — the reframe away from a dispatched feedback agent is precisely what removed the agency/hijack surface the earlier shape carried (see the residual below);
- captures diagnostics and builds a **repro bundle** that, by its nature, **carries sensitive copyrighted document excerpts** (the user's own private working material);
- uses an **LLM as one structuring step** (with a raw-text fallback when the LLM is unavailable) to turn raw diagnostics into a readable report;
- files an issue and commits the bundle into a **separate, PRIVATE feedback repository**, via a **fine-grained PAT scoped to that one repo** (Issues + Contents, with expiry), delivered **out-of-band** (e.g. 1Password) and **never embedded in the shipped binary**, revocable at any time.

The reframe to a deterministic command is what made this safe enough to sanction: there is no autonomous-agent dispatch surface for the feedback text to hijack into actions. What remains is a *data* surface, not an *agency* surface — and the invariants below are written to bound exactly that.

This ADR records project-kit's **methodology capability contract** for the axis. The concrete instance's specifics (tn's exact bundle contents, its repo name, its PAT delivery mechanics) live in **that adopter's own decision space**, not here. As project-kit's own architecture-decision record, harness/credential/platform specifics are in scope here, unlike the harness-neutral COR. The operator has signed off the axis and the invariants. `status: proposed` is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md); accept before any implementation cites it.

## Decision

**A capability MAY introduce a designed data-egress path — a deliberate upload of local data to a destination the capability controls, under a second outbound credential — subject to the six invariants below. The invariants are the contract; a capability that cannot meet all six does not get the path.**

**Invariant 1 — Second-credential isolation.** The outbound credential is **scoped** (a single destination resource, the minimal permissions for the job, an expiry), **delivered out-of-band** (not in the product), **stored in adopter-local config** (outside the repo and outside any distributable artifact), **never embedded in a shipped or distributable artifact**, and **revocable**. The credential is a distinct, lower-privilege principal from anything the user already holds — it cannot be the user's own broad token.

**Invariant 2 — Data-egress consent boundary.** Any path that moves user data off the machine **shows what is being uploaded and obtains acknowledgement**, and does so **upstream of every outbound call** — both the LLM structuring call *and* the upload. The consent point is before data leaves the process, not after. The user always sees *what* leaves before it leaves.

**Invariant 3 — Destination-privacy (hard invariant).** Sensitive payloads land **only** in a **private** destination; the second credential is scoped so it **cannot reach a public one**. This is a **hard invariant, not a per-adopter toggle** — it is the property that keeps PII and copyrighted excerpts closed. A capability cannot offer a "post to a public tracker" mode for sensitive payloads; the credential's scope makes the public destination structurally unreachable.

**Invariant 4 — Pipeline isolation.** The data-egress path is **structurally isolated** from the tool's deterministic / offline core: **the core never depends on (imports or invokes) the egress path** — the dependency points one way, outward to egress, never inward. (In the tn instance: `cli → feedback`, never `core → feedback`.) This keeps the core's **zero-network, zero-second-credential** property **verifiable** — one can inspect the core in isolation and confirm it neither reaches the network nor touches the second credential.

**Invariant 5 — Graceful degradation.** Offline, expired-credential, or missing-credential runs **degrade to a local artifact** (the bundle written to disk for the user to send manually). Data is **never silently lost** and **never silently sent**. Failure of the egress path is a visible fall-back to local, not a silent drop and not a silent retry that leaks.

**Invariant 6 — Honesty gloss.** Wherever the egress is surfaced, it carries the **same honesty gloss [ADR-015](ADR-015-command-declared-network-egress.md) mandates verbatim for egress**: *this is a data-upload path; it is not a security boundary (no TLS inspection).* The user is told plainly that the path moves their data and that the transport is not an exfil control — the [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) honesty-about-gaps obligation applied to deliberate egress, at the bar ADR-015 set.

**The residual (recorded honestly — this is not a zero-risk path).** The LLM-structuring step is a **bounded data-injection / second-order-injection surface, not an agency one**. Because the command is deterministic, the LLM cannot *act*; but its output is committed to the feedback repo and shown to the maintainer, so a crafted document excerpt could shape that output. The governing invariant is therefore: **the LLM's output is untrusted display text downstream — never instructions.** It is rendered and stored as content the maintainer reads, never interpreted as commands by any downstream agent or tool. And invariant 2's consent boundary **must cover the LLM call**, not only the upload — because the LLM call is itself an outbound transmission of the sensitive excerpts.

## Rationale

**Why this is a new axis and not a ride on ADR-015 or ADR-016.** Reachability (ADR-015) and intent (ADR-016) are both *agent-action* framings: what a session may reach, what an agent may invoke. A deliberate data-upload under a second credential is neither — it is a *tool* moving *data* the user owns to a destination the *capability* owns. Folding it into ADR-015 ("just allow the host") would let the egress in without recording the consent / isolation / destination-privacy obligations that make it safe, and would conflate "a session may reach host X" with "this tool deliberately ships your private documents to host X under its own credential." Those are different trust decisions; the second deserves its own record so a future surface cannot obtain a sensitive-data upload path by quietly declaring a host. The trio is deliberately symmetric — intent / confinement / data-egress — so that each distinct trust decision has its own home and none is smuggled through another's channel.

**Why second-credential isolation (invariant 1).** A data-egress path needs to write to a destination, which needs a credential; the failure mode is a broad, long-lived, shipped credential that becomes an exfil and lateral-movement vector. Scoping (one resource, minimal permission, expiry) bounds the blast radius; out-of-band delivery and never-in-the-artifact keep it out of the distributable (a credential in a shipped binary is a credential leaked to every user); adopter-local storage keeps it outside the repo and outside provenance. This is the same single-purpose, minimal-privilege discipline the permission model applies to agent reach, applied to the tool's own outbound principal.

**Why the consent boundary sits upstream of every outbound call (invariant 2).** The user must learn *what leaves* before it leaves, or consent is fiction. Placing the show-and-acknowledge after the upload, or before only the upload but after the LLM call, would let sensitive excerpts reach the LLM endpoint before the user ever saw the payload — a silent first egress. Upstream-of-every-outbound-call is the only placement that makes "you always see what leaves before it leaves" true for both transmissions.

**Why destination-privacy is hard, not a toggle (invariant 3).** This is the load-bearing invariant. The entire point of the path is that sensitive material (PII, copyrighted excerpts) stays closed; a per-adopter "make it public" toggle would be a single config flip away from publishing exactly the material the path exists to protect — and worse, a flip that looks innocuous. Making the *credential scope* enforce private-only (the credential cannot reach a public destination) means the closure is structural, not a discipline an adopter can accidentally disable. A toggle relocates the safety property into operator vigilance; a hard invariant keeps it in the mechanism.

**Why pipeline isolation (invariant 4).** The core's value proposition — a deterministic, offline tool — is only credible if it is *verifiably* offline. If `core → feedback` were allowed, the core would transitively import the network and the second credential, and "the core does not phone home" would become an un-auditable claim. The `cli → feedback`-only dependency direction keeps the network and the credential out of the core's import graph, so the offline property is inspectable in isolation. This is the same dependency-direction discipline [ADR-003](ADR-003-permission-core-code-home.md) applies to keep the neutral decision core free of adapter edges, applied to keep the offline core free of egress edges.

**Why graceful degradation (invariant 5).** Two dishonest failure modes must both be foreclosed: silent loss (the user's feedback evaporates when offline) and silent send (the data goes out anyway through some fallback the user did not see). Degrading to a local artifact is the honest third path — the user keeps their bundle and chooses what to do with it. This mirrors the broader fail-visible posture: a failure is surfaced and handed back to the operator, never hidden.

**Why the honesty gloss (invariant 6).** [ADR-015](ADR-015-command-declared-network-egress.md) already settled that egress surfaces must say "not a security boundary; no TLS inspection," verbatim, because the transport allowlist is not an exfil control. A deliberate data-upload is *more* sensitive than session reachability, so the same gloss is the floor, not a nicety: the user is told plainly the path moves their data and the transport does not confine it. Softening it would be the [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) dishonesty ADR-015 and ADR-014 forbid.

**Why record the residual rather than claim zero.** Claiming a data-egress path under an LLM-structuring step has *no* residual surface would be the overstatement COR-028 forbids. The honest framing is precise about *which* surface remains: the reframe to a deterministic command removed the *agency* surface (no dispatched agent to hijack into actions), leaving a *bounded data-injection* surface (a crafted excerpt shaping committed, maintainer-read output). Naming it as "LLM output is untrusted display text, never instructions" pins the mitigation so a future consumer does not re-interpret LLM output as commands and re-open an agency surface the reframe closed.

### Alternatives considered

- **Fold data-egress into ADR-015 ("just declare the host").** Rejected — reachability is not a deliberate sensitive-data upload. Routing it through the confinement axis would admit the egress without recording the isolation / consent / destination-privacy obligations that make it safe, and would let a future surface obtain a private-data upload path by quietly declaring a host. The distinct trust decision needs its own record.
- **Fold it into ADR-016 (an intent grant).** Rejected — a deterministic upload command is not an agent action and carries no intent grant; ADR-016 governs *what an agent may invoke*, not *what data a tool deliberately ships*. The framings do not fit.
- **Forbid capability data-egress entirely.** Rejected — the user-feedback need is real and legitimate (a non-technical end-user must be able to report a problem with their private material attached), and a blanket ban would push adopters to build the path *without* the invariants, off the record. Sanctioning it *under invariants* is safer than forbidding it and watching it appear anyway, unbounded.
- **Make destination-privacy a per-adopter toggle.** Rejected — it relocates the PII/copyright-closure property from the mechanism into operator vigilance, one accidental flip away from publishing the exact material the path protects. The hard invariant keeps the closure structural.
- **Keep the autonomous feedback-agent shape (the pre-reframe design) and govern it here.** Rejected upstream of this ADR — an autonomous agent processing attacker-influenced feedback text is an *agency* surface (prompt-injection into actions), a materially larger trust surface than a deterministic command. The reframe to `tn feedback` is a precondition of sanctioning the path at all; this ADR governs the deterministic-command shape, and the residual section records exactly what the reframe did and did not remove.
- **Build the `user-feedback` capability packaging now (a skill / scaffolding for the path).** Rejected as premature per [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md) — there is one consumer today. The extracted-now part is *this ADR and the trust model* (the contract every future consumer must meet); the capability packaging waits for a second consumer to earn the abstraction. See the deferral in Implications.

## Implications

- **Completes the capability-contribution trio; supersedes nothing.** Intent ([ADR-016](ADR-016-capability-contributed-intent-grants.md)) / confinement ([ADR-015](ADR-015-command-declared-network-egress.md)) / data-egress (this) are the three deliberately-symmetric axes by which a capability declares its own permission/trust needs. This ADR sits **beside** [ADR-004](ADR-004-autonomy-intent-confinement.md)'s egress-as-`enforcement: none` residual — it does not relabel that residual as enforced; it governs a *deliberate* upload the residual never spoke to. No prior decision is superseded.
- **Governs the methodology's capability contract, not a concrete instance's specifics.** This ADR records *the invariants every data-egress capability must meet* (pkit's architecture). A concrete instance's specifics — tn's exact bundle contents, its private-repo name, its PAT delivery — live in **that adopter's own decision space**, not here. The contract is universal ([COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md)); the instance is the adopter's.
- **Trust-domain change; operator sign-off recorded.** Introducing a data-egress path expands what leaves the user's machine and under whose credential — a trust-domain change that **requires operator sign-off**. That sign-off has been given for *this axis and these invariants*. Recording it here means a future surface cannot relitigate or silently widen the axis without a fresh, recorded sign-off: the gate is on the record, not in a session's memory.
- **The residual is recorded, not waved away.** The LLM-structuring step is a bounded data-injection surface; the governing invariant is "LLM output is untrusted display text downstream, never instructions," and the consent boundary (invariant 2) covers the LLM call as an outbound transmission. The reframe to a deterministic command removed the *agency* surface; this names what remains so a future consumer does not re-open it.
- **Deferred — the `user-feedback` capability packaging (COR-007).** A reusable skill / scaffolding for the data-egress path is **deferred to a second consumer**, justified by [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)'s recurrence-not-yet-met: one consumer today (tn's `feedback`). The extracted-now part is this ADR and the trust model; the packaging waits. Recorded so the build does not over-reach into speculative scaffolding, and so the second consumer is the trigger to revisit.
- **No migration.** No file/directory rename or removal in a kit-owned tree, no `schema_version` bump, no breaking CLI signature change, no capability-subtree restructure — `pkit migrations check-diff --include-working-tree --base main` is expected clean. This is a decision record, not a surface change to an installed adopter's state.
- **No version bump in this change-set.** A newly-recorded capability-contract axis is a surface change per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md), but the bump is left to lifecycle time (the maintainer runs the acceptance gate and the version bump together); this change authors the ADR file only, `status: proposed`, like [ADR-017](ADR-017-confinement-allowance-cannot-override-harness-deny.md).
