## Context

[ADR-004](ADR-004-autonomy-intent-confinement.md) split agent autonomy into intent × confinement and made confinement a session-wide axis pkit *manages but delegates* to the OS sandbox — pkit may write the `sandbox` block (§76: "a future confinement preset is pkit-authored against the `sandbox` block… writes only the `sandbox` block"), but the OS enforces it. [ADR-005](ADR-005-permission-profile-surface.md) built the *intent* half's deferred surface (profiles) and deliberately kept confinement **referenced, not written** by a profile. The sandbox writer (#274) was the *first* exercise of ADR-004 §76: `sandbox enable` writes `enabled` / `autoAllowBashIfSandboxed` / `failIfUnavailable` and a credential `denyRead` floor, additively, preserving operator keys. [ADR-007](ADR-007-setup-command-class.md) added the setup-command class; `permissions setup autonomy` composes the primitives into the operator's one-gesture path.

A gap remains, surfaced by real use. The OS sandbox confines writes to the project directory and reads away from credentials — but that breaks legitimate out-of-project tooling: `uv` needs to write `~/.cache/uv`; the SSH agent and a commit-signing helper need a unix socket; `docker` is sandbox-incompatible and `gh` fails TLS under Seatbelt. Today the operator hand-edits the `sandbox` block (`filesystem.allowWrite`, `excludedCommands`, `network` socket keys) to make autonomy usable — exactly the manual configuration pkit exists to remove. The requirement is to manage these allowances with commands that compose up into `setup autonomy`, so reaching autonomous-but-project-confined needs no editing.

A `critic` + `architect` consult established the load-bearing distinction this record is built on: **confinement allowances divide by their effect on the boundary, and the two halves must be governed differently.** A *narrowing* allowance (a build-cache `allowWrite`, a needed socket path) makes the box usable **without enlarging the agent's reach**; a *widening* allowance (`excludedCommands` carves a command **out** of the box to run fully unconfined; `enableWeakerNetworkIsolation`) **lowers the boundary**. Auto-applying or committing the widening class reintroduces the precise trap ADR-004 §39/§49 and ADR-005 rule 6 forbid — a committed file or a lockfile scan silently lowers the floor for every checkout while the operator believes `setup autonomy` confined everything ("believes a boundary holds that does not"). This ADR is the *confinement* half's deferred surface — the symmetric sibling to ADR-005's intent half — resolving ADR-004 §76. Proposed status is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md).

## Decision

**pkit gains a confinement-allowance surface, organised by the boundary-effect split: every allowance is classified `narrowing` (makes the box usable, no reach increase) or `widening` (carves reach out of the box). Narrowing allowances are managed data that may be detected, committed, and auto-applied by `setup autonomy`; widening allowances are only ever applied by a loud, per-invocation, never-persisted operator gesture, and are always reported as boundary reductions.**

1. **Confinement toolkits are data (new `confinement-toolkit` schema, COR-018).** A *toolkit* is a named bundle of the sandbox allowances one tool (or tool-group) needs — `uv`, `gh`, `docker`, `node`, … — with each entry **classified `narrowing` or `widening`** in the data itself. A new `confinement-toolkit` schema + JSON Schema companion under `.pkit/schemas/`; backbone ships a small known set, projects extend/override by same-name in the project tree (the privilege-catalog / profile shipped-data pattern). This is the confinement-axis peer of `privilege-catalog` (intent-axis); it belongs in neither `permission-config` (adopter pointer-state) nor `permission-profile` (kept confinement-free by ADR-005 §23).

2. **One sandbox-block writer, additive with provenance.** All `sandbox`-block mutation routes through a single internal helper shared by `sandbox enable` and the new verbs; there is exactly one place that knows the merge rule. Writes are **additive and provenance-tagged**: pkit records which entries it authored. `disable` removes **only pkit-authored entries no longer claimed by any enabled toolkit** — never an operator's hand-added `allowWrite`/`excludedCommands` entry (the ADR-002 §52 silent-deletion footgun, transposed onto the sandbox block). Without provenance, removal could not distinguish a toolkit's `~/.cache/uv` from the operator's; provenance is the invariant that makes clean teardown possible.

3. **Narrowing path — managed, detectable, committable, auto-applied.** `sandbox accommodate <tool>…` applies a toolkit's **narrowing** entries (additive, idempotent, provenance-tagged). `sandbox accommodate --detect` scans project lockfiles/manifests and applies the narrowing accommodations of recognised tools — safe to auto-apply *because narrowing-only*. The adopter's chosen set is recorded in `permission-config` (`confinement_accommodations: […]`, a new optional field — narrowing only); `setup autonomy` applies it (or runs `--detect` on first run) as part of standing up confinement.

4. **Widening path — explicit, loud, never persisted, always reported.** `sandbox exclude <cmd>` (and the weaker-TLS toggle) is the only way to apply a widening allowance. It carries a per-invocation "this command runs OUTSIDE the box — unconfined" banner, is **never written to a committed file**, is **never proposed by `detect`**, and is **never applied by `setup autonomy`**. It mirrors the existing `--dangerously-allow-unconfined` discipline. Every excluded command is reported by `sandbox status` and counted by `probe` ("N commands run outside the box") so the reduction is always visible. When `setup autonomy` meets a detected tool that *needs* widening (docker/gh), it does not do it silently — it names the explicit `sandbox exclude` gesture for the operator to run.

5. **Profile references a hint; config disposes; the goal applies; activate never writes.** A profile MAY carry `recommended_accommodations` (narrowing only) as a **documentation hint** surfaced by `profile show` and seeded into config by `setup autonomy` on first run. The **authoritative** list is `permission-config.confinement_accommodations`; the profile's field never causes a write on its own. `profile activate` stays nudge-only (ADR-005 §27, ADR-007 rule 1) — reading the field gives it no sandbox side-effect. Shipped data proposes; project config disposes (the `active_profile` precedent, ADR-005 §25).

6. **Read surface + honesty (COR-028).** `sandbox toolkit list` / `show <name>` render the toolkits with each entry marked narrowing or widening. A widening entry (notably weaker-TLS) carries an honesty gloss that it *widens* reach and that egress is not a security boundary here (ADR-004 §61). The command never lets the operator believe a widened boundary is confined.

## Rationale

**Why the boundary-effect split is the organising principle.** The whole permission subsystem rests on ADR-004 §49: a mechanism that widens the boundary must never do so on inference, because a believed-but-absent boundary is worse than none. Narrowing and widening allowances look alike as JSON keys but are opposite on exactly this axis — so governing them identically (the naive "manage all sandbox allowances" surface) would either over-restrict the safe majority or, fatally, auto-apply the dangerous minority. Splitting by effect lets pkit deliver the full "no manual editing" win for the safe class (build caches, sockets — the bulk of real pain) while keeping the dangerous class on the loud, auditable, per-operator path the architecture already mandates for `--dangerously-allow-unconfined`. The split *is* the safety property; it is encoded in the data (rule 1) so every command and `detect` enforce it mechanically, not by ad-hoc judgement.

**Why a new schema, and why it is not premature (COR-007).** The recurrence is concrete and present, not speculative: `sandbox enable` already automated one narrowing floor (the credential `denyRead`); the operator is *already* hand-editing `allowWrite`/sockets for uv/gh/docker — the second and third instances of the same shape. A `confinement-toolkit` schema extracts a demonstrated pattern and is the confinement-axis twin of the accepted `privilege-catalog`. The principles-not-inventory discipline applies to the *shipped set*, not the abstraction: ship the schema + writers + the two or three toolkits the near-term adopters actually need (uv, gh), and let the rest land demand-driven; the toolkit *concept* is the principle, the shipped bundles are illustrative.

**Why provenance is load-bearing, not a detail.** `sandbox enable`'s additive discipline only ever appends; teardown of *operator* keys is a no-op by design. The toolkit writer is the first thing that must *remove* entries from shared operator-owned lists — and selective removal without knowing who authored an entry is the ADR-002 §52 Fork-1 silent-deletion footgun. Provenance-tagged writes (disable removes only unclaimed pkit-authored entries) is the structural answer, the sandbox-block analogue of ADR-005's layered grant-source.

### Alternatives considered

- **One undifferentiated "manage sandbox allowances" surface (commit a `confinement_toolkits` list applied wholesale by setup).** Rejected — a committed file applying `excludedCommands` silently lowers the boundary for every checkout, the exact thing ADR-005 rule 6 forbade for profile YAML; relocating it to `config.yaml` + `setup` does not change the effect. The split exists to make this representable safely.
- **`detect` proposes widening exclusions from lockfile presence.** Rejected — inferring "carve docker out of the box" from a Dockerfile's existence is the §49 trap mechanized. `detect` proposes narrowing only.
- **Hardcode a narrowing floor in `sandbox enable`; no schema until a second adopter (the critic's minimal v1).** Considered and not taken *for this record's scope*: the recurrence is judged already met (rule 1 rationale), and the product owner chose the full data-driven surface now. The hardcode path remains the honest fallback if the schema proves over-built; the build keeps the shipped toolkit set minimal to bound that risk.
- **Fold confinement-allowance data into `permission-config` or `permission-profile`.** Rejected — config is adopter pointer-state and the profile is deliberately confinement-free (ADR-005 §23); both would conflate the two axes ADR-004 separated.

## Implications

- **Child of ADR-004, sibling of ADR-005; no decision modified.** This resolves ADR-004 §76's deferred confinement-write surface the way ADR-005 resolved the intent half. It does not modify ADR-004 or ADR-005 (no supersession), and ADR-007's contract is unchanged — `setup autonomy` gains a narrowing-apply step, not a new contract (rule 2 "owns nothing" holds: every effect is a primitive's, individually reversible; widening never rides the orchestrator, per rule 4 and ADR-007 rule 6).
- **New `confinement-toolkit` schema (COR-018) + two field additions.** `permission-config` gains optional `confinement_accommodations`; `permission-profile` gains optional `recommended_accommodations`. Optional additive fields are backward-compatible — confirm whether they bump each schema's `schema_version` (and thus trigger a COR-010 migration) during the build; the new schema itself is a pure addition. Surface change for versioning (new CLI verbs + new schema) → bump per PRJ-002.
- **Single sandbox-block writer.** `sandbox enable`, `accommodate`, `exclude`, and `disable` mutate the block through one shared additive-with-provenance helper co-located in the existing permissions module (no new module — one consumer surface, one settings file).
- **`--strict` interaction is honest, not silent.** Under `--strict` (`allowUnsandboxedCommands: false`) `excludedCommands` is the only out-of-box path, raising the stakes on widening. The resolution is visibility, not prohibition: an excluded command is a deliberate, reported reduction (`sandbox status` + `probe` count it); `setup autonomy` never excludes silently, and `probe`'s verification is the backstop that the autonomy floor's claim matches reality.
- **Build sanction on acceptance.** Accepting this record sanctions building the surface: the `confinement-toolkit` schema + a minimal shipped set (uv, gh), the `sandbox toolkit list/show`, `sandbox accommodate [--detect]`, `sandbox exclude` verbs, the provenance-tagged shared writer, the config/profile fields, and the `setup autonomy` narrowing-apply integration — with tests, README surface rows, and a version bump.

> **Amendment ([ADR-014](ADR-014-macos-sandbox-platform-stance.md), #19) — the uv-cache narrowing accommodation is necessary-but-insufficient on macOS.** §13's premise that hand-accommodating `uv`'s `~/.cache/uv` write makes the box usable for pkit's own runtime holds on **Linux/WSL2 (bubblewrap)** but **not on macOS**. Under a forced Seatbelt-confined session the uv-cache `narrowing` accommodation got past the cache denial, but `uv` 0.9.8 then **panicked at its SystemConfiguration proxy probe** — a fixed, non-configurable `SCDynamicStore` mach-service denial in Claude Code's Seatbelt profile that no `narrowing` allowance and no env lever (`UV_OFFLINE`/`ALL_PROXY`/`NO_PROXY`) can satisfy. So on macOS the uv toolkit's `narrowing` entry is necessary but not sufficient: past the cache, only a **widening** `excludedCommands` exclusion (ADR-014, riding rule 4's loud/explicit/never-silent/always-reported path) — or a future fixed `uv` release behind a version floor — lets the `pkit`/`uv` command run. **Auto-accommodating the uv cache (#22) therefore helps only where `uv` can run confined (Linux/bubblewrap), not macOS** — there the cache accommodation is a no-op toward usability and the exclusion gesture is what stands up autonomy. This corrects §13's premise in place; it is **not** a supersession — the boundary-effect split (rules 1–6) stands unchanged, and ADR-014's macOS exclusion is exactly rule 4's widening path, not a new mechanism.
