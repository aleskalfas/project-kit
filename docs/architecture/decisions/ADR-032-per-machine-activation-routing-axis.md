## Summary

> **Correction (2026-06-25, issue #304).** This record originally classified `active_profile` as pure pkit state and asserted "`decide.py` is untouched" while also moving `active_profile` to a per-machine sidecar. Implementation surfaced that the two collide: the live `PreToolUse` hook reads `active_profile` from `config.yaml` *through* `decide.py`'s `load_model` (→ `_active_profile_grants`) to layer the profile's grants into the decision model, so a frozen `decide.py` reading the old location would stop layering the profile and silently regress enforcement (lenient-posture auto-allows become prompts). `active_profile` is therefore a **decision-model input**, not pure state-routing (the socket and `enabled` *were* genuinely pure routing — this is why `active_profile` was split from #303). The correction below reclassifies `active_profile` honestly and narrows the frozen-code claim to **"the `decide()` verdict path is byte-identical"**: `load_model` gains a minimal additive edit (read `active_profile` from the new sidecar, fall back to `config.yaml`), which is what makes "enforcement unchanged" *actually* achievable. The socket + `enabled` instances and Rules A+B are unchanged. Affected passages: the Decision's "Touch `decide.py`" alternative, the Implications "`decide.py` is untouched" bullet, and the `active_profile` worked-instance bullet.

[ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md) routes each sandbox-block entry to a committed or gitignored home by *one* axis — its **boundary effect** (a narrowing floor commits; a widening goes local). That axis is correct but incomplete: some permission/sandbox state is per-machine for a reason that has nothing to do with whether it narrows or widens the boundary, and routing it by effect alone sends it to the wrong file. The socket accommodation (`network.allowUnixSockets`) is *narrowing*, so ADR-029's effect axis routes it **committed** — baking a host-specific socket path (`/Users/<operator>/.1password/agent.sock`) into the tracked `.claude/settings.json`, a live regression against [ADR-010](ADR-010-host-environment-detection.md) rule 3's "never committed" guarantee for host facts. The sandbox toggle (`sandbox.enabled`) is split across a committed pkit key and the harness's own local key, which is the root of the believed-off-but-actually-on disable bug. And the autonomy pointer (`active_profile`) sits in tracked `config.yaml`, committing one operator's personal risk posture as a team default.

This record adds a **second routing axis** that composes with ADR-029's, expressed as two durable rules. **Rule A — class + composition:** a thing is *per-machine* when its value is **host-derived** (a fact only true on this machine) **or** its presence is an **operator activation** (per-operator state whose correct value varies by who is at the machine); route local if **either** this axis or ADR-029's effect axis says local. The class signal is data-driven from the existing `socket:ssh-agent` provenance tag, not a runtime guess. **Rule B — destination by ownership:** a per-machine entry routes to **its owner's** per-machine store — a **harness-co-owned** key goes to the harness's own local file and pkit defers (never authoring a parallel key), while **pkit-owned** state goes to a pkit-authored gitignored sidecar (the existing `runtime_ignore` pattern, not a new merge tier). Three known fields — socket, `enabled`, `active_profile` — are worked instances of Rules A+B, not three separate decisions; a fourth per-machine field falls under this record by classification, not by amendment. This is a **sibling to ADR-029, not an amendment** — its effect axis is unchanged; this adds a second axis OR-composed at the routing point. It **realizes [ADR-010](ADR-010-host-environment-detection.md) rule 3 by construction** (the socket can no longer reach a committed file) and corrects that rule's now-stale prose (post-ADR-029, `settings.json` is the committed floor; the per-machine file is `settings.local.json`). The genuinely-shared floor — the credential `denyRead`, `failIfUnavailable`, `autoAllowBashIfSandboxed`, `allowedHosts`, hooks/agents, and `config.yaml` policy — stays committed because it is **operator-invariant**: identical for every operator everywhere. An enforcement-policy layer (a committable "this project requires the sandbox") is considered and deferred per [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md). This record was authored `status: proposed` and **accepted** via operator sign-off (and re-signed-off for the correction note above) per the [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md) acceptance gate.

## Context

[ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md) settled the question "where does a sandbox-block entry live?" with a single discriminator: its **effect on the boundary**. A narrowing floor (the credential `denyRead`, `failIfUnavailable`, narrowing `allowWrite`/`allowedHosts`) ships committed so every checkout gets it; a widening (`excludedCommands`, the strict seal) routes to the gitignored `settings.local.json` so one machine's boundary reduction never lowers the floor for all. That axis is sound and stays exactly as decided.

But the effect axis is not the *only* reason a piece of state is per-machine, and three known fields expose the gap — each per-machine for a reason orthogonal to narrowing-vs-widening. They are the worked instances that motivate the two durable rules below; read them as evidence of the class, not as three separate decisions.

1. **The socket accommodation is narrowing yet host-specific.** [ADR-010](ADR-010-host-environment-detection.md) established host-environment detection as a *second detection-source axis*: the SSH-agent socket (`$SSH_AUTH_SOCK`, flowing to `network.allowUnixSockets`) is not a repo fact but a host fact, resolved per-session and — rule 3 — **written to per-machine settings only, never committed**, because the value does not exist on another machine. It is unambiguously *narrowing* (it makes the box usable without enlarging reach). So ADR-029's effect axis, seeing "narrowing," routes it to the committed floor — the precise wrong outcome ADR-010 rule 3 forbids. This is a **live regression**: with ADR-029's effect-routing in place, the narrowing socket lands in the tracked `.claude/settings.json`, baking a host path (`/Users/<operator>/.1password/agent.sock`) into committed state. (Corroborated on disk: the `socket:ssh-agent` provenance entry resolves to `/Users/kalfas/.1password/agent.sock` in `sandbox-provenance.yaml` — a host path with no business in a tracked file.) Effect alone is the wrong key for this entry; *value-provenance* is the missing one. **Instance reading:** host-derived value (Rule A) + pkit-owned destination (Rule B, sidecar branch).

2. **`sandbox.enabled` is split across two files and two owners.** pkit writes a committed `enabled`; the harness's `/sandbox` panel writes its own `enabled` into the gitignored `settings.local.json`. Two keys, two writers, one toggle — the source of the believed-off-but-actually-on disable bug, where `sandbox disable` flips the committed key but the harness-local key keeps the box on. The state is per-machine not because of its boundary effect but because **the operator activates it** and the harness already co-owns that activation locally. **Instance reading:** operator activation (Rule A) + harness-co-owned destination (Rule B, defer branch).

3. **`active_profile` commits a personal posture as a team default.** [ADR-005](ADR-005-permission-profile-surface.md) rule 3 put the `active_profile` pointer in the adopter-owned `config.yaml` as project state. But choosing the *autonomous* profile is an operator's personal risk/oversight choice — how much to let agents run unprompted on **this** machine under **this** operator's supervision — its correct value varies by who is at the machine, not a default the repo should ship to every checkout. Committing it makes one operator's autonomy stance everyone's. **Instance reading:** operator activation (Rule A) + pkit-owned destination (Rule B, sidecar branch).

The unifying observation: each of these is *per-machine*, but for a reason the effect axis cannot see — a host-derived value or an operator activation. That is a second, distinct routing axis that composes with ADR-029's effect axis by OR at the routing point: route local if *either* axis says local. The three instances share that class (Rule A) but reach their per-machine home by **different owners** (Rule B) — and that ownership distinction is the load-bearing structure, not the count of three.

The forcing question (issues #290 / #300, and the architect design review on them) is whether to handle these document-only (annotate where they live, rely on vigilance) or **structurally** (route them by a recorded second axis so the placement holds by construction). The operator decided structural. As project-kit's own ADR, harness and platform specifics are in scope here. This record is `accepted` (operator sign-off per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md), including the correction note in the Summary).

## Decision

**Sandbox/permission state is routed to its home file by *two* composed axes: ADR-029's boundary-effect axis and a new value-provenance + per-machine-activation axis. The per-machine axis is captured by two durable rules — a class-and-composition rule (Rule A) and a destination-by-ownership rule (Rule B). Both rules are load-bearing; the worked instances (socket / `enabled` / `active_profile`) are applications of them, not independent conditions.**

### Rule A — class and composition

**A thing is *per-machine* when its value is host-derived OR its presence is an operator activation; route it to the gitignored per-machine store if EITHER this axis or ADR-029's effect axis classifies it local. Commit only when BOTH axes classify it shared.**

- **Host-derived value:** the value is a fact true only on this machine ([ADR-010](ADR-010-host-environment-detection.md) rule 3) — it does not exist, or is wrong, on any other machine.
- **Operator activation — the decidable test:** *per-operator state whose correct value varies by who is at the machine or by their personal risk posture.* Its complement is a **portable floor:** state that is **identical for every operator everywhere**, which commits. Apply the test to classify; do not enumerate exceptions. (Worked: `autoAllowBashIfSandboxed` and `failIfUnavailable` are *operator-invariant* — true identically on every machine for every operator — so they are floors and commit, no special-casing required. `active_profile` and `enabled` vary by operator, so they route local.)
- **Composition is OR-to-local; ADR-029 is untouched.** ADR-029's effect axis classifies every entry exactly as before. The single sandbox-state writer routes an entry local if the effect axis says *widening* **or** Rule A says *per-machine*; it commits only when both say shared. A narrowing floor entry is still narrowing — this record adds an independent reason an entry may still route local. Sibling axis, not a re-classification.
- **The class signal is data-driven, not a runtime guess.** The `socket:ssh-agent` provenance tag already recorded in `sandbox-provenance.yaml` is the input that tells the writer "this entry's value is host-derived → route local." The necessity and boundary-effect **verdicts stay runtime** exactly where they already run ([ADR-008](ADR-008-confinement-allowances.md) classification; ADR-029 effect routing). This record adds a data-driven routing key; it moves no verdict into runtime inference.

### Rule B — destination by ownership

**A per-machine entry routes to *its owner's* per-machine store. Two branches by owner:**

- **Harness-co-owned key (the defer branch) → the harness's own local file; pkit defers.** When the harness already owns and writes a per-machine key, the entry lands where the harness writes it, and **pkit never authors a parallel key**. This is the split-brain fix: pkit writing its own copy of a harness-owned toggle is exactly what produced the believed-off-but-actually-on disable bug. The ownership distinction in this branch is the genuine decision — a harness-co-owned key must never be re-fused into a pkit store.
- **pkit-owned state (the sidecar branch) → a pkit-authored gitignored sidecar.** When the per-machine state is pkit's own, it routes to a gitignored per-machine file pkit alone authors, declared `runtime_ignore` — **reusing the existing `sandbox-provenance.yaml` sidecar pattern, not a new config-merge subsystem or ownership tier** ([COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md)'s local tier already exists; [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md): reuse the demonstrated pattern).

### Genuinely-shared state stays committed

State that is shared by *both* axes — shared effect AND operator-invariant under Rule A's test — stays in committed state: the credential `denyRead` floor, `failIfUnavailable`, `autoAllowBashIfSandboxed`, `allowedHosts`, the hook and agent wiring, and `config.yaml` policy (`ownership_mode`, `posture`, `schema_version`, `confinement_accommodations`). The Rule A test settles the near-neighbours directly: `autoAllowBashIfSandboxed` ("when sandboxed, auto-allow bash") and `failIfUnavailable` are identical for every operator everywhere, so they are floors, not activations — no enumerated defense needed.

### Enforcement-policy layer considered and deferred (COR-007)

A committable "this project *requires* the sandbox" enforcement policy — distinct from operator *activation* — is a real future option: a team that must mandate the box for every checkout would commit that requirement, and it would compose with Rule A (a committed requirement the local activation must satisfy). It is **deferred, not rejected**: per [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md), build it when a team concretely needs to mandate the box, not speculatively now. No second consumer exists today; activation suffices.

## Rationale

**Why a second axis rather than re-classifying entries under ADR-029.** The tempting fix is to call the socket "widening-ish" or otherwise bend the effect axis to route it local. That corrupts the effect axis — the socket genuinely *is* narrowing, and ADR-029's classification is correct. The honest model is that "where does this live?" has *two* independent inputs: what the entry does to the boundary (effect) and whether the entry is shareable across machines (Rule A's provenance/activation class). The socket is narrowing **and** host-specific; both facts are true, and the local routing follows from the *second* one. Modelling it as a composed second axis keeps each axis clean and makes the composition rule (OR-to-local) explicit, so a future reader does not try to reconcile the two into one over-broad discriminator.

**Why OR-to-local is the correct composition.** The whole subsystem's safety posture is "a per-machine fact must never become a committed default" ([ADR-010](ADR-010-host-environment-detection.md) rule 3, [ADR-005](ADR-005-permission-profile-surface.md) rule 6, [ADR-004](ADR-004-autonomy-intent-confinement.md) dp-4's family of "never silently lower the floor for every checkout"). Either reason — host-derived value or operator activation — is sufficient to make an entry unsafe to commit. So the safe composition is OR-to-local: if *any* axis says local, route local; commit only when *both* say shared. AND-to-local would let a host path commit merely because it is narrowing — the regression this record fixes.

**Why two rules, and what must not collapse.** Earlier framing named three members (socket / `enabled` / `active_profile`) as if each were its own decision. They are not: all three collapse to **Rule A (one class) + Rule B (one two-branch destination rule)**. What must **not** collapse is the **ownership distinction inside Rule B** — the harness-co-owned `enabled` reaching local via the harness's own file is a different owner than pkit's sidecar, and a future "unify the writers" refactor that re-fused harness-owned `enabled` into pkit's sidecar would reintroduce the split-brain bug. So the anti-refactor warning is precise and scoped: do not erase the owner boundary in Rule B; the count of instances carries no weight, and a fourth field is welcome by classification.

**Why the sidecar, not a new merge tier (COR-002 / COR-007).** pkit-owned per-machine state (Rule B's sidecar branch) could be handled by inventing a config-overlay/merge subsystem (committed `config.yaml` + a local `config.local.yaml` deep-merged). That is a new ownership tier and a new merge path — disproportionate to moving a pointer. [COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md)'s layering already names the local tier; reusing the existing gitignored sidecar pattern (`sandbox-provenance.yaml`, declared `runtime_ignore`) is the proportionate carrier ([COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md): reuse the demonstrated pattern, do not build a subsystem for one pointer). No new merge tier is introduced.

**Why this realizes ADR-010 rule 3 by construction.** [ADR-010](ADR-010-host-environment-detection.md) rule 3 promised the host-derived socket is "never committed." Pre-ADR-029 that held because `settings.json` was per-machine. ADR-029 made `settings.json` the *committed floor* and `settings.local.json` the per-machine file — which silently invalidated rule 3's mechanism for the socket (effect-routing now sends the narrowing socket to the committed floor). This record restores rule 3's guarantee *by construction*: Rule A routes the socket to the local file regardless of its narrowing effect, so the host path cannot reach a committed file on any code path. Rule 3's prose, written before ADR-029, references `.claude/settings.json` as the per-machine file — that reference is now **stale**; post-ADR-029 the per-machine file is `settings.local.json`. This record corrects it (see Implications forward-pointer).

**Why fail-closed is untouched.** [ADR-004](ADR-004-autonomy-intent-confinement.md) dp-4 governs *what happens when the sandbox is unavailable* (`failIfUnavailable: true` → refuse rather than run unconfined-and-unprompted). That control is operator-invariant under Rule A's test, so it is a committed-floor key, and it is orthogonal to where `enabled` lives. Moving `enabled` to the harness's local file (Rule B's defer branch) changes *whether the box is on*, not *what happens if it cannot start* — the fail-closed invariant is unaffected.

### Alternatives considered

- **Handle the per-machine fields document-only (annotate placement, rely on vigilance).** Rejected by the operator's decision and by the subsystem's own discipline — the socket regression is *live* (a host path in tracked state right now), and "holds only if everyone remembers not to commit it" is the precise vigilance failure ([ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md), [ADR-008](ADR-008-confinement-allowances.md) rule 4) the permission subsystem exists to eliminate. Structural routing closes it by construction.

- **Re-classify the socket as widening under ADR-029's single axis.** Rejected — the socket is genuinely narrowing; bending the effect axis to force a local route corrupts a correct classification. The honest model is a second, independent axis composed by OR.

- **AND-compose the two axes (commit unless both say local).** Rejected — would let a host-specific narrowing socket commit because it is narrowing, which *is* the regression. The safety posture demands OR-to-local: any single per-machine reason is sufficient to keep state out of committed files.

- **A new config-merge subsystem (`config.local.yaml` deep-merged over `config.yaml`) for pkit-owned per-machine state.** Rejected — a new ownership tier and merge path, disproportionate per [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md). The existing `runtime_ignore` sidecar pattern is the proportionate carrier; [COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md)'s local tier already exists.

- **Collapse Rule B's two branches — route every per-machine field through a single pkit-owned store.** Rejected — this is the one collapse that *would* break. A harness-co-owned key (e.g. `enabled`) re-fused into a pkit sidecar reintroduces the split-brain (pkit writes one place, the harness reads another). Rules A+B may absorb a fourth field freely, but the **owner distinction in Rule B** is invariant: harness-owned keys defer to the harness; pkit-owned keys go to the sidecar.

- **Build the enforcement-policy layer now (committable "require the sandbox").** Deferred, not rejected ([COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)) — no team yet needs to mandate the box across checkouts; operator activation suffices. It is recorded as a clean future composition (a committed requirement the local activation must satisfy) so it can land demand-driven without relitigating this axis.

- **Change the `decide()` verdict path.** Rejected — out of scope. The **`decide()` verdict path is byte-identical**; this is sandbox/permission *state routing*, not a verdict-logic change. One additive, minimal edit to `load_model` is required and *not* out of scope: because `active_profile` is a decision-model input (see the corrected worked-instance bullet), relocating it to the per-machine sidecar means `load_model` must read it from the new location (with a `config.yaml` fallback) so the profile's grants keep layering into the model. The original "freeze the whole file" framing was the error — a frozen `load_model` reading the old `config.yaml` location after the value moved would *break* enforcement, not preserve it. Constraining the freeze to `decide()` is what keeps enforcement unchanged.

## Implications

- **Worked instances of Rules A+B.** The three known per-machine fields each classify under the two rules, and the implementation realizes them accordingly:
  - **socket (`network.allowUnixSockets`)** — host-derived value (Rule A) + pkit-owned destination (Rule B's sidecar branch). Resolves per-session into the gitignored local file; the committed floor carries **at most a path-free intent marker** ("this project wants the SSH-agent socket accommodated"), **never the resolved value**. Fixes the live regression: the effect axis alone routed the narrowing socket committed; Rule A overrides that to local.
  - **`sandbox.enabled`** — operator activation (Rule A) + harness-co-owned destination (Rule B's defer branch). Lands in `settings.local.json` where the harness `/sandbox` panel already writes it; pkit retires its parallel committed key and defers. This unifies the split keys into one local key, so `sandbox disable` writes the single key the harness reads and disable reverses correctly. Committed default is off (opt-in).
  - **`active_profile`** — operator activation (Rule A) + pkit-owned destination (Rule B's sidecar branch). Moves from tracked `config.yaml` to a gitignored per-machine sidecar declared `runtime_ignore`; `config.yaml` keeps shared policy (`ownership_mode`, `posture`, `schema_version`, `confinement_accommodations`). **`active_profile` is a decision-model input, not pure state-routing** (unlike the socket and `enabled`, which are read only by the routing/state layer). The live `PreToolUse` hook reads it through `decide.py`'s `load_model` (→ `_active_profile_grants`) to layer the active profile's grants into the decision model. Relocating it therefore requires the additive `load_model` source-edit (read from the sidecar, fall back to `config.yaml` for adopters mid-migration); the `decide()` verdict path stays byte-identical. **Same-code invariant ([ADR-003](ADR-003-permission-core-code-home.md) — one decision core shared by hook and CLI) preserved and correctly per-machine:** because both the hook and the `pkit permissions` CLI go through `load_model`, reading the per-machine `active_profile` there keeps hook and CLI consistent *and* makes each operator's hook layer their own profile. This is the right design, not a workaround — and it is why `active_profile` was split out of #303 (the socket and `enabled`, being pure routing, conformed there; `active_profile` could not, because it feeds the decision core).

  A **fourth per-machine field** falls under this record by running the same two rules — classify by Rule A, place by Rule B's owner branch — not by amending it.

- **Socket resolve-replace is an ADR-010 rule 4 application note, not a new principle.** The socket resolves into the local file under recompute-replace — keyed by logical source + provenance, replacing the prior value rather than unioning — which is [ADR-010](ADR-010-host-environment-detection.md) rule 4 verbatim, merely applied to this entry. The local file is a session cache; `$SSH_AUTH_SOCK` plus the committed path-free intent is the source of truth. No new mechanic is introduced here.

- **Sibling to ADR-029; no supersession; ADR-029 gains a forward-pointer.** ADR-029's boundary-effect axis stands exactly as decided. This record adds a second axis OR-composed at the routing point. [ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md) gains a one-line forward-pointer to this record. No `superseded_by` frontmatter on ADR-029.

- **Realizes ADR-010 rule 3 by construction and corrects its stale prose.** The socket can no longer reach a committed file on any code path. [ADR-010](ADR-010-host-environment-detection.md) rule 3's file reference (`.claude/settings.json` as "the per-machine file") is stale post-ADR-029 — the per-machine file is now `settings.local.json`. ADR-010 rule 3 gains a forward-pointer/correction note. This is a documentation correction, not a decision change — rule 3's *guarantee* (never committed) is unchanged and now holds by construction.

- **Routing is data-driven at the write site.** The `socket:ssh-agent` provenance tag in `sandbox-provenance.yaml` is lifted into the routing decision as the host-derived-value signal. No new runtime verdict; the necessity/effect verdicts stay where ADR-008/ADR-029 already run them.

- **`enabled` unifies to one local key; the disable bug is closed.** pkit's committed `enabled` key is retired in favour of the single local `enabled` the harness `/sandbox` panel co-owns (Rule B's defer branch). `failIfUnavailable` stays committed; [ADR-004](ADR-004-autonomy-intent-confinement.md) dp-4 is unaffected.

- **`active_profile` moves to a `runtime_ignore` sidecar; `config.yaml` keeps shared policy.** A new gitignored per-machine sidecar (the `sandbox-provenance.yaml` pattern) holds `active_profile` (Rule B's sidecar branch); no new config-merge subsystem; no new ownership tier (COR-002 local tier reused).

- **Migration discipline — no-clobber relocation, convergence-on-next-run, one-shot strip on main.** The relocation MUST be **no-clobber** (the #288 lesson: a shipped strip that deletes working copies on pull destroys live values — preserve live values when moving them). Convergence happens on the **next `setup autonomy`** run (the writer relocates host socket / `enabled` / `active_profile` to their per-machine homes, writing the local destination before stripping the committed source so the runtime view never regresses mid-move — the ADR-029 relocation-pass discipline). The implementing PR additionally lands a **one-shot strip of the committed socket path on `main`** (the live regression must not persist on the tracked branch). Operator `_manual` / foreign entries are left untouched. Whether the relocation also warrants a one-shot idempotent lifecycle migration for already-installed adopters is a [COR-010](../../../.pkit/decisions/core/COR-010-resource-lifecycle.md) judgement for the implementing change; `pkit migrations check-diff --include-working-tree --base main` gates it.

- **The `decide()` verdict path is byte-identical; `load_model` gains one additive edit.** The verdict logic in `decide.py` does not change. Because `active_profile` is a decision-model input (it is layered into the model via `load_model` → `_active_profile_grants`), moving it to the per-machine sidecar requires `load_model` to read it from the new location with a `config.yaml` fallback — a minimal, additive source-edit, not a verdict-logic change. This is what makes "enforcement unchanged" actually achievable: a frozen `load_model` still reading the old `config.yaml` location after the value moved would silently regress enforcement (lenient-posture auto-allows degrade to prompts). Routing the read through `load_model` keeps the hook and the `pkit permissions` CLI on the same decision core ([ADR-003](ADR-003-permission-core-code-home.md)) and correctly per-machine. The socket and `enabled` instances touch no decision code — they are pure state-routing.

- **Versioning.** Behaviour-observable surface change (state now routes by a second axis; committed `settings.json`/`config.yaml` no longer carry host socket / split `enabled` / `active_profile`) → the implementing change bumps `.pkit/VERSION` per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md). This record alone does not bump.

- **Implementation is separate work, sanctioned on acceptance.** The second-axis routing in the single sandbox-state writer, the socket resolve-into-local + path-free committed intent marker, the `enabled` unification to the harness-co-owned local key, the `active_profile` sidecar (`runtime_ignore`), the no-clobber relocation pass on `setup autonomy`, the one-shot `main` socket strip, the ADR-029 + ADR-010 forward-pointers, README surface updates (per-machine vs committed state), and tests land in the implementing issue/branch. This record sanctions building them on acceptance.

- **Cross-harness.** The principle — "state that is host-derived or operator-activated lives where its owner keeps per-machine state; shared policy and the portable floor live where they ship" — is harness-neutral. Each future adapter realizes Rule B through its own local-settings / co-ownership mechanics (or declares the residual gap if its harness has no local tier), per [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md)'s honesty discipline.
