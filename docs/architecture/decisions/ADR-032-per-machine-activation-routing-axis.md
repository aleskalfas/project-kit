## Summary

[ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md) routes each sandbox-block entry to a committed or gitignored home by *one* axis — its **boundary effect** (a narrowing floor commits; a widening goes local). That axis is correct but incomplete: three pieces of permission/sandbox state are per-machine for a reason that has nothing to do with whether they narrow or widen the boundary, and routing them by effect alone sends them to the wrong file. The socket accommodation (`network.allowUnixSockets`) is *narrowing*, so ADR-029's effect axis routes it **committed** — baking a host-specific socket path (`/Users/<operator>/.1password/agent.sock`) into the tracked `.claude/settings.json`, a live regression against [ADR-010](ADR-010-host-environment-detection.md) rule 3's "never committed" guarantee for host facts. The sandbox toggle (`sandbox.enabled`) is split across a committed pkit key and the harness's own local key, which is the root of the believed-off-but-actually-on disable bug. And the autonomy pointer (`active_profile`) sits in tracked `config.yaml`, committing one operator's personal risk posture as a team default.

This record adds a **second routing axis** that composes with ADR-029's: route to gitignored per-machine state if **either** axis says local. The second axis is *per-machine activation*, and a thing is per-machine when its value is **host-derived** (a fact only true on this machine) **or** its presence is an **operator activation** (a personal choice, not a shared policy). Three members route local by this axis — but by three related-but-distinct routes, named precisely below so the "axis" is not over-generalized: the socket routes local because its **value** is a host fact; `enabled` routes local by **deferring to the harness's existing co-ownership** of that toggle; `active_profile` routes local as a **pkit-owned sidecar** because activation is a per-operator stance. The decision is data-driven at the write site by lifting the existing `socket:ssh-agent` provenance tag into the routing, not a runtime guess; the necessity/effect *verdicts* stay where they already run. This is a **sibling to ADR-029, not an amendment** — its effect axis is unchanged; this adds a second axis OR-composed at the routing point. It **realizes [ADR-010](ADR-010-host-environment-detection.md) rule 3 by construction** (the socket can no longer reach a committed file) and corrects that rule's now-stale prose (post-ADR-029, `settings.json` is the committed floor; the per-machine file is `settings.local.json`). The genuinely-shared floor — the credential `denyRead`, `failIfUnavailable`, `autoAllowBashIfSandboxed`, `allowedHosts`, hooks/agents, and `config.yaml` policy — stays committed. An enforcement-policy layer (a committable "this project requires the sandbox") is considered and deferred per [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md). The record lands `status: proposed`; operator sign-off flips it to `accepted` per the [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md) acceptance gate.

## Context

[ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md) settled the question "where does a sandbox-block entry live?" with a single discriminator: its **effect on the boundary**. A narrowing floor (the credential `denyRead`, `failIfUnavailable`, narrowing `allowWrite`/`allowedHosts`) ships committed so every checkout gets it; a widening (`excludedCommands`, the strict seal) routes to the gitignored `settings.local.json` so one machine's boundary reduction never lowers the floor for all. That axis is sound and stays exactly as decided.

But the effect axis is not the *only* reason a piece of state is per-machine, and three pieces expose the gap — each per-machine for a reason orthogonal to narrowing-vs-widening:

1. **The socket accommodation is narrowing yet host-specific.** [ADR-010](ADR-010-host-environment-detection.md) established host-environment detection as a *second detection-source axis*: the SSH-agent socket (`$SSH_AUTH_SOCK`, flowing to `network.allowUnixSockets`) is not a repo fact but a host fact, resolved per-session and — rule 3 — **written to per-machine settings only, never committed**, because the value does not exist on another machine. It is unambiguously *narrowing* (it makes the box usable without enlarging reach). So ADR-029's effect axis, seeing "narrowing," routes it to the committed floor — the precise wrong outcome ADR-010 rule 3 forbids. This is a **live regression**: with ADR-029's effect-routing in place, the narrowing socket lands in the tracked `.claude/settings.json`, baking a host path (`/Users/<operator>/.1password/agent.sock`) into committed state. (Corroborated on disk: the `socket:ssh-agent` provenance entry resolves to `/Users/kalfas/.1password/agent.sock` in `sandbox-provenance.yaml` — a host path with no business in a tracked file.) Effect alone is the wrong key for this entry; *value-provenance* is the missing one.

2. **`sandbox.enabled` is split across two files and two owners.** pkit writes a committed `enabled`; the harness's `/sandbox` panel writes its own `enabled` into the gitignored `settings.local.json`. Two keys, two writers, one toggle — the source of the believed-off-but-actually-on disable bug, where `sandbox disable` flips the committed key but the harness-local key keeps the box on. The state is per-machine not because of its boundary effect but because **the operator activates it** and the harness already co-owns that activation locally.

3. **`active_profile` commits a personal posture as a team default.** [ADR-005](ADR-005-permission-profile-surface.md) rule 3 put the `active_profile` pointer in the adopter-owned `config.yaml` as project state. But choosing the *autonomous* profile is an operator's personal risk/oversight choice — how much to let agents run unprompted on **this** machine under **this** operator's supervision — not a default the repo should ship to every checkout. Committing it makes one operator's autonomy stance everyone's.

The unifying observation: each of these is *per-machine*, but for a reason the effect axis cannot see. Two route local because they are **operator activations** (a personal choice, not shared policy); one routes local because its **value is host-derived** (true only here). That is a second, distinct routing axis — **value-provenance + per-machine-activation** — and it composes with ADR-029's effect axis by OR at the routing point: route local if *either* axis says local.

The forcing question (issues #290 / #300, and the architect design review on them) is whether to handle these three document-only (annotate where they live, rely on vigilance) or **structurally** (route them by a recorded second axis so the placement holds by construction). The operator decided all three go structural. As project-kit's own ADR, harness and platform specifics are in scope here. The record lands `status: proposed`; operator sign-off is the acceptance gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md).

## Decision

**Sandbox/permission state is routed to its home file by *two* composed axes: ADR-029's boundary-effect axis and a new value-provenance + per-machine-activation axis. The composition is OR at the routing point — an entry goes to the gitignored per-machine file if EITHER axis classifies it local; it stays in committed state only when BOTH axes classify it shared. A thing is per-machine when its value is host-derived OR its presence is an operator activation.** Three members route local by this second axis, by three related-but-distinct routes that the design names rather than collapses. Each condition is load-bearing.

1. **The composition rule: OR at the routing point; ADR-029 untouched.** ADR-029's effect axis classifies every entry exactly as before. The new axis runs alongside it. The single sandbox-state writer routes an entry to the gitignored per-machine file if the effect axis says *widening* **or** the per-machine axis says *host-derived/operator-activation*; it commits only when both axes say shared. ADR-029's classification of any entry is **not changed** — a narrowing floor entry is still narrowing; this record adds an independent reason an entry may still route local. This is a sibling axis, not a re-classification.

2. **socket (`network.allowUnixSockets`) — host-derived value; resolve into the gitignored local file, never committed.** The socket accommodation routes local by its **value provenance**: the value is a host fact ([ADR-010](ADR-010-host-environment-detection.md) rule 3). It resolves per-session into `settings.local.json` under recompute-replace ([ADR-010](ADR-010-host-environment-detection.md) rule 4 — keyed by logical source + provenance, replacing the prior value rather than unioning), and the committed floor carries **at most a path-free intent marker** ("this project wants the SSH-agent socket accommodated"), **never the resolved value**. The local file is a session cache; `$SSH_AUTH_SOCK` plus the committed path-free intent is the source of truth. This **fixes the live regression** in Context point 1: the effect axis alone routed the narrowing socket committed; the value-provenance axis overrides that to local, so the host path can never reach a tracked file.

3. **`sandbox.enabled` — operator activation by harness co-ownership; write to the gitignored local file, committed default off.** `enabled` routes local because it is an **operator activation** — but its route to *local* is distinct from `active_profile`'s: it lands in `settings.local.json` because that is **where the harness `/sandbox` panel already writes it**, and pkit **defers to that co-ownership rather than fighting the harness across files**. This **unifies** the currently-split committed-pkit-key-plus-local-harness-key into **one local key**, eliminating the believed-off-but-actually-on disable bug: `sandbox disable` writes the single local `enabled: false` the harness also reads, so disable reverses correctly. The committed default is *off* (no `enabled` in committed state; the box is opt-in per the existing enable gesture). **[ADR-004](ADR-004-autonomy-intent-confinement.md) dp-4's fail-closed invariant is unaffected** — file *location* is orthogonal to `failIfUnavailable`, which stays a committed-floor key (condition 6); moving where `enabled` lives does not touch what happens when the sandbox cannot start.

4. **`active_profile` — operator activation as a pkit-owned sidecar; move out of committed `config.yaml`.** `active_profile` routes local because **autonomy is a personal risk/oversight choice, not a committable team default** ([ADR-005](ADR-005-permission-profile-surface.md) rule 3's "shipped data proposes, project config disposes" — here the per-operator stance is the disposer, and it overrides the team-default reading). It moves from tracked `config.yaml` to a **gitignored per-machine sidecar declared `runtime_ignore`** — reusing the existing `sandbox-provenance.yaml` sidecar pattern, **not** a new config-merge subsystem. `config.yaml` keeps shared policy: `ownership_mode`, `posture`, `schema_version`, and toolkit-name accommodations. The sidecar's route to local is distinct again: not host-value (point 2), not harness-co-ownership (point 3), but a **pkit-owned per-machine file** that pkit alone authors.

5. **The routing is declared/mechanical, not a runtime judgement.** The value-provenance signal is **lifted from existing data**: the `socket:ssh-agent` provenance tag already recorded in `sandbox-provenance.yaml` is the input that tells the writer "this entry's value is host-derived → route local," so the value-provenance axis is **data-driven at the write site, not guessed at runtime**. The necessity and boundary-effect **verdicts stay runtime** exactly where they already run ([ADR-008](ADR-008-confinement-allowances.md) classification; ADR-029 effect routing) — this record adds a data-driven routing key, it does not move any verdict into runtime inference.

6. **Genuinely-shared state stays committed, and why each per-machine member differs from it.** These stay in committed state because they are shared by *both* axes (shared effect AND shared provenance): the credential `denyRead` floor, `failIfUnavailable`, `autoAllowBashIfSandboxed` (an autonomy *floor* — portable, identical on every machine, not an operator activation), `allowedHosts`, the hook and agent wiring, and `config.yaml` policy (`ownership_mode`, `posture`, `schema_version`). Each per-machine member differs from these on the new axis: the socket's **value** varies per host (the floor's values do not); `enabled` is an **operator activation co-owned by the harness** (`failIfUnavailable` is a portable floor, not an activation); `active_profile` is a **per-operator stance** (`posture` is the shared policy the profile is chosen *within*). `autoAllowBashIfSandboxed` in particular is committed-shared though it sounds activation-like: it is the portable floor that says "when sandboxed, auto-allow bash" — true identically everywhere — not a per-operator choice, so it stays committed.

7. **Enforcement-policy layer considered and deferred (COR-007).** A committable "this project *requires* the sandbox" enforcement policy — distinct from operator *activation* — is a real future option: a team that must mandate the box for every checkout would commit that requirement, and it would compose with this axis (a committed requirement that the local activation must satisfy). It is **deferred, not rejected**: per [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md), build it when a team concretely needs to mandate the box, not speculatively now. No second consumer exists today; activation suffices.

## Rationale

**Why a second axis rather than re-classifying entries under ADR-029.** The tempting fix is to call the socket "widening-ish" or otherwise bend the effect axis to route it local. That corrupts the effect axis — the socket genuinely *is* narrowing, and ADR-029's classification is correct. The honest model is that "where does this live?" has *two* independent inputs: what the entry does to the boundary (effect) and whether the entry is shareable across machines (provenance/activation). The socket is narrowing **and** host-specific; both facts are true, and the local routing follows from the *second* one. Modelling it as a composed second axis keeps each axis clean and makes the composition rule (OR-to-local) explicit, so a future reader does not try to reconcile the two into one over-broad discriminator.

**Why OR-to-local is the correct composition.** The whole subsystem's safety posture is "a per-machine fact must never become a committed default" ([ADR-010](ADR-010-host-environment-detection.md) rule 3, [ADR-005](ADR-005-permission-profile-surface.md) rule 6, [ADR-004](ADR-004-autonomy-intent-confinement.md) dp-4's family of "never silently lower the floor for every checkout"). Either reason — host-derived value or operator activation — is sufficient to make an entry unsafe to commit. So the safe composition is OR-to-local: if *any* axis says local, route local; commit only when *both* say shared. AND-to-local would let a host path commit merely because it is narrowing — the regression this record fixes.

**Why three distinct routes, named not collapsed.** It is tempting to declare a clean three-member "activation axis" where all three reach `local` by one mechanism. They do not, and pretending they do would mislead the implementer. The socket reaches local because its **value** is a host fact (resolve-and-cache into the local file, recompute-replace). `enabled` reaches local by **deferring to the harness's pre-existing co-ownership** of that key (pkit writes where the harness already writes, to end the split-brain). `active_profile` reaches local as a **pkit-owned sidecar** (pkit alone authors a `runtime_ignore` file). Same destination, three routes — host-value / harness-co-ownership / pkit-sidecar. Naming them keeps the implementation precise and prevents a future "unify the three writers" refactor that would re-fuse the harness's `enabled` co-ownership into pkit's sidecar mechanism and reintroduce a split.

**Why the sidecar, not a new merge tier (COR-002 / COR-007).** `active_profile` could be handled by inventing a config-overlay/merge subsystem (committed `config.yaml` + a local `config.local.yaml` deep-merged). That is a new ownership tier and a new merge path — disproportionate to moving one pointer. [COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md)'s layering already names the local tier; reusing the existing gitignored sidecar pattern (`sandbox-provenance.yaml`, declared `runtime_ignore`) is the proportionate carrier ([COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md): reuse the demonstrated pattern, do not build a subsystem for one pointer). No new merge tier is introduced.

**Why this realizes ADR-010 rule 3 by construction.** [ADR-010](ADR-010-host-environment-detection.md) rule 3 promised the host-derived socket is "never committed." Pre-ADR-029 that held because `settings.json` was per-machine. ADR-029 made `settings.json` the *committed floor* and `settings.local.json` the per-machine file — which silently invalidated rule 3's mechanism for the socket (effect-routing now sends the narrowing socket to the committed floor). This record restores rule 3's guarantee *by construction*: the value-provenance axis routes the socket to the local file regardless of its narrowing effect, so the host path cannot reach a committed file on any code path. Rule 3's prose, written before ADR-029, references `.claude/settings.json` as the per-machine file — that reference is now **stale**; post-ADR-029 the per-machine file is `settings.local.json`. This record corrects it (see Implications forward-pointer).

**Why fail-closed is untouched.** [ADR-004](ADR-004-autonomy-intent-confinement.md) dp-4 governs *what happens when the sandbox is unavailable* (`failIfUnavailable: true` → refuse rather than run unconfined-and-unprompted). That control is a committed-floor key and is orthogonal to where `enabled` lives. Moving `enabled` to the local file changes *whether the box is on*, not *what happens if it cannot start* — the fail-closed invariant is unaffected, and `failIfUnavailable` stays committed (condition 6).

### Alternatives considered

- **Handle all three document-only (annotate placement, rely on vigilance).** Rejected by the operator's decision and by the subsystem's own discipline — the socket regression is *live* (a host path in tracked state right now), and "holds only if everyone remembers not to commit it" is the precise vigilance failure ([ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md), [ADR-008](ADR-008-confinement-allowances.md) rule 4) the permission subsystem exists to eliminate. Structural routing closes it by construction.

- **Re-classify the socket as widening under ADR-029's single axis.** Rejected — the socket is genuinely narrowing; bending the effect axis to force a local route corrupts a correct classification. The honest model is a second, independent axis composed by OR.

- **AND-compose the two axes (commit unless both say local).** Rejected — would let a host-specific narrowing socket commit because it is narrowing, which *is* the regression. The safety posture demands OR-to-local: any single per-machine reason is sufficient to keep state out of committed files.

- **A new config-merge subsystem (`config.local.yaml` deep-merged over `config.yaml`) for `active_profile`.** Rejected — a new ownership tier and merge path for one pointer, disproportionate per [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md). The existing `runtime_ignore` sidecar pattern is the proportionate carrier; [COR-002](../../../.pkit/decisions/core/COR-002-merge-delivery.md)'s local tier already exists.

- **Collapse the three members into one uniform "activation axis" mechanism.** Rejected — the three reach `local` by distinct routes (host-value / harness-co-ownership / pkit-sidecar). A uniform mechanism would either fight the harness's `enabled` co-ownership or treat the host-derived socket as an activation it is not. Precision here prevents a future re-fusing refactor.

- **Build the enforcement-policy layer now (committable "require the sandbox").** Deferred, not rejected ([COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)) — no team yet needs to mandate the box across checkouts; operator activation suffices. It is recorded as a clean future composition (a committed requirement the local activation must satisfy) so it can land demand-driven without relitigating this axis.

- **Touch `decide.py`.** Rejected — out of scope. The intent-model decision path is untouched; this is sandbox/permission *state routing*, not a model-composition change.

## Implications

- **Sibling to ADR-029; no supersession; ADR-029 gains a forward-pointer.** ADR-029's boundary-effect axis stands exactly as decided. This record adds a second axis OR-composed at the routing point. [ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md) gains a one-line forward-pointer to this record. No `superseded_by` frontmatter on ADR-029.

- **Realizes ADR-010 rule 3 by construction and corrects its stale prose.** The socket can no longer reach a committed file on any code path. [ADR-010](ADR-010-host-environment-detection.md) rule 3's file reference (`.claude/settings.json` as "the per-machine file") is stale post-ADR-029 — the per-machine file is now `settings.local.json`. ADR-010 rule 3 gains a forward-pointer/correction note. This is a documentation correction, not a decision change — rule 3's *guarantee* (never committed) is unchanged and now holds by construction.

- **Routing is data-driven at the write site.** The `socket:ssh-agent` provenance tag in `sandbox-provenance.yaml` is lifted into the routing decision as the host-derived-value signal. No new runtime verdict; the necessity/effect verdicts stay where ADR-008/ADR-029 already run them.

- **`enabled` unifies to one local key; the disable bug is closed.** pkit's committed `enabled` key is retired in favour of the single local `enabled` the harness `/sandbox` panel co-owns. `sandbox disable` writes that one local key, so it reverses correctly. Committed default is off (opt-in). `failIfUnavailable` stays committed; [ADR-004](ADR-004-autonomy-intent-confinement.md) dp-4 is unaffected.

- **`active_profile` moves to a `runtime_ignore` sidecar; `config.yaml` keeps shared policy.** A new gitignored per-machine sidecar (the `sandbox-provenance.yaml` pattern) holds `active_profile`; `config.yaml` retains `ownership_mode`, `posture`, `schema_version`, and `confinement_accommodations`. No new config-merge subsystem; no new ownership tier (COR-002 local tier reused).

- **Migration discipline — no-clobber relocation, convergence-on-next-run, one-shot strip on main.** The relocation MUST be **no-clobber** (the #288 lesson: a shipped strip that deletes working copies on pull destroys live values — preserve live values when moving them). Convergence happens on the **next `setup autonomy`** run (the writer relocates host socket / `enabled` / `active_profile` to their per-machine homes, writing the local destination before stripping the committed source so the runtime view never regresses mid-move — the ADR-029 relocation-pass discipline). The implementing PR additionally lands a **one-shot strip of the committed socket path on `main`** (the live regression must not persist on the tracked branch). Operator `_manual` / foreign entries are left untouched. Whether the relocation also warrants a one-shot idempotent lifecycle migration for already-installed adopters is a [COR-010](../../../.pkit/decisions/core/COR-010-resource-lifecycle.md) judgement for the implementing change; `pkit migrations check-diff --include-working-tree --base main` gates it.

- **`decide.py` is untouched.** The intent-model decision path is not changed; this is sandbox/permission state routing only.

- **Versioning.** Behaviour-observable surface change (state now routes by a second axis; committed `settings.json`/`config.yaml` no longer carry host socket / split `enabled` / `active_profile`) → the implementing change bumps `.pkit/VERSION` per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md). This record alone does not bump.

- **Implementation is separate work, sanctioned on acceptance.** The second-axis routing in the single sandbox-state writer, the socket resolve-into-local + path-free committed intent marker, the `enabled` unification to the harness-co-owned local key, the `active_profile` sidecar (`runtime_ignore`), the no-clobber relocation pass on `setup autonomy`, the one-shot `main` socket strip, the ADR-029 + ADR-010 forward-pointers, README surface updates (per-machine vs committed state), and tests land in the implementing issue/branch. This record sanctions building them on acceptance.

- **Cross-harness.** The principle — "state that is host-derived or operator-activated lives where it cannot be committed; shared policy and the portable floor live where they ship" — is harness-neutral. Each future adapter realizes the per-machine axis through its own local-settings / co-ownership mechanics (or declares the residual gap if its harness has no local tier), per [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md)'s honesty discipline.
