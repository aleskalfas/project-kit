## Summary

[COR-036](../../../.pkit/decisions/core/COR-036-process-composition.md) decided the *rule* — a process may resolve one determinate inner process's terminal outcome, composition embeds that resolution via a `subprocess` state, cycles are forbidden, and the single-inner wait is `awaiting-subprocess-outcome`. This ADR records the *engine strategy* that was built to honour that rule, in `src/project_kit/process.py` ([ADR-020](ADR-020-process-engine-code-home.md)'s engine home), on branch `feat/199`. It is the on-call/maintainer reference for how cross-process resolution actually works and how it fails.

**The strategy, in one breath:** to learn an inner process's outcome, the engine builds a *fresh inner `ProcessEngine`* on the inner address and a determinate inner subject (`ProcessEngine.for_subject`), then asks that inner engine its *own* `resolve_position` and reads the terminal it landed on. As built, this is **single-level by construction (depth 1)**: `resolve_position` runs only the inner's *detection* predicates — it does *not* recurse through the inner's own `subprocess-outcome` gates — so composition reads one inner's detected outcome, one level deep, and never descends into a chain of inner-of-inner resolutions. The answer flows into the parent only through an engine-computed `subprocess-outcome` gate — the parent's own position stays its inferred subprocess-state position, never the inner's. A `_resolution_stack` of process addresses (seeded with the engine's own address, extended on the one inner load, checked before that load) is threaded through `for_subject`; **today it catches the direct self-embed (`A runs A`)**, and it is the correct extension seam for when resolution deepens to recurse through nested subprocess gates — at which point its transitive-cycle protection (`A runs B runs A`) becomes live.

Three things make this composition sound rather than a special case: the inner outcome is *itself* a definite-answer-over-reality produced by the inner's deterministic detection (so [COR-033](../../../.pkit/decisions/core/COR-033-process-substrate.md) P3 still holds — the parent's position is never stored or synthesized); resolution terminates today because it is bounded at single-level depth (the stack *guarantees* termination if/when resolution deepens — P6); and every cross-process answer enters through one engine-computed gate kind that mirrors the existing `authorisation-artifact` kind, so there is exactly one seam, not a scatter of synthesized positions.

## Context

COR-036 settled the rule but explicitly deferred the *how* to acceptance-gated implementation ("standing up the recursive resolution + acyclicity guard … are downstream tasks"). That implementation has now landed and introduces machinery with cross-cutting weight that future maintainers and on-call engineers need recorded: a new cross-process control-flow path inside an engine that was flat until now (single-level as built, with a threaded stack as the seam for deepening it), a new termination boundary, and a new engine-computed gate kind. ADR-020 is the sibling that decided *where* the engine lives (in the binary, invoked only as `pkit process …`); this ADR decides *how the engine resolves across a process boundary* and sits at the same altitude.

The decision space has four architecturally-significant choices, each of which had a plausible alternative the implementation rejected:

1. **How to obtain the inner outcome** — single-level engine instantiation (with a stack threaded as the deepening seam) vs. a flattened single-engine walk or a stored/cached outcome.
2. **How to guarantee termination** — a threaded per-invocation stack vs. a global visited-set or a depth limit.
3. **How the outcome reaches the parent's decision** — an engine-computed gate kind vs. a synthesized parent position or a capability predicate.
4. **What fails closed vs. what is a correct wait** — the failure-semantics taxonomy at the boundary.

## Decision

**In plain terms:** the engine answers "what outcome did the inner process reach?" by spinning up a second engine on the inner process and reading *its* answer, refusing to do so if that would loop forever, and feeding the answer back through one dedicated gate kind — never by inventing a position for the parent or trusting a capability to report the inner's state.

### 1. Single-level cross-process resolution

`ProcessEngine.resolve_subprocess_outcome(state_id)` is the genuinely-new capability. For a state carrying a `subprocess` declaration it:

- reads the inner `runs: <capability>:<process-id>` address and the determinate inner subject (a supplied keyed id, or `None` for a singleton inner);
- builds a *fresh* inner engine via `ProcessEngine.for_subject(inner_def, repo_root, inner_subject, resolution_stack=…)` — `for_subject` enforces COR-032's required-subject rule, so a keyed inner with no supplied subject fails closed there;
- calls the inner engine's *own* `resolve_position()` — the same deterministic detection every process uses — and reports the terminal it landed on as a `SubprocessResolution{address, outcome, indeterminate, reason}`.

**As built, this is single-level by construction (depth 1).** `resolve_position` runs only the inner's *detection* predicates; it does *not* recurse through the inner's own `subprocess-outcome` gates. So the engine reads the inner's currently-detected position and asks "is it terminal?" — it never descends into a chain of inner-of-inner resolutions. Composition reads one inner's detected outcome, one level deep. (Deepening resolution to recurse through nested subprocess gates is the seam section 2's threaded stack exists to make safe; it is not the shipped behaviour.)

This composes cleanly with COR-033 P3 (position is inferred from reality) precisely because the inner outcome is *not* a new kind of value: it is one more definite answer over reality, produced by the inner's deterministic detection, read live. The parent's own position stays its inferred subprocess-state position; the inner outcome is an *input* the parent reads, never the parent's stored or synthesized position. Resolution is strictly read-only — it runs detection predicates and reads journals, it never writes.

### 2. The acyclicity guard — a threaded resolution stack

Each engine carries `_resolution_stack: tuple[str, ...]` — the chain of process addresses on the active resolution path (its own address, plus the inner it is loading). It is **seeded with the engine's own address** when empty, **extended with the inner address on the inner load** (threaded through `for_subject`'s `resolution_stack=` parameter), and **checked before the inner definition is loaded**: an address already on the stack fails closed with the offending chain in its `reason`, surfaced exactly like an unrecognised gate kind.

**What the guard reaches today.** Because resolution is single-level (section 1) — it reads the inner's detected position and stops, never descending through the inner's own subprocess gates — the stack never deepens past the one inner being loaded. So in practice the guard **catches the direct self-embed (`A runs A`)**: the engine's own address is already on the seeded stack when it loads an inner equal to itself. A **transitive cycle (`A runs B runs A`) does not reach the guard today** — resolving `A` reads `B`'s detected position and stops, so the stack never gets back round to `A`. The transitive case is therefore safe *incidentally*, by bounded single-level depth, **not** because the guard fires. The threading beyond depth 1 is, as built, the correct seam standing ready rather than a live check.

**Why a stack, not a global visited-set — the design for the deepening case.** The stack is the right structure *for when resolution deepens* to recurse through nested subprocess gates, and it is cheap, correct insurance now. Two load-bearing reasons. First, *no leak across invocations*: the engine is stateless across calls (COR-033) and recomputes reality every time; a global set would either persist stale membership or need explicit reset, and would conflate two unrelated sibling resolutions under one parent (a process legitimately embedded twice in different branches is not a cycle). Second, *correctness of the cycle definition*: a cycle is a repeat on the *current resolution path*, which is exactly what a path-stack encodes — a visited-set would false-positive on a diamond (the same inner reached by two non-cyclic paths). When resolution deepens, this is the termination guarantee P6 needs: with no address revisitable on a path, the recursion is finite, so "what outcome did the inner reach?" still reduces to a definite answer. It is **distinct from COR-034's deferred `deadlock`** reason, which is a mutually-blocking cycle of *peer* subjects at runtime — a different graph over a different relation; this guard is a static-shape acyclicity check on the embedding graph.

### 3. The `subprocess-outcome` gate kind — engine-computed

The inner outcome reaches the parent's decision through a new `subprocess-outcome` gate kind, **computed by the engine** in `_evaluate_subprocess_gate`, not by a capability predicate. It resolves the inner embedded at the current state and passes iff the inner reached exactly the gate's named `outcome`. This deliberately **mirrors the `authorisation-artifact` kind**: in both, the engine computes the gate `result` itself rather than trusting a predicate's `result`, because the fact being gated is one the engine alone can establish soundly (cross-authority there; cross-process resolution here). The gate carries `outcome` and forbids a `predicate` (the resolution *is* the check) — the schema enforces that shape.

This was chosen over two alternatives. A **synthesized parent position** (making the subprocess state's position literally *be* the inner's outcome) was rejected: it breaks P3's inferred-from-reality guarantee, needs the parent's position to be stored or invented, and re-opens COR-034's rejected definition-state path. A **capability predicate** reporting the inner outcome was rejected: cross-process resolution is the engine's own machinery (COR-036 designs it once, in the engine), and a predicate would either re-implement the engine outside it or trust a capability to report another process's state — neither is sound. One engine-computed gate kind keeps a single seam.

### 4. Failure semantics

The boundary fails closed on every condition under which the inner outcome cannot be soundly established, and treats only one condition — a determinate inner that simply has not finished yet — as a correct wait:

- **Fail closed** (`indeterminate=True`, `outcome=None`, the reading gate refuses like an unrecognised gate kind): a cyclic embedding (address on the stack — at single-level depth, the direct self-embed `A runs A`); an inner address that does not resolve to a loadable definition; a keyed inner with no supplied subject (COR-032's required-subject rule, raised by `for_subject`); an indeterminate inner position (an inner detection predicate could not be evaluated); and a malformed gate (names no `outcome`, or sits on a state with no `subprocess`).
- **Correct wait** (`indeterminate=False`, `outcome=None`): a determinate inner that has not yet reached *any* terminal state. The parent is genuinely `awaiting-subprocess-outcome` — not an error. This wait is an auto-clearing overlay reusing COR-034's model, where the "condition" *is* the live single-level resolution carried by the `subprocess-outcome` gates (no `resume_when`); it clears the instant a wired outcome resolves and a legal move opens, detected by reusing `has_no_legal_move`.
- **Unwired-outcome correct-wait:** if the inner finishes at an outcome the parent's author never wired to an outgoing transition, the parent is *correctly* still `awaiting-subprocess-outcome` — it has no move its author gave it. This reflects an incomplete parent definition (the author owns outcome→transition wiring), not an engine bug, and is deliberately not special-cased.

### Boundaries — what this strategy is NOT

- **Single-resolution only.** It resolves *one* determinate inner subject and never enumerates a keyed inner's subjects. Enumerate-and-fold across many inner subjects ("how many reached `verified`?") is cascade's deferred breadth (COR-032's enumeration line); cascade will *consume* this resolution as its per-subject step, not replace it.
- **Nest/call only.** A parent embeds an inner and reads its outcome. The overflow / hand-off timing (a terminal state spawning or unblocking a concurrent peer process) is the orchestration altitude and is deferred (COR-033's named slot).
- **The parent's position is never the inner's.** The inner outcome is an input the parent reads through a gate; the parent's own position stays its inferred subprocess-state position. There is no stored, cached, or synthesized position anywhere in the path.

## Rationale

The central design tension COR-036 named is that a subprocess state's position "is another process's outcome," which most stretches P3's inferred-from-reality guarantee. The single-level-instantiation strategy resolves that tension by *not treating the inner outcome as a position at all*: it is a composed definite-answer-over-reality, obtained by running the inner's own deterministic detection, and it enters the parent only as a gate input. Determinism is preserved because the composition is single-level and so trivially terminates — the "composed definite answer" claim holds today on bounded depth. The acyclicity guard becomes load-bearing rather than defensive *the moment resolution deepens* to recurse through nested subprocess gates: at that point a cyclic embedding would have no definite answer (a non-terminating resolution has none), and the threaded stack is what restores the guarantee. Threading it now — when only the direct self-embed can actually fire it — is cheap insurance and the correct seam, not the thing that makes today's resolution terminate.

Reusing the engine on the inner (rather than flattening composition into one engine walk) keeps the inner process genuinely free-standing — it runs through exactly the same detection it would run standalone, knows nothing of its parent, and stays reusable under any parent. All coupling lives in the parent's `subprocess` declaration and `subprocess-outcome` gates, as COR-036's subroutine contract requires. A flattened walk would have had to special-case "am I inside an embedding?" throughout detection, leaking the parent relationship into the inner's resolution.

Mirroring `authorisation-artifact` for the gate kind is the consistency choice: the engine already had one precedent for "a gate the engine computes because only the engine can establish the fact soundly." Cross-process resolution is the second instance of that exact shape, so it takes the same form rather than inventing a third mechanism — keeping the gate-evaluation surface coherent (COR-007's recurrence applied to the gate-kind taxonomy).

### Alternatives considered

- **Flatten composition into a single-engine walk** (one engine that descends into embeddings inline). Rejected — it leaks the parent relationship into the inner's detection, breaks the inner's free-standing reusability, and loses the clean "ask the inner engine its own position" reduction that keeps P3 honest.
- **Cache / store the resolved inner outcome on the parent.** Rejected — storing it breaks P3's inferred-from-reality guarantee (it needs sync against the inner's live reality) and re-opens COR-034's rejected definition-state alternative. Resolving live keeps it a composed definite answer.
- **A global visited-set for cycle detection** instead of a path-threaded stack. Rejected as the structure to thread now (and to grow into when resolution deepens) — it leaks membership across the engine's stateless invocations, conflates independent sibling resolutions under one parent, and false-positives on a non-cyclic diamond (same inner reached by two paths). A path-stack encodes "repeat on the current path," which is the actual definition of a cycle. The choice is forward-looking: at single-level depth either structure would catch the direct self-embed, but the path-stack is the one that stays correct when resolution recurses through nested gates.
- **A depth limit** instead of an exact acyclicity check. Rejected — a depth bound is a heuristic that either rejects legitimate deep-but-finite embeddings or admits cycles below the bound; it does not give the termination *guarantee* P6 will need once resolution deepens. The stack gives an exact, fail-closed answer.
- **Expose the inner outcome as a synthesized parent position.** Rejected — breaks P3, needs a stored/invented position, re-opens the definition-state path COR-034 rejected.
- **Expose the inner outcome via a capability predicate** rather than an engine-computed gate. Rejected — cross-process resolution is the engine's own designed-once machinery; a predicate would re-implement it outside the engine or trust a capability to report another process's state.

## Implications

- **Engine control flow.** `src/project_kit/process.py` gains a cross-process path (`resolve_subprocess_outcome` → `for_subject` → inner `resolve_position`) in an engine that was flat until COR-036. As built it is **single-level**: `resolve_position` runs only the inner's detection, so the path bottoms out at depth 1 and terminates on bounded depth — it does not currently recurse through the inner's subprocess gates. Every engine instance carries `_resolution_stack`, seeded on construction with its own address and threaded onto the one inner load; this is the seam through which resolution would deepen, at which point the stack becomes the live termination guard. On-call note: today a cycle that surfaces is the direct self-embed (`A runs A`), reported as a fail-closed gate with the chain in the reason, not a stack overflow; the transitive-cycle (`A -> B -> A`) report becomes reachable only once resolution deepens.
- **Gate-evaluation surface.** `precheck_transitions` branches on `kind == "subprocess-outcome"` to `_evaluate_subprocess_gate` (engine-computed) before the predicate-backed `evaluate_gate` path; the existing `deterministic` / `authorisation-artifact` kinds are unchanged. A `subprocess-outcome` gate carries `outcome` and no `predicate` (schema-enforced).
- **Blocked overlay.** `evaluate_blocked` gains an `awaiting-subprocess-outcome` branch that reuses `has_no_legal_move` (no `resume_when`; auto-clears), refining COR-034's forwarded cross-subject reason to the single-inner case. The wait's audit colour (`_subprocess_wait_reason`) re-runs the live resolution for a human-readable status line; the live decider remains `has_no_legal_move`.
- **Failure-mode contract for on-call.** Indeterminate resolution (cycle / unloadable inner / keyed-without-subject / indeterminate inner position / malformed gate) refuses the move fail-closed and names why. A determinate-but-unfinished or determinate-but-finished-at-an-unwired-outcome inner is a *correct wait*, not a failure — the right response is to wire the missing outcome→transition or to wait, not to debug the engine.
- **Additive, no migration.** A process declaring no `subprocess` / `interface` / `subprocess-outcome` gate is byte-unchanged and unaffected; existing definitions validate as before (COR-010 — no migration owed).
- **Precedent for cascade.** Cascade (deferred) builds its enumerate-and-fold *on* this single-inner resolution — calling `resolve_subprocess_outcome` once per enumerated subject — rather than inventing a second cross-process mechanism. This ADR records the per-subject step cascade will stand on; the enumeration/fold and the many-inner aggregate wait remain cascade's own future decisions.
- **Relationship to records.** Records the engine strategy for COR-036 (the rule); sibling to ADR-020 (engine code-home); preserves COR-033 P3/P6 (inferred position, deterministic validator) — P3 because the inner outcome is read live from the inner's own detection, P6 because single-level resolution terminates on bounded depth today (the threaded stack carries the termination guarantee forward if resolution deepens); upholds COR-032 (single-subject — one inner resolution, the required-subject rule for a keyed inner, no enumeration); refines COR-034's `awaiting-subprocess-outcome` forwarding to the single-inner case; reuses the `authorisation-artifact` engine-computed-gate precedent for the `subprocess-outcome` kind.
