## Summary

**In plain terms:** [COR-041](../../../.pkit/decisions/core/COR-041-external-source-distribution.md) decided the *semantics* of the `externally-sourced` origin — content is pulled whole, pinned to a ref, and reconciled by matching the pin rather than merging. This ADR is the project-kit **write-path contract** that realises those semantics: exactly *how* `pkit` fetches that content from a git source at its pinned ref, where the fetched subtree lands in the consumer, how the pin is recorded, and in what order the fetch and the compatibility gate run. It is the distribution-side member of the write-path-ADR family, the one-altitude-down twin of [ADR-039](ADR-039-pkit-entry-point-router.md)'s whole-kit `uvx project-kit@<pin>` resolver — a *capability* fetched at a pin instead of the *whole kit*.

The load-bearing rule is an **ordering**: fetch-at-pin first, then read the *fetched* manifest, then run [COR-010](../../../.pkit/decisions/core/COR-010-resource-lifecycle.md)'s `requires_backbone` compatibility gate against the **consumer's** backbone, then activate or refuse. Compatibility now resolves against a manifest that did not exist locally until the fetch ran — a consumer cannot gate on content it has not yet pulled. Authentication for the interactive/local case leans wholly on [PRJ-004](../../../.pkit/decisions/project/PRJ-004-distribution-channel.md)'s existing GitHub auth (SSH / PAT); **unattended CI auth is explicitly out of scope** here, deferred to a PRJ successor exactly as PRJ-004 deferred its own cross-repo CI auth. And an unreachable source or an auth gap **degrades loudly and never bricks mid-command**, inheriting ADR-039's D2 posture verbatim.

## Context

COR-041 fixed the universal semantics of the `externally-sourced` origin and — per COR-025 — delegated the project-specific fetch/auth/land/record write-path to a realising ADR. This is that ADR. It carries the *how* that COR-041 deliberately does not: the channel, the authentication model, the landing path, the pin record, and the failure posture, all of which are project-kit's own mechanism rather than adopter-universal principle.

The consumer that grounds this work (per the #131 review) is an organisation privately sharing an adopter-owned capability — a proprietary discipline or a company house-style — across its own repositories, maintained in one authoring repo and consumed whole everywhere internal, without publishing to the public kit and without forking it. That same content is `incubated-in-repo` in its authoring repo and `externally-sourced` in each consumer (COR-041); origin is a per-repo relationship, not a property of the bytes.

Two existing decisions set the shape this contract inherits. PRJ-004 fixed project-kit's *whole-kit* distribution channel — direct git URL on `github.com`, pinned by tag/branch/SHA, authenticated by the developer's existing GitHub auth, no registry. This ADR reuses that channel one altitude down: a single capability subtree at a pinned ref instead of the whole wheel. ADR-039's D2 fixed the failure posture for its whole-kit pin resolver — an unresolvable pin degrades to running self with a loud drift warning, never a hard fail mid-command. Externally-sourced fetch faces the identical hazard class (offline, untagged ref, auth gap) and adopts the identical answer.

The critic and architect reviewers ran on the design in the #131 review that produced COR-041 and this contract.

## Decision

**`pkit` fetches externally-sourced content whole from a git source at its pinned ref, lands it in the consumer's own capability subtree under a single owner, records the pin in install state, and gates activation on the fetched manifest — degrading loud when the source is unreachable.** Seven points fix the write-path.

1. **Fetch-at-pin; the pin is the source of truth.** Content declared `externally-sourced` names a git source and a ref (tag, branch, or SHA — PRJ-004's pin forms, one altitude down). On sync/install `pkit` fetches the named subtree at exactly that ref. The pin is the single source of truth for what the consumer holds; **an upgrade is a repointing of the pin**, nothing more — there is no separate "update" verb, only a new pinned ref reconciled by the match-the-pin case COR-041 established. Reconciliation brings the consumer's copy to exactly the pinned ref's content; it is never a merge (COR-001 forbids the merge/conflict machinery, and the content lands under a single owner so no conflict can arise).

2. **Local fetch-auth leans on PRJ-004's existing GitHub auth; CI auth is out of scope.** For the **interactive/local** case — a developer running `pkit sync`/install against a private source — authentication is the developer's *existing* GitHub credential exactly as PRJ-004 fixes for the whole kit: SSH key via `~/.ssh/config`, or PAT via the git credential helper for the HTTPS form. The fetch introduces **no new credential surface**; it reuses the one the developer already has for `git clone`. **Unattended CI authentication to a private source is explicitly out of scope of this ADR** — a PRJ successor owns it, consistent with PRJ-004 having deferred its own cross-repo CI auth to a per-adopter operational concern. Naming it here as deferred keeps the local path shippable without blocking on the CI credential-store design.

3. **Landing + pin record.** The fetched subtree lands at the consumer's own `.pkit/capabilities/<name>/` — the same project-side path any capability occupies, held under a **single owner** (the pinned external source), so the no-shared-files invariant (COR-001) holds unchanged: no consumer-owned file shares that subtree, so match-the-pin reconciliation can overwrite it wholesale without destroying local work. The **pinned ref is recorded in install state** as non-derivable state per COR-010 — the ref cannot be re-derived from the consumer's config, so it belongs in the manifest (the same class as an opaque board UUID), and its diff is meaningful: repointing the pin is the visible upgrade gesture.

4. **Ordering (load-bearing): fetch → read fetched manifest → compatibility-gate → activate-or-refuse.** Compatibility resolution runs in a fixed order: (a) fetch-at-pin (point 1); (b) read the **fetched** content's manifest; (c) run COR-010's `requires_backbone` gate — the capability's declared range against the **consumer's** backbone version; (d) activate the capability, or refuse with the surfaced conflict for the adopter to address. The novel property this pins, and the reason ordering is load-bearing, is that **compatibility now evaluates a fetched manifest** — the manifest does not exist locally until step (a) completes, so a consumer cannot gate on content it has not yet pulled. This slots into COR-010's existing reconciliation order (compatibility resolution precedes any state change); the fetch is simply the step that *materialises* the manifest the gate then reads.

5. **The author owns the compatibility claim; the consumer validates it at fetch time.** `requires_backbone` on externally-sourced content is an **authored claim**, declared and maintained by the author against the backbone lineage the author tests on; the consumer's gate (point 4) validates it at fetch time. The author repo's own backbone version is irrelevant to consumers — only the declared range is. Critically, **the auto-broaden that this session built does not fire in an author/adopter repo**: `release.py`'s `_broaden_kit_components_requires_backbone` (via `versioning.py`) walks kit-shipped `package.yaml` files and broadens their upper bound, but it runs **only on a project-kit backbone release**, not on a capability release in an author's repo. The mechanism does not compose across repos. So for externally-sourced content the range is a hand-authored claim, not an auto-maintained one; author-side tooling to keep it current on capability release is tracked as #494. Until that lands, an author who ships a capability tested against a newer backbone must widen the range by hand or consumers on that backbone will be refused at point 4.

6. **Cross-source name collisions surface, never clobber.** When two external sources, or an external source and a kit-shipped or incubated capability, claim the same `<name>` at the point-3 landing path, `pkit` **surfaces the collision** for the adopter to resolve rather than silently overwriting one with the other. This extends COR-031's collision rule (an incubated capability whose name collides with a newly kit-shipped one is surfaced, not skipped) to the externally-sourced origin. The single-owner landing (point 3) makes a *clean* fetch conflict-free; this point covers the distinct case of two *claimants* on the same name, which is an adopter decision, not a fetch mechanic.

7. **Unreachable-source / auth-gap degradation inherits ADR-039 D2.** A source that cannot be reached at fetch time — offline, an untagged/moved ref, or an auth gap (the CI case point 2 defers) — **warns loudly and degrades; it never hard-fails mid-command**. `pkit` continues with the consumer's last-known-good pinned content already on disk (or, on first install where none exists, skips activation of that one capability with a loud warning and proceeds with the rest of the command). This is ADR-039's D2 posture verbatim, one altitude down: a degraded-but-running command beats a bricked one, and a fetch failure over one external source must not strand every other operation in the command.

## Rationale

**Why fetch-at-pin rather than track-a-branch-tip.** The pin-as-source-of-truth model is PRJ-004's registry-free minimalism inherited deliberately (COR-041's rationale). A moving branch tip would make "what does the consumer hold" unanswerable without a network round-trip and would smuggle upgrades in silently; a pinned ref makes the held content a local fact and makes upgrade an explicit, diffable repoint of one recorded value (point 3). This is the same reason ADR-039 pins the whole kit by ref rather than tracking latest.

**Why the ordering is the load-bearing decision.** Every other point is a placement or a posture; point 4 is the one that could be got *wrong* in a way that wedges a consumer. Reading a *stale or absent* local manifest — or reading the *author repo's* backbone rather than the consumer's — would either refuse compatible content or activate incompatible content. Fixing the order (fetch, then read the fetched manifest, then gate against the consumer's backbone) is what makes compatibility resolution correct for content that arrives over the network. COR-041 states the principle that compatibility evaluates a fetched manifest; this ADR pins the *sequence* that realises it.

**Why local auth now and CI auth later.** The grounded consumer (#131) is a developer running sync locally; that path is fully served by the GitHub auth PRJ-004 already assumes, at zero new credential surface. Unattended CI auth is a genuinely harder problem (credential-store shape, deploy keys, token scoping) that PRJ-004 itself deferred one altitude up. Shipping the local path without blocking on the CI design is the same staged move PRJ-004 made; folding CI auth in here would couple a shippable contract to an unsettled one.

**Why surface collisions rather than pick a winner.** A silent clobber at the shared landing path would let one source's content vanish under another's with no signal — the exact failure COR-031 rejected for incubated-vs-kit-shipped names. The adopter is the only party that knows which claimant they meant; the write-path's job is to make the conflict visible, not to guess.

**Why inherit ADR-039's degradation posture unchanged.** The failure class is identical (a pin that cannot resolve over the network) and ADR-039 already reasoned it to ground: a hard fail makes the tool *more* fragile than not having the feature, because a transient network gap would break unrelated work. Re-deriving a different posture here would fragment how project-kit handles pin-fetch failure across its two altitudes for no gain.

### Alternatives considered

- **Merge the fetched content into a consumer-owned region.** Rejected at the principle level by COR-041/COR-001 — it is exactly the merge/conflict machinery the methodology structurally refuses. This ADR could not have chosen it; the single-owner landing (point 3) is what keeps match-the-pin conflict-free.
- **Gate compatibility before fetching (read a cached or declared range).** Rejected — there is no local manifest to read until the fetch runs; a cached range goes stale against a repointed pin, and gating on the *author repo's* backbone instead of the *declared* range validates the wrong thing. Fetch-then-read-fetched-manifest (point 4) is the only order that gates on current, correct content.
- **Auto-broaden `requires_backbone` for externally-sourced content the way kit releases do.** Rejected as unavailable, not merely unwanted — the broaden mechanism runs only on a project-kit backbone release and does not compose into an author's capability-release flow (point 5). Pretending it applied would leave authors believing a range was maintained that never ran. The honest position is an authored claim plus tracked author-side tooling (#494).
- **Block on unattended CI auth before shipping.** Rejected — couples a shippable local path to PRJ-004's already-deferred CI-credential design. Naming CI auth as out-of-scope (point 2) ships the grounded consumer's path now.
- **Hard-fail on an unreachable source.** Rejected for the same reason ADR-039 D2 rejected it — a transient fetch failure over one source must not brick the whole command.

## Implications

- **`pkit`'s fetch/sync path gains the externally-sourced fetch step**, authored at implementation time once this record is accepted: fetch-at-pin (point 1), land at `.pkit/capabilities/<name>/` under single ownership (point 3), record the pin in install state as non-derivable manifest state (point 3), and slot the point-4 ordering into COR-010's existing reconciliation sequence. No code is pinned here; this is the design contract the implementation builds against.
- **The compatibility gate reads a fetched manifest** (point 4). This is the concrete realisation of the forward-pointer COR-041 added to COR-010 — compatibility resolution now evaluates content that arrives over the network, not only content already on disk.
- **`requires_backbone` on externally-sourced content is an authored claim** (point 5), and author-side tooling to keep it current on capability release is filed as **#494**. Until #494 lands, the range is maintained by hand; the auto-broaden in `release.py` / `versioning.py` does not fire outside a project-kit backbone release.
- **Unattended CI authentication is a deferred PRJ successor** (point 2), consistent with PRJ-004's own deferral of cross-repo CI auth. This ADR ships the interactive/local path only.
- **Name-collision surfacing extends COR-031's rule to this origin** (point 6); the lifecycle surfaces a same-named external/kit/incubated clash rather than skipping or clobbering.
- **Accepted by the maintainer before the trunk implementation builds against it** (per COR-041's realising-ADR delegation) — a forward design contract, not self-accepted. The fetch step, the pin record, the point-4 gate, the collision surfacing, and the point-7 degradation land citing this record once it is accepted.
- **Stands on** COR-041 (the origin semantics this realises — the *what* to this ADR's *how*), COR-010 (the `requires_backbone` gate and reconciliation order the point-4 sequence slots into), COR-001 (the no-merge / single-owner invariant point 3 rests on), PRJ-004 (the git-URL channel and existing-GitHub-auth model points 1/2 inherit one altitude down), and ADR-039 D2 (the degrade-loud failure posture point 7 inherits). This ADR pins the write-path contract, not COR-041's principle — it must not accrete or restate the origin semantics; on-call reference: if a consumer holds content that does not match its recorded pin, or is refused against a backbone the author's declared range should cover, point 4 or 5 is where to look first.
