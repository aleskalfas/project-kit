## Summary

[COR-040](../../../.pkit/decisions/core/COR-040-process-open-region.md) decided the *rule* — a process may declare a state an **open region**: one state with no internal edges, carrying region-scoped invariants and exactly one guarded outgoing transition, its **exit gate**. Inside the region the engine reduces to one deterministic question, "may this subject leave yet?". This ADR records the *engine strategy* that honours that rule, in `src/project_kit/process.py` ([ADR-020](ADR-020-process-engine-code-home.md)'s engine home), on branch `feat/418-implement-the-open-region-slot`. It is the on-call/maintainer reference for how a region-scoped invariant is read against the resolved position, how a region is enforced, and how a shut exit fails. It was flagged as owed at implementation time during the 2026-07-01 architectural review of COR-040, whose Engine implication names exactly this sibling ADR.

**The strategy, in one breath:** an open region needs **no new engine machinery** — it is composed entirely from parts that already exist. Region scoping is a filter: `evaluate_invariants` gains an `applies_to` read that narrows the process-wide invariant set to those whose scope names the subject's *currently resolved* position (`resolve_position().state_id`), leaving unscoped invariants process-wide exactly as [COR-035](../../../.pkit/decisions/core/COR-035-process-invariants.md) shipped them. That filter is the one genuinely-new coupling — between detection ("where is the subject?") and invariant reporting ("which always-checks apply here?") — and it stays a deterministic, single-subject, read-only pass. Enforcement is **gate composition, not a new engine coupling**: the region's sole outgoing transition is an ordinary checkable gate ([COR-033](../../../.pkit/decisions/core/COR-033-process-substrate.md) P4) — `deterministic` and/or `authorisation-artifact` — whose predicate references the region's conditions. The engine gains **no** invariant→move-blocking primitive: invariants still only report, and the exit gate (a gate) checks. A shut exit is a **closed gate** (the gate predicate returns `result=False, indeterminate=False`), not a blocked overlay; status names the unmet region-scoped invariants as the reason the exit is shut.

Three things make this composition sound rather than a special case. Region scoping reuses the existing `resolve_position` read and the existing predicate runner, so [COR-033](../../../.pkit/decisions/core/COR-033-process-substrate.md) P3 (position inferred from reality) and single-subject ([COR-032](../../../.pkit/decisions/core/COR-032-keyed-process-subjects.md)) both hold unchanged — no cross-subject read, no stored position. The exit gate is one of the two *existing* checkable gate kinds, so the engine's move-legality surface is byte-unchanged (no new gate kind, unlike composition's `subprocess-outcome` and cascade's `cascade-outcome`). And [COR-035](../../../.pkit/decisions/core/COR-035-process-invariants.md)'s report-only decision is preserved verbatim: the boundary is enforced by the gate predicate a region declares, never by an invariant blocking a move.

## Context

COR-040 settled the rule but explicitly deferred the *how* to acceptance-gated implementation: its Engine implication states "the engine strategy (how scoped invariants are evaluated against the resolved position, and the failure semantics of a shut exit) is recorded in a sibling engine-strategy ADR at implementation time (ADR custody per COR-025), per the substrate's per-slot ADR precedent." COR-040 also names "one open item left to implementation" — whether a single exit gate can express a compound "(predicate) AND (cross-authority artifact)" directly, or whether the adopter folds the artifact's existence into the predicate while preserving the engine's authorship check. This ADR settles both.

The open-region slot is the fulfilment of COR-033's P6 extension point and the un-defer of COR-035's `applies_to` scoping + boundary-enforcement pair. It is the **sibling** to the composition and cascade engine-strategy ADRs — [ADR-022](ADR-022-subprocess-resolution-strategy.md) (resolve one inner outcome) and [ADR-023](ADR-023-cascade-fold-strategy.md) (fold many member outcomes) — and sits at the same altitude: where those record how the engine reaches *across* a process boundary, this records how the engine reads a region-scoped always-check *within one subject's* reality and how it enforces a region's boundary. It is **the smaller sibling by design**: composition and cascade each added a new engine-computed gate kind and a new cross-process control-flow path; the open region adds *neither* — its whole novelty is a position filter on an existing report and a discipline about which existing gate kind carries the exit.

The decision space has four architecturally-significant choices, each with a plausible alternative the implementation rejects:

1. **How a region-scoped invariant is evaluated against the resolved position** — filter the existing process-wide invariant report by `applies_to` against `resolve_position().state_id` vs. a separate region-invariant evaluator or a position-parameterised runner.
2. **How the region boundary is enforced** — the exit gate's predicate references the region's conditions (gate composition) vs. coupling region invariants to move-blocking in the engine (COR-040's rejected (B)).
3. **What a shut exit *is*, at failure** — a closed gate (predicate returns false, `validate`/status names the unmet condition) vs. a COR-034-style blocked overlay.
4. **Whether a single exit gate can express `(compound predicate) AND (cross-authority sign-off artifact)`** — the COR-040 open item, and whether folding the artifact into a predicate preserves the engine's authorship guarantee (COR-033 P4).

## Decision

**In plain terms:** the engine answers "may this subject leave the open region?" by evaluating one ordinary gate on the region's sole outgoing transition — the same way it evaluates any gate — while surfacing the region's always-checks (filtered to the region the subject is actually in) as the human-readable reason the gate is shut. It invents nothing: no new gate kind, no invariant-blocks-a-move behaviour, no position storage.

### 1. Region-scoped invariant evaluation — a filter on the existing report, keyed on the resolved position

`evaluate_invariants` today runs *every* declared invariant against current reality, position-independently (COR-035: "invariants hold process-wide, so they are evaluated against current reality regardless of the subject's position"). The one change this slot makes: each invariant may carry an optional `applies_to` naming the state it is scoped to, and the engine **filters** — an invariant with `applies_to: <state>` is evaluated (and surfaced) only when the subject's **currently resolved position** is `<state>`; an invariant with **no** `applies_to` stays process-wide, evaluated on every read, exactly as COR-035 shipped.

- **The resolved position is read once, from `resolve_position`.** The filter keys on `resolve_position().state_id` — the same inferred-from-reality detection every other engine path uses. There is no second position mechanism and no stored position: the scope test is `invariant.applies_to is None or invariant.applies_to == position.state_id`. This is the single new coupling the slot introduces — between detection and invariant reporting — and it is a pure read: a scoped invariant is *not evaluated at all* when the subject is elsewhere (so an out-of-region check cannot report a spurious violation), and is evaluated live when the subject is in its region.
- **It stays deterministic, single-subject, read-only.** The scoped check runs through the *same* `PredicateRunner` (and its per-invocation `(command, args)` cache) that backs detection, gates, and unscoped invariants — single-subject, threaded with the subject id, never across subjects (COR-032). Fail-closed is unchanged: a scoped invariant whose `check` is indeterminate is reported as NOT holding, mirroring the existing COR-035 handling.
- **An indeterminate position degrades safely.** When `resolve_position` is indeterminate (a detection predicate could not be evaluated), the engine cannot know which region the subject is in, so a scoped invariant's region cannot be confirmed. The engine treats a scoped invariant as **not-applicable** (not evaluated, not surfaced) under an indeterminate position — it never *reports* it as violated on an unknown position (that would be a spurious violation the operator cannot act on), and the indeterminate position is already the fail-closed signal that blocks `move`. Unscoped invariants continue to evaluate under an indeterminate position, exactly as COR-035 already does.

### 2. Boundary-enforcement is the exit gate's predicate — no invariant→move-blocking primitive

The region's boundary is enforced by its **exit gate**, an ordinary checkable gate (COR-033 P4) on the region's sole outgoing transition. An open region that "must hold its invariants to be left" expresses that as the exit-gate predicate referencing those conditions — COR-040's chosen (A) over the rejected (B). Concretely:

- **The exit gate is an existing gate kind.** It is `deterministic` (a predicate the engine evaluates) and/or `authorisation-artifact` (a cross-authority sign-off the engine confirms exists). It is **not** a new gate kind — unlike `subprocess-outcome`/`cascade-outcome`, the engine's `precheck_transitions` and `evaluate_gate` are byte-unchanged for the open region. Move-legality inside a region reduces to the region's single outgoing transition's existing gate evaluation.
- **The engine gains NO "an invariant blocks a move" behaviour.** COR-035 kept invariants report-only and rejected coupling them to move-blocking; this slot preserves that verbatim. A region invariant *reports* (via `validate` and status); the exit *gate* *checks*. When a binding wants the region's invariant condition to also gate the exit, it writes that condition **into the exit gate's predicate** — a predicate may check any condition, including one also surfaced as an invariant. This is COR-035's own guidance ("a hard always-gate is expressed as a gate predicate") realised for the region.
- **Fail-closed-at-mutation stays with the binding.** The engine is a validator, not a write-path interceptor (COR-040). A binding needing a hard invariant to fail-closed the instant it would be violated enforces that in its own commands; the exit gate is the *structural* backstop (the subject cannot leave while unmet), not a replacement for the command-level guard.

### 3. A shut exit is a closed gate, not a blocked overlay

When a region condition is unmet at the exit, the exit gate's predicate returns `result=False, indeterminate=False` — a **determinate "not yet"**, exactly the closed-gate semantics ADR-022 established (a determinate inner that has not reached the named outcome "simply does not pass … that is NOT indeterminate, it is a closed gate"). This is the load-bearing failure-semantics call, and it is chosen over a COR-034-style blocked overlay:

- **The shut exit itself is a closed gate.** The status view already renders a gate-closed move with a `✗` marker and the gate's own reason. For an open region, the exit gate's reason is the region's readiness prose, and the region-scoped invariant *violations* are surfaced distinctly (section 5) as the explanation for why the gate is shut. `validate` reports the same region-scoped invariants (now filtered to the region) as violated. Nothing new is needed: a closed gate is already a first-class, self-explaining engine output.
- **The blocked overlay is layered by the binding, not implied by the region.** A subject parked in a region with a shut exit is "stuck" in the ordinary sense — it has no legal move (`has_no_legal_move` is true). Whether that surfaces as a COR-034 `blocked` overlay is the binding's *declaration*, exactly as for any other parked state: if the region declares an `awaiting-human` wait (the exit is a `user` sign-off) or an `awaiting-condition` wait (`resume_when` mirrors the exit predicate), the existing `evaluate_blocked` machinery derives it live. The open region introduces **no new blocked reason** — it reuses `awaiting-human` / `awaiting-condition` where the binding declares one, and otherwise the subject is simply a no-legal-move parked position with a closed exit gate. This is deliberately unlike composition/cascade, which each added an *auto-clearing* wait reason (`awaiting-subprocess-outcome` / `awaiting-cascade-outcome`) because their "condition" was a live cross-process resolution the engine owns; a region's exit condition is an ordinary gate, so it reuses the ordinary wait model.
- **Indeterminate is the only fail-closed case.** A shut exit whose predicate is *indeterminate* (predicate errored / timed out / unresolved) fails closed (`result=False, indeterminate=True`) and refuses the move like an unrecognised gate kind — distinct from the determinate "not yet". This matches ADR-022's taxonomy exactly: determinate-not-yet is a correct wait; indeterminate is a fail-closed refusal that names why.

### 4. The compound exit — cross-authority stays a gate the engine computes; the conjunction lives in the wrapper

COR-040's one open item: can a single exit gate express `(compound predicate) AND (cross-authority sign-off artifact)` directly, or does the adopter fold "the artifact exists, authored by a different authority" into the predicate — and does that folding preserve the engine's authorship guarantee (COR-033 P4)?

**The engine-level answer: it cannot be one gate, and folding the artifact into a `deterministic` predicate would break the authorship guarantee. The conjunction lives in the binding's wrapper; the cross-authority half stays an `authorisation-artifact` gate the engine computes.**

- **The engine has no AND-of-two-gate-kinds primitive.** A transition carries exactly one `gate` of one `kind`; `precheck_transitions` reads `gate.kind` and dispatches to one evaluator. This is the identical constraint ADR-023 records for pm's close-eligibility ("the engine has no AND-of-two-gate-kinds primitive … so the conjunction necessarily lives in the wrapper"). The open-region exit inherits it: a single gate cannot be *both* a `deterministic` compound predicate *and* an `authorisation-artifact` sign-off.
- **The authorship guarantee lives only inside `authorisation-artifact`.** The engine computes `result = exists and produced_by is not None and produced_by != actor` and **ignores any predicate-supplied `result`** ("cross-authority is non-overridable", COR-033 P4). A `deterministic` gate, by contrast, uses the predicate's *own* `result` verbatim. **Therefore folding "the artifact exists, authored by a different authority" into a `deterministic` predicate does NOT preserve the authorship guarantee** — a `deterministic` gate never runs the `produced_by != actor` check, so the acting agent could author its own sign-off and the predicate could return `true`. That is precisely the cross-authority hole `authorisation-artifact` exists to close.
- **The concrete build target.** The exit *transition* carries the `authorisation-artifact` gate — the half only the engine can establish soundly (the human's "ready-to-realize" sign-off, `produced_by != actor`-checked by the engine). The compound readiness predicate is expressed as **region-scoped invariants** (`applies_to: <region>`, section 1) that surface on status as the reason the exit is shut. Where a binding wants a *hard structural* AND — "the readiness compound must hold *and* the sign-off must exist to leave" — it composes the two in its **wrapper**, reading the compound predicate (as a standalone read, the durable primitive ADR-023 established for exactly this conjunctive-eligibility pattern) AND asking the engine whether the `authorisation-artifact` exit gate passes. The software-engineer builds **no** compound-gate primitive into `process.py`; the engine stays the single source of the cross-authority check, and the conjunction stays in the binding that owns the composite rule.
- **A predicate-only exit remains fully expressible as one gate.** When a region's exit is a pure compound predicate with *no* cross-authority requirement, a single `deterministic` gate whose predicate ANDs the conditions is correct and sufficient — one gate, no wrapper. The two-part answer above is only load-bearing when a cross-authority sign-off is one of the conjuncts.

### 5. Status-view semantics — region violations are the reason the exit is shut

A subject in an open region renders the ordinary where/why/what-next, with region-scoped invariant violations surfaced **distinctly as the reason the exit gate blocks**:

- **Where/why** — the region is the resolved position; its `meaning` renders as today.
- **The exit gate** appears in "Legal moves (live precheck)" with its live marker: `✓` when it passes (the subject may leave), `✗` when the exit is shut (a closed gate), `?` when indeterminate. The gate's own reason renders beneath it.
- **Region-scoped invariant violations render in the "Invariant violations" section**, now filtered to the region the subject is in (section 1): a scoped invariant not applicable to the current region does not appear, so the operator sees exactly the always-checks that govern *this* region. A violated region invariant (`✗`) or an indeterminate one (`?`) is the human-readable explanation for why the exit is shut — the operator reads the shut exit gate in "Legal moves" and the unmet region invariants together, so status is self-explaining about *why* the subject cannot leave yet. The JSON status carries the same filtered invariant set (`holds=false` entries) plus the closed exit-gate `legal_moves` entry, so an agent reads the same explanation structurally.

### Boundaries — what this strategy is NOT

- **No new node kind, no sub-state sublanguage.** An open region is one normal state distinguished only by its region-scoped invariants and its single guarded exit (COR-040). Genuinely-staged inner work is a *composed* process (COR-036) the exit gate reads an outcome from — not an internal edge of the region.
- **No new gate kind and no engine-computed exit.** Unlike composition/cascade, the exit gate is one of the *existing* predicate-backed kinds (`deterministic` / `authorisation-artifact`); `precheck_transitions`/`evaluate_gate` are unchanged.
- **No invariant→move-blocking primitive.** Invariants report; the exit gate checks. COR-035's report-only posture is preserved verbatim.
- **No cross-subject reach.** A region-scoped invariant governs the region's own single subject (COR-032); it does not reach into a composed inner process's separate subject (which declares its own invariants).
- **No liveness guarantee.** The engine gives a definite "may this leave yet?" every check; it does not promise a region ever exits. A stuck region is a visible binding problem (status shows the subject parked with a shut exit and the unmet conditions named), not a non-terminating engine computation.

## Rationale

The central design tension COR-040 named is that an open region must be *enforceable* — a subject must not leave until its conditions hold — while COR-035 deliberately kept invariants report-only and rejected move-blocking. The gate-composition strategy resolves that tension by *not making invariants do the enforcing at all*: enforcement is the exit gate (a gate, whose job is exactly move-legality), and the invariants merely report the same conditions for human legibility. This is the smallest possible change that makes the region enforceable — it reuses the existing gate machinery untouched and leaves COR-035's report-only decision verbatim, fulfilling its deferral additively rather than revising it.

Making region scoping a *filter on the existing report* (rather than a new region-invariant evaluator) is the consistency choice: COR-035 already evaluates every invariant through one runner; adding `applies_to` as a position-keyed filter over that same pass means scoped and unscoped invariants share one code path, one cache, one fail-closed rule, and one status surface — there is no second place for the invariant semantics to drift. Keying the filter on `resolve_position().state_id` reuses the inferred-from-reality read every other path uses, so P3 and single-subject transfer for free.

Keeping the shut exit a *closed gate* rather than a new blocked overlay is the failure-taxonomy consistency choice, extended from ADR-022: a determinate condition-not-yet-met is a closed gate (the parent waits), and only indeterminacy fails closed. Layering the COR-034 blocked overlay only where the binding *declares* one keeps the region from silently inventing wait semantics — the region reuses `awaiting-human` / `awaiting-condition` exactly as any other parked state would, because its exit is an ordinary gate, not a live cross-process resolution the engine owns.

The item-4 answer is the authorship-guarantee consistency choice. COR-033 P4 deliberately made cross-authority non-overridable by computing it inside the engine's `authorisation-artifact` evaluation and ignoring the predicate's own `result`. Folding the artifact's existence into a `deterministic` predicate would move that check out of the engine and into a capability's self-report — reopening the exact hole `authorisation-artifact` closes. Keeping the sign-off an `authorisation-artifact` gate and pushing the conjunction into the wrapper (the standalone-read + AND primitive ADR-023 already established) is what lets a compound exit stay expressible *without* the engine ever trusting a capability for a cross-authority fact, and *without* the engine growing an AND-of-gate-kinds primitive it has consistently refused.

### Alternatives considered

- **A separate region-invariant evaluator (a second method distinct from `evaluate_invariants`).** Rejected — it duplicates the runner, cache, fail-closed rule, and status surface, giving the invariant semantics two places to drift. A position-keyed `applies_to` filter on the one existing pass keeps scoped and unscoped invariants on one code path.
- **Couple region-scoped invariants to move-blocking in the engine (COR-040's rejected (B)).** Rejected — that is the move-blocking coupling COR-035 explicitly deferred and COR-040 explicitly rejected; the same enforcement is achieved additively by the exit-gate predicate referencing the conditions, leaving the engine's report-only invariant behaviour untouched and adding no primitive.
- **A new engine-computed exit gate kind (mirroring `subprocess-outcome`).** Rejected — the exit is an ordinary predicate-checkable condition (or a cross-authority sign-off, which the *existing* `authorisation-artifact` kind already computes); it is not a fact only the engine can establish across a process boundary. Reusing the two existing kinds keeps `precheck_transitions`/`evaluate_gate` byte-unchanged and adds no gate-kind to the taxonomy.
- **A new blocked reason (`awaiting-region-exit`, auto-clearing) for a shut exit.** Rejected — composition/cascade needed a bespoke auto-clearing wait because their "condition" was a live cross-process resolution the engine owns; a region's exit is an ordinary gate, so it reuses `awaiting-human` / `awaiting-condition` where the binding declares one, and otherwise is a plain no-legal-move closed-gate position. Inventing a wait reason would duplicate the existing model.
- **Fold the cross-authority sign-off's existence into a `deterministic` exit predicate (the tempting single-gate form of the item-4 open question).** Rejected — a `deterministic` gate uses the predicate's own `result` and never runs the engine's `produced_by != actor` check, so it does not preserve COR-033 P4's authorship guarantee. The sign-off must stay an `authorisation-artifact` gate the engine computes; the conjunction with a compound readiness predicate lives in the wrapper.
- **Report a scoped invariant as violated under an indeterminate position.** Rejected — the engine cannot confirm which region the subject is in, so a "violation" it surfaces would be spurious and unactionable. A scoped invariant is not-applicable (not evaluated, not surfaced) under an indeterminate position; the indeterminate position is itself the fail-closed signal that blocks `move`.

## Implications

- **Engine control flow.** `src/project_kit/process.py` `evaluate_invariants` gains an `applies_to` read that filters each invariant against `resolve_position().state_id` (unscoped ⇒ process-wide, unchanged; scoped ⇒ evaluated only when the subject is in the named region; scoped-under-indeterminate-position ⇒ not-applicable). No new gate kind, no new blocked reason, no new cross-process path — `precheck_transitions`, `evaluate_gate`, and `evaluate_blocked` are byte-unchanged for the open region (the exit is a `deterministic` / `authorisation-artifact` gate, and any blocked overlay is the binding's declared `awaiting-human` / `awaiting-condition`). Position stays inferred; the validator stays deterministic; no cross-subject read.
- **Shape contract.** An invariant gains an optional `applies_to` scope (absent ⇒ process-wide). A state may be marked an open region (the exact field layout lives in the process-area shape reference, not here). The shape's node `additionalProperties` already permits capability-local `applies_to`; the invariant `$def` grows the recognised optional scope. Both additive — existing processes declare neither and validate byte-unchanged, so **no migration is owed** (COR-010).
- **Status / validate surface.** `render_status_*` and `render_validate_*` surface the *filtered* invariant set (scoped invariants appear only in their region), so region-scoped violations render distinctly as the reason a shut exit gate blocks; the exit gate renders as an ordinary closed-gate `legal_moves` entry (`✗` determinate-not-yet, `?` indeterminate). No new render section — the existing "Invariant violations" + "Legal moves" sections carry it.
- **Failure-mode contract for on-call.** A shut exit is a **closed gate**: `result=False, indeterminate=False` — a determinate "not yet", NOT a failure; the right response is to satisfy the region conditions (or record the sign-off), not to debug the engine. An **indeterminate** exit predicate fails closed and refuses the move, naming why — that is the only fail-closed case at the exit. A scoped invariant reported as violated names the unmet region condition; an indeterminate scoped check is fail-closed (reported not-holding) exactly as COR-035 already does.
- **The compound-exit contract for bindings.** A cross-authority sign-off exit is an `authorisation-artifact` gate (engine-computed authorship). A pure-predicate exit is a single `deterministic` gate. A compound "(readiness predicate) AND (cross-authority sign-off)" exit is NOT one gate: the sign-off stays `authorisation-artifact`, the readiness is region-scoped invariants surfaced on status, and any hard structural AND is composed in the wrapper (the standalone-read + AND primitive from ADR-023). Never fold the sign-off into a `deterministic` predicate — that loses the authorship guarantee.
- **Grounding binding.** The slot lands with trip-planning `build` (its DEC-014, proposed): `build`'s evidence-backed / commands-only invariants carry `applies_to: build` and surface on status; its "ready-to-realize" exit is an `authorisation-artifact` gate (the human sign-off, engine-checked for cross-authority), with each track's recorded-readiness compound as region-scoped invariants and, where wanted, a wrapper conjunction. The core slot and the binding accept in sequence (the binding cites COR-040).
- **Relationship to records.** Records the engine strategy for COR-040 (the rule); sibling to ADR-022 (single-inner resolution) and ADR-023 (cross-subject fold), one altitude below them in novelty (no new gate kind, no cross-process path). Preserves COR-033 P3 (position inferred — the `applies_to` filter reads `resolve_position` live, stores nothing) and P4 (cross-authority stays the engine-computed `authorisation-artifact` gate). Holds COR-032 (single-subject — a region invariant governs the region's own subject, no cross-subject read). Preserves COR-035's report-only posture verbatim (invariants report; the exit gate checks) and un-defers its `applies_to` scoping + boundary-enforcement pair additively. Reuses COR-036/COR-037 composition/cascade for genuinely-staged inner work the exit gate reads an outcome from. Reuses ADR-023's standalone-read + wrapper-AND primitive for the compound exit. **Supersedes nothing.**
