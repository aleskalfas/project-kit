## Summary

On macOS, `gh` provably cannot run inside Claude Code's Seatbelt box — the same failure class [ADR-014](ADR-014-macos-sandbox-platform-stance.md) proved for `uv`. Go's `crypto/x509` on macOS reaches the `com.apple.SecurityServer` mach-service (Security.framework) that the sandbox blocks, so `gh`'s TLS handshake dies at `x509: OSStatus -26276` (documented in Claude Code issues #29533 / #23416). This is a **mach-service denial, not a missing-cert problem** — and Go on macOS *ignores* cert files (`SSL_CERT_FILE` is a no-op on darwin), so no `allow-read` accommodation of any cert path or keychain can reach past it. The only non-exclusion fixes are a rebuilt `gh` (a code change, not an accommodation) or `enableWeakerNetworkIsolation` (weakening the whole box). `git`-over-HTTPS is unaffected — it uses OpenSSL, not Security.framework.

[ADR-027](ADR-027-required-sandbox-exclusion-auto-apply.md) carved a single narrow exception to ADR-008 rule 4: `setup autonomy` MAY auto-apply a sandbox exclusion for the **platform-mandatory, required-only** subclass, when necessity is *verified* rather than believed. ADR-027 verified that necessity for exactly one member — `uv` — gated on a `uv` version floor. **This record generalises that subclass from one member to a necessity-verified set, and admits `gh` on macOS as its second member.** `gh` clears ADR-014's evidence bar (a fixed mach-service denial, accommodation mechanically impossible), so it is *forced not chosen* — the precondition the whole carve-out rests on.

`gh`'s exclusion is a materially bigger boundary statement than `uv`'s. `uv` excluded leaves pkit's own *local* runtime off-box; `gh` excluded leaves `gh`'s **network egress fully off-box and unbounded** ([ADR-004](ADR-004-autonomy-intent-confinement.md)'s §61 residual gap, for a command that exists to talk to GitHub). This record re-argues ADR-027's no-confirm condition (its condition 7) afresh for the egress case — and concludes the no-boundary-to-falsely-believe-in argument holds, because the box *provably cannot host `gh` at all* on macOS, so the honest end-state is "`gh` runs off-box, egress unbounded, loudly reported" versus "`gh` does not work." There is no confined-`gh` state to falsely believe in. But the egress cost is real, larger than `uv`'s, and the apply banner names it loudly.

Three honest differences from `uv` are recorded as conditions, not glossed: (1) `gh`'s failure has **no version coordinate** — it is permanent on macOS until Claude Code ships mach-lookup allowlisting / `enableWeakerNetworkIsolation` or `gh` ships fallback roots, so necessity is recorded as **platform-permanent with a review-on-Claude-Code-update caveat**, not a version-floor self-heal; (2) auto-apply is **detect-fenced** — fires only when the `.github/` detect glob signals real project use of `gh`, never on bare `gh`-on-PATH; (3) the required subclass stays fenced by a three-part test (forced AND low-marginal-reach AND necessity-verified) that `docker` fails — `docker` stays nudge-only.

This **amends ADR-027** — it generalises ADR-027's required-subclass from `uv`-only to a verified set and re-derives ADR-027 condition 7 for the egress case. It does not supersede it; ADR-027's decision stands and gains a forward-pointer. Because the amendment widens a security boundary's auto-apply scope and re-argues a security-boundary condition (no-confirm under unbounded egress), it carries an explicit foundational-re-authorisation flag: operator sign-off is required and is the gesture that flips this record to `accepted` per the [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md) acceptance gate. `decide.py` is untouched.

## Context

[ADR-014](ADR-014-macos-sandbox-platform-stance.md) established the macOS platform stance and the evidence bar for it: a command is excluded from the Seatbelt box only when it *provably cannot run confined* — a fixed, non-configurable mach-service denial that no `narrowing` accommodation and no env lever can satisfy. ADR-014 proved this for `uv` (the `SCDynamicStore` denial at its SystemConfiguration proxy probe) and excluded it as a *widening* allowance kept loud and explicit per [ADR-008](ADR-008-confinement-allowances.md). [ADR-008](ADR-008-confinement-allowances.md) rule 4 governs that widening class with four guarantees — applied-only-by-explicit-gesture, loud, never-persisted, always-reported — and originally forbade `setup autonomy` from applying any widening.

[ADR-027](ADR-027-required-sandbox-exclusion-auto-apply.md) carved one exception to rule 4's never-applied-by-setup clause: `setup autonomy` MAY auto-apply a widening exclusion, but only for the *platform-mandatory, required-only* subclass, and only after *verifying* the exclusion is still necessary. ADR-027 populated that subclass with exactly one member — `uv` — and bound its necessity verifier to a `uv` version floor (auto-apply while the panic provably still occurs; self-heal and leave confined once a fixed `uv` ships). ADR-027's condition 2 ("required-only") explicitly relegated `gh` and `docker` to the nudge-only path *as of that record* — "carving out `gh`, `docker`, or any tool the box could host but the operator wants unconfined for convenience." This record revisits that line for `gh` specifically, on new evidence that the box on macOS **cannot** host `gh` at all.

**Evidence — `gh` fails under macOS Seatbelt at ADR-014's bar.** Documented in Claude Code issues #29533 and #23416:

1. `gh` fails under macOS Seatbelt because Go's `crypto/x509` on macOS calls the mach-service `com.apple.SecurityServer` (Security.framework) for certificate verification, which the Seatbelt profile blocks — surfacing as `x509: OSStatus -26276` at the TLS handshake.
2. The accommodation is **ruled out by the mechanism**: Go on macOS *ignores* certificate files entirely — `SSL_CERT_FILE` is a no-op on darwin — so no `allow-read` of any cert path, bundle, or keychain can route the verification around the blocked mach-service. This is structurally the same as `uv`'s `SCDynamicStore` denial: a **mach-service denial**, the failure class ADR-014 proved accommodation-proof.
3. The only non-exclusion fixes are (a) a rebuilt `gh` that uses `GODEBUG` fallback roots — a code change to the binary, not an accommodation pkit can apply — or (b) `enableWeakerNetworkIsolation`, which weakens the entire box rather than carving out the one command. Neither is a `narrowing` accommodation; the narrowing path is mechanically closed.
4. `git`-over-HTTPS is **unaffected** — it links OpenSSL, not Security.framework, so it never reaches the blocked mach-service. The exclusion is scoped to `gh`, not to all GitHub traffic.

This evidence clears ADR-014's bar exactly: a fixed, non-configurable mach-service denial, no accommodation reachable. [ADR-008](ADR-008-confinement-allowances.md)'s narrowing-first discipline (try the accommodation before the widening) is therefore **satisfied** — the narrowing was tried-in-principle and is mechanically impossible, which is precisely the condition that licenses widening over narrowing.

The forcing question for this record (its producing issue, #279) is whether the macOS `gh` exclusion — now shown forced, not chosen — should join ADR-027's auto-applied required subclass, and on what conditions, given that excluding `gh` makes a *materially larger* boundary statement than excluding `uv` (unbounded network egress for a network tool). The producing session ran the design review whose findings this record encodes; the accommodation-versus-exclusion question is RESOLVED (the accommodation is ruled out by the mechanism above). As project-kit's own architecture-decision record, harness and platform specifics are in scope here, unlike the harness-neutral [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md). The record lands `status: proposed`; operator sign-off flips it to `accepted` per the [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md) acceptance gate.

## Decision

**ADR-027's platform-mandatory required-only auto-apply subclass generalises from one member (`uv`) to a necessity-verified set; `gh` on macOS joins it as the second member. `setup autonomy` auto-applies the `gh` sandbox exclusion on macOS when the `.github/` detect glob is present, under all of ADR-027's conditions, with the necessity-verifier and the no-confirm argument re-derived for `gh`'s permanent-and-egress-unbounded shape.** Each condition below is load-bearing; remove any one and `gh`'s membership in the subclass is not defensible.

1. **Necessity verified at ADR-014's bar; accommodation ruled out.** `gh`'s exclusion is admitted only because the box *provably cannot host it* on macOS — the `com.apple.SecurityServer` mach-service denial (Evidence 1), with the cert-read accommodation closed by Go's darwin behaviour (Evidence 2). This is the same failure class as `uv`'s `SCDynamicStore` denial that ADR-014 proved accommodation-proof. The narrowing-first discipline of [ADR-008](ADR-008-confinement-allowances.md) is satisfied: the narrowing was tried-in-principle and is mechanically impossible, which is what licenses the widening here.

2. **No-confirm holds for the egress case — re-derived afresh, not inherited.** [ADR-027](ADR-027-required-sandbox-exclusion-auto-apply.md) condition 7 allowed no-confirm for the `uv` exclusion because there was no boundary to falsely believe in. That argument is re-run here for `gh`'s materially larger boundary statement — `gh`-excluded leaves `gh`'s network egress fully off-box and unbounded ([ADR-004](ADR-004-autonomy-intent-confinement.md) §61's residual gap), for a command whose entire purpose is talking to GitHub. The argument still holds, and on the same ground: the box **provably cannot host `gh` at all** on macOS, so confirming the exclusion authorises the only physically-available outcome. The honest end-state is "`gh` runs off-box (egress unbounded, loudly reported)" versus "`gh` does not work" — there is no confined-`gh` state for the operator to falsely believe in, so a confirmation prompt would be ceremony over a non-choice. **But the egress cost is real and larger than `uv`'s, and it is stated loudly:** the apply banner names the egress carve-out explicitly ("`gh` runs OUTSIDE the box — its network egress is unconfined"), `sandbox status` / `probe` report it, and the `network-egress: none` residual gap ([ADR-004](ADR-004-autonomy-intent-confinement.md) §61) is the standing truth this exclusion lands inside, not a new hole it digs.

3. **Permanent necessity, no version-floor self-heal.** Unlike `uv`, `gh`'s failure has **no version coordinate** — it is not a release-specific panic that a fixed binary clears under its own version number. It is permanent on macOS until one of: Claude Code ships mach-lookup allowlisting for `com.apple.SecurityServer`, Claude Code ships `enableWeakerNetworkIsolation` as a usable lever, or `gh` ships `GODEBUG` fallback roots in its released binary. Necessity is therefore recorded as **platform-permanent with a review-on-Claude-Code-update caveat** — the self-heal/self-disable analogue of ADR-027 condition 6, expressed as a *review trigger* rather than an automatic version check. Honest residual: there is **no cheap auto-self-disable** for `gh` the way ADR-027's version floor self-disables `uv`. The exclusion is reviewed on Claude Code (or `gh`) updates, not auto-healed. The verifier-binding for the `gh` member is "macOS AND the platform-permanent mach-service denial still holds (reviewed, not version-gated)," distinct from `uv`'s version-floor binding.

4. **Detect-fenced.** Auto-apply fires **only when the `.github/` detect glob is present** — the signal of real project use of `gh` (workflows, issue templates, a GitHub-hosted repo). It **never** fires on a bare `gh`-on-PATH. A machine that has `gh` installed but no `.github/` in the project gets no auto-applied exclusion; the looser platform+name predicate may still drive the *nudge* path, but only the detect-fenced conjunct drives the *auto-apply* path. This keeps the auto-widening tied to observed need, not to the mere presence of a tool.

5. **The required subclass stays fenced by a three-part test; `docker` stays optional.** Membership in the auto-applied required subclass requires **all three**: *forced* (the box provably cannot host the command — necessity-verified at ADR-014's bar) AND *low-marginal-reach* (the carve-out's boundary reduction is bounded and reportable) AND *necessity-verified* (a per-member verifier confirms the failure still holds). `uv` passes (forced by `SCDynamicStore`; reach is local-runtime; version-floor verifier). `gh` passes (forced by the mach-service denial; reach is egress, larger than `uv`'s but bounded to one command and loudly reported; review-triggered verifier). **`docker` fails** the test on marginal-reach: excluding `docker` is the largest boundary reduction of the three — arbitrary host mounts plus unbounded network — and `docker` is *not* forced (the box can host it). `docker` stays strictly nudge-only on [ADR-008](ADR-008-confinement-allowances.md) rule 4's path; the auto path never touches it. The three-part test is the fence that keeps the set from drifting into "auto-apply any convenient widening."

6. **Classification home — runtime verdict, declared candidate set.** The necessity *verdict* for a member stays **runtime-derived** (the #247 stance: pkit decides at run time whether the exclusion is still necessary, against live platform/version/evidence state — it is not frozen into committed data). What becomes **declared, auditable data** is the *candidate set + per-member verifier-binding*: which commands are eligible for the required subclass, and which verifier each member's necessity is checked against (`uv` → version floor; `gh` → macOS-permanent-reviewed). This is the [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md) extraction triggered by the arrival of the *second* member — the recurrence test is met (the same shape now repeats across `uv` and `gh`), so the per-member knowledge is lifted from inline code into auditable declared data, while the verdict logic stays runtime. This is the implementing structure: a declared candidate set the writer reads, each entry naming its verifier; the runtime path consults the verifier per member and applies or self-heals accordingly.

7. **Loud + revertable + provenance — rides ADR-027's machinery unchanged.** The `gh` exclusion rides [ADR-027](ADR-027-required-sandbox-exclusion-auto-apply.md)'s `_required:` provenance tag, lands in the gitignored `.claude/settings.local.json` by [ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md)'s routing (a widening, so never committed — by construction), and is individually revertable by `sandbox exclude --remove gh`. The apply banner names the egress cost (condition 2). The **manual remove / re-apply loop is acceptable** for a permanently-required exclusion: because there is no auto-self-disable (condition 3), an operator who removes the `gh` exclusion will have it re-applied on the next `setup autonomy` run while the macOS denial holds — the same convergence behaviour as `uv`, and the honest one for a permanently-forced carve-out.

8. **`decide.py` untouched.** This record changes the candidate-set data, the `setup autonomy` auto-apply step, and the sandbox writer's per-member verifier consult. It does **not** touch the permission-decision core (`decide.py` / the shared `decide()`): intent enforcement is unaffected by an OS-box exclusion ([ADR-014](ADR-014-macos-sandbox-platform-stance.md) decision point 2 — an excluded command remains hook-gated). The same-code invariant of [ADR-002](ADR-002-permission-realizer-ownership.md) / [ADR-003](ADR-003-permission-core-code-home.md) stands.

## Rationale

**Why generalise the subclass rather than special-case `gh`.** ADR-027 verified the required subclass with a population of one and an inline, `uv`-specific verifier (the version floor). The arrival of a second member that clears the same evidence bar by the same mechanism (a macOS mach-service denial, accommodation-proof) is exactly [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)'s recurrence signal: the same shape of work now repeats, so the per-member knowledge earns extraction into a declared, auditable candidate set rather than a second inline special-case. Generalising the subclass — not bolting `gh` on as a one-off — is what keeps the carve-out's *fence* (condition 5's three-part test) explicit and auditable, so the third candidate (`docker`) is rejected by a stated rule rather than by omission.

**Why the no-confirm argument must be re-derived, not inherited.** ADR-027 condition 7's no-confirm rests on "no boundary to falsely believe in." For `uv` that was cheap: `uv`-excluded carves out pkit's own local runtime, a modest boundary statement. For `gh` the boundary statement is materially larger — unbounded network egress for a network tool — so inheriting the no-confirm conclusion without re-argument would smuggle a bigger boundary reduction in under a smaller one's licence. The re-derivation (condition 2) confirms the *argument* still holds, because the licensing condition is not "the carve-out is small" but "the box provably cannot host the command, so there is no confined state to believe in" — which is as true for `gh` as for `uv`. The egress cost is then stated honestly and loudly rather than waved away: the conclusion is the same, but it is reached on `gh`'s own evidence, and the cost is named.

**Why permanent-reviewed rather than version-gated.** ADR-027's version floor is a *self-disabling* verifier: a fixed `uv` removes its own exclusion. `gh`'s failure has no such coordinate — it is a property of Go's darwin TLS path under Seatbelt, cleared only by a harness change, a `gh` rebuild, or a box-weakening lever, none of which carry a version number pkit can gate on. Recording `gh`'s necessity as platform-permanent-with-review-caveat is the honest verifier for that shape. Pretending a version floor exists would be the believed-not-verified failure ADR-027 exists to prevent, in reverse: a verifier that never fires. The honest residual — no cheap auto-self-disable, reviewed on Claude Code updates — is recorded rather than engineered away.

**Why detect-fencing.** ADR-027's `uv` member needs no detect fence: pkit *is* the `uv`-shim, so a macOS autonomy setup always needs `uv` excluded. `gh` is different — it is an optional tool many projects do not use. Auto-applying its exclusion on bare `gh`-on-PATH would carve a boundary for a command the project may never invoke, re-importing a speculative-widening flavour of the very trap ADR-027 fences. The `.github/` detect glob ties the auto-apply to observed project use, keeping the auto-widening necessity-driven (the project uses `gh`) rather than presence-driven (the machine has `gh`).

**Why `docker` stays out.** The three-part test (condition 5) is the load-bearing fence. `gh` clears it on all three counts; `docker` fails two — it is not *forced* (the box can host it) and its marginal reach is the *largest* of the candidates (arbitrary mounts + net). Admitting `gh` while keeping `docker` nudge-only is not inconsistency; it is the test doing its job. A subclass that admitted `docker` would be the "auto-apply any convenient widening" generalisation ADR-027 and ADR-008 rule 4 both forbid.

### Alternatives considered

- **Accommodate `gh` (read the system cert bundle / keychain into the box).** Rejected on the mechanism. Go on macOS ignores cert files (`SSL_CERT_FILE` is a darwin no-op) and verifies via the `com.apple.SecurityServer` mach-service, which Seatbelt blocks. No `allow-read` of any cert path or keychain reaches the verification. The narrowing is mechanically impossible, which is what licenses the widening (ADR-008 narrowing-first satisfied).

- **Keep `gh` nudge-only, as ADR-027 condition 2 left it.** Rejected. ADR-027 relegated `gh` to nudge-only on the *assumption* that the box "could host" it and the operator merely "wants it unconfined for convenience." The new evidence falsifies that assumption: the box provably *cannot* host `gh` on macOS. With `gh` shown forced-not-chosen at ADR-014's bar, every macOS operator with a `.github/` project must run the identical `sandbox exclude gh` gesture by construction — the same deep-knowledge friction `setup autonomy` exists to remove, with no safety gain because the exclusion is forced. Retained nudge-only for genuinely optional widenings (`docker`).

- **Require an operator confirmation prompt for `gh` because its egress carve-out is large.** Rejected — the prompt would authorise the only physically-available outcome (the box cannot host `gh`), adding ceremony over a non-choice (condition 2). The honest response to the larger boundary statement is a *louder banner naming the egress cost*, not a confirmation that gates nothing. Confirmation is reserved for widenings where a confined alternative genuinely exists.

- **Gate `gh`'s necessity on a version floor like `uv`.** Rejected — `gh`'s failure has no version coordinate (it is cleared only by a harness change, a rebuilt `gh` with fallback roots, or a box-weakening lever, none version-gateable). A version-floor verifier for `gh` would never fire — a verifier that lies about being able to self-disable. The review-on-Claude-Code-update caveat is the honest verifier for a permanent platform necessity.

- **Auto-apply on bare `gh`-on-PATH, no detect fence.** Rejected — carves a boundary for a command the project may never use, a presence-driven rather than necessity-driven widening. The `.github/` detect glob ties auto-apply to observed need.

- **Admit `docker` to the auto-applied required subclass too.** Rejected — `docker` fails the three-part test (condition 5): it is not forced (the box can host it) and its marginal reach is the largest of the candidates. Admitting it would generalise the subclass into the "auto-apply any convenient widening" trap the fence exists to prevent.

## Implications

- **ADR-027 is amended, not superseded.** This record generalises ADR-027's required-only subclass from `uv`-only to a necessity-verified set and re-derives ADR-027 condition 7 (no-confirm) for `gh`'s egress-unbounded case. ADR-027's decision (the carve-out, its seven conditions, the `uv` member and its version-floor verifier) stands unchanged; `gh` joins the set under those conditions, with the necessity-verifier and no-confirm argument re-derived for its permanent-and-egress shape. This is an amendment widening the subclass's membership and re-arguing a security-boundary condition — **ADR-027 gains a forward-pointer to this record; no `superseded_by` frontmatter on ADR-027.**

- **Foundational re-authorisation flag (escalation).** Because this record widens the auto-apply scope of a security-boundary carve-out (a second member, with a materially larger boundary statement) and re-argues a security-boundary condition (no-confirm under unbounded egress), it carries an **explicit foundational-re-authorisation flag**: operator sign-off is required and is the gesture that flips this record from `proposed` to `accepted`. This is not a silent ride on ADR-027's prior acceptance — the egress cost and the membership widening are a fresh decision the operator authorises.

- **ADR-014's evidence bar is reused, its scope unchanged.** `gh`'s exclusion is admitted by clearing ADR-014's exact bar (a fixed mach-service denial, accommodation-proof). ADR-014's `uv` decision and its narrowing-first discipline stand unchanged; this record adds a second command that meets the same bar by the same mechanism. No edit to ADR-014 beyond an optional cross-reference.

- **ADR-008 rule 4 is unchanged; the narrowing-first discipline is honoured.** Rule 4's never-persisted, always-reported, loud guarantees stand for the `gh` exclusion exactly as for `uv`. The narrowing-first discipline is satisfied (the accommodation is mechanically impossible), which is the precondition for the widening. The required-subclass exception to rule 4's never-applied-by-setup clause is ADR-027's, now extended to a second member by this record.

- **ADR-029 routing carries the `gh` exclusion unchanged.** As a widening, the `gh` exclusion routes to the gitignored `.claude/settings.local.json` by ADR-029's boundary-effect routing — never committed, by construction. The `_required:` provenance tag (ADR-027 condition 3) attributes it; the provenance read unions both settings files (ADR-029 condition 5) so revert / self-heal stays correct. No change to ADR-029.

- **ADR-004 §61's residual gap is the standing truth this lands inside.** The `network-egress: none` residual gap (ADR-004 §61) already declares that egress as a security boundary is an out-of-harness concern. `gh`-excluded does not dig a new hole; it lands a command inside that already-declared gap, and reports it loudly (condition 2). The §61 honesty-about-the-gap stance is the frame, not a casualty.

- **Implementation is separate work, not this ADR.** The declared candidate set + per-member verifier-binding (condition 6), the `gh` member with its macOS-permanent-reviewed verifier, the `.github/` detect-fence (condition 4), the `setup autonomy` auto-apply + the egress-naming banner (condition 2), the `sandbox status` / `probe` attribution, and tests land in the implementing issue (#279, branch `feat/279-auto-apply-gh-exclusion-on`). This record sanctions building them on acceptance.

- **Versioning.** Behaviour-observable surface change (`setup autonomy` gains a second auto-applied exclusion; the candidate set becomes declared data; the apply banner gains egress-cost text) → the implementing change bumps `.pkit/VERSION` per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md). This decision record alone does not bump.

- **Migration.** The candidate-set extraction (condition 6) introduces declared data the sandbox writer reads. Whether lifting `uv`'s inline verifier-binding into the declared set warrants a one-shot idempotent migration is a [COR-010](../../../.pkit/decisions/core/COR-010-resource-lifecycle.md) judgement for the implementing change; `pkit migrations check-diff --include-working-tree --base main` gates it. The ADR file itself is migration-clean.

- **Cross-platform and cross-harness.** The auto path is inert off macOS — the `com.apple.SecurityServer` denial is Seatbelt-specific; on Linux/WSL2 (bubblewrap) `gh` runs confined, so there is no required exclusion and nothing for the auto path to do. The `gh` member is keyed to the verified macOS-permanent necessity, so it carries to a future adapter only where that adapter declares an equivalently forced exclusion — it does not hand future adapters a general auto-widen licence.
