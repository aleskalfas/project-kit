## Summary

The rule "the project-manager agent must use the validated scripts, not raw `gh`" lives today as a hand-written line in project-kit's *own* permissions file. That's the wrong home: it's the *capability's* policy, not the project's — so every adopter would have to copy it by hand, and it wouldn't disappear if they uninstalled the capability.

**Decision:** a capability ships its own small grants file (`.pkit/capabilities/<cap>/permissions/grants.yaml`), and the permission model reads it automatically for each *installed* capability. Install the capability → its grants apply; uninstall → they vanish. Nothing is hand-copied.

Four properties make it safe:

- **Your settings win on what's *allowed*** — a capability can't take away something you granted. But a capability's *deny* holds unless you deliberately override it (a safe default you can still escape with an explicit grant).
- **Only installed capabilities count** — discovery walks the manifest, so a leftover folder from a botched uninstall can't silently change your permissions.
- **It's visible** — `pkit permissions overview` shows which capability contributed a deny; nothing changes an agent's reach silently.
- **The privilege definition stays central** for now — only the *policy* (deny it to this agent) moves into the capability.

First use: the project-management capability denies `agent:project-manager` the raw `gh` issue/PR writes, replacing the manual grant in project-kit's own permissions file. The detailed decision, the layering rationale, and the rejected alternatives follow.

## Context

The forcing question (issue #72, under EPIC #70) is whether an *installed capability* can contribute to the **intent** half of the permission model — the *what-may-be-invoked* axis pkit owns ([ADR-004](ADR-004-autonomy-intent-confinement.md) rule 1) — the way [ADR-015](ADR-015-command-declared-network-egress.md) let a capability contribute to the **confinement** half. The concrete first need: the project-management capability wants `agent:project-manager` denied the `issue-tracker-write` privilege (raw `gh issue`/`gh pr` mutations), because that agent reaches the tracker through its own validated path, not raw `gh`. Today that policy is carried as a *manual adopter grant* in project-kit's own `.pkit/permissions/project/grants.yaml`:

```yaml
- subject: agent:project-manager
  privilege: '[privilege-catalog:issue-tracker-write]'
  effect: deny
```

That placement is wrong on ownership: the deny is the *capability's* policy (it travels with project-management, applies to every adopter who installs it, and should vanish when the capability is uninstalled), but it sits in the *adopter's* hand-authored, adopter-owned file. Every adopter of project-management would have to hand-copy it, and no adopter-owned file should carry a capability's standing intent. The model loader (`load_model` in `.pkit/permissions/decide.py`) composes the effective model today as `guardrail_denies + active-profile-grants + adopter grants.yaml` ([ADR-005](ADR-005-permission-profile-surface.md) rule 2) — there is **no layer** through which an installed capability declares intent grants.

The operator approved **direction A: build the general, reusable substrate** — a capability-contributed intent-grant layer composed by the model loader — rather than hardcoding this single project-manager deny as a one-off floor. This ADR records that decision and the points settled in the architect review that produced it. As project-kit's own architecture-decision record, harness and path specifics are in scope here, unlike the harness-neutral COR. The `proposed` status is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md); accept before any implementation cites it.

## Decision

**A capability may ship a kit-owned intent-grant fragment (`.pkit/capabilities/<cap>/permissions/grants.yaml`) that the single model loader composes into the effective permission model. The fragment is a new kit-owned file role read by `load_model`, discovered by walking the installed-component manifest, layered before the active profile and the adopter's `grants.yaml` so the operator's profile choice and manual grants still win on the allow side while a capability *deny* survives via deny-wins.**

**1. Composition & layering.** `load_model` gains a new layer, ordered:

```
guardrail_denies + capability-fragments + active-profile-grants + adopter grants.yaml
```

Capability fragments sit **before** the profile and the adopter's grants. The consequence, read against `decide()`'s deny-wins, order-independent composition: on the **allow** side the operator's chosen profile and any manual `grant`/`revoke` are peers unioned later, so the operator's intent is never narrowed away by a capability allow it did not ask for; on the **deny** side a capability `effect: deny` survives regardless of layer because deny wins. An adopter who wants to *override* a capability deny does so by authoring an explicit `grant` — a deliberate, auditable gesture in their own file, not a silent default.

**2. Discovery / install-gate.** Fragment discovery **walks the manifest `components:` list** (`.pkit/manifest.yaml`), reusing [project-management:DEC-030](../../../.pkit/capabilities/project-management/decisions/DEC-030-capability-contributed-adapter-overlays.md)'s manifest-scoped-walker pattern — **not** a glob of `.pkit/capabilities/`. An uninstalled or orphan capability directory (left by a botched uninstall, stash, or rebase) contributes nothing, because it is not a registered component. Install-state is the gate: a capability's fragment is composed exactly when the capability is an installed component.

**3. Same-code invariant (ADR-002 / ADR-003).** The fragment-merge logic lives in the propagated `decide.py` `load_model`, which the PreToolUse hook runs with bare `python3` in the adopter tree where the global `pkit` is not importable ([ADR-003](ADR-003-permission-core-code-home.md)). It MUST therefore use the existing stdlib-only YAML fallback path (`load_yaml`'s ruamel-or-stdlib loader) and the existing manifest read — **no `src/project_kit` import, no third-party-only dependency.** One loader, both readers (hook + CLI), byte-identical decisions ([ADR-002](ADR-002-permission-realizer-ownership.md) same-code invariant). The fragment reuses the existing `grants.yaml` shape (`schema_version` + a `grants:` list of `{subject, privilege, effect, scope?}`), so no new schema and no new parser path are introduced.

**4. This is the realizer-channel DEC-030 anticipated — not the overlay channel.** The fragment is deliberately **not** routed through [project-management:DEC-030](../../../.pkit/capabilities/project-management/decisions/DEC-030-capability-contributed-adapter-overlays.md)'s adapter-overlay mechanism. DEC-030 **reserves the `permissions` key**: an overlay's `permissions` block is silently overridden by `merge-settings.sh` step 1 and is "effectively a no-op," and DEC-030 explicitly **defers permission contribution to the COR-028 realizer** as the sanctioned owner of that region. An overlay reaches `.claude/settings.json`; the intent model is consumed from the model loader, not from settings — so an overlay-routed grant would be silently dropped. DEC-030 is therefore the **authority that confirms the overlay channel is wrong** for this; the grant fragment is the kit-owned-file-role-read-by-the-model-loader that DEC-030's deferral note points at. **No DEC-030 amendment is needed** — this lands squarely in the channel DEC-030 reserved against itself.

**5. Relationship to ADR-015 — this is the *first* capability-fragment-merging substrate, not a twin of ADR-015.** [ADR-015](ADR-015-command-declared-network-egress.md) is the **confinement-axis** sibling; this is the **intent-axis** one. They are siblings in spirit (a capability declaring its own permission needs) but **not** structural twins, and this ADR does not reuse ADR-015's machinery: ADR-015's GitHub egress ships in the **backbone** `confinement-toolkit.yaml`, gated by a `detect:` glob and routed through the single sandbox-block writer — it is **not** a capability fragment, so **no capability-fragment-merging substrate exists today**. This ADR builds the first one. The merge point also differs: ADR-015 writes the `sandbox` block of `.claude/settings.json` through the provenance-tagged writer; this composes a layer inside the model loader and writes nothing to settings.

**6. The privilege stays in the backbone catalog; the policy is the capability's (deferred seam CA-1).** `issue-tracker-write` remains a backbone `privilege-catalog` privilege (`.pkit/schemas/privilege-catalog.yaml`). The privilege **vocabulary** is harness- and capability-neutral (it names a class of action any project might recognise); only the **policy** (deny it to this agent) is the capability's. The fragment references the privilege by its [COR-019](../../../.pkit/decisions/core/COR-019-schema-reference-form.md) token `[privilege-catalog:issue-tracker-write]`, exactly as an adopter grant does. Relocating the *privilege definition itself* into a capability-owned catalog fragment is recorded as **deferred seam CA-1** (see Implications) — not built now, because the recurrence test is not yet met (one consumer).

**7. Narrowing-but-reported honesty.** A capability's default-on `deny` **narrows** an agent's reach (it removes a privilege, lowering capability), so it is safe to auto-apply on capability install per [ADR-008](ADR-008-confinement-allowances.md)'s narrowing class — it carves nothing *out* of the box and grants nothing the operator did not have. But it must be **surfaced**: `pkit permissions overview` / `explain` MUST show that a capability contributed the deny, so the operator can see why an agent is denied a privilege and which capability owns that policy. This mirrors [ADR-015](ADR-015-command-declared-network-egress.md)'s **narrowing-but-reported** posture — auto-applied like narrowing, mandatorily-reported so the model never silently changes an agent's reach without the operator being able to see the source.

**8. First consumer.** The project-management capability ships `.pkit/capabilities/project-management/permissions/grants.yaml` granting `effect: deny` to `agent:project-manager` on `[privilege-catalog:issue-tracker-write]`. This **replaces** the manual grant currently in project-kit's own `.pkit/permissions/project/grants.yaml` (which the implementation removes), relocating the policy from the adopter-owned file to the capability that owns it.

## Rationale

**Why the general substrate (direction A), not a hardcoded floor.** A hardcoded project-manager deny baked into `decide.py` or the backbone catalog would (a) couple the neutral decision core to one capability's policy — the same coupling [ADR-003](ADR-003-permission-core-code-home.md) forbids when it keeps recognizers as data and bars core→adapter edges; (b) leave every *other* capability with a permission need still homeless, re-forcing the manual-adopter-grant footgun this issue exists to remove; and (c) make the deny survive even after project-management is uninstalled, the orphan-contribution problem DEC-030's manifest walk solves. The substrate is justified on the recurrence test ([COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)) read at the *channel* level: capability-contributed permission needs are already a demonstrated pattern across both permission axes — ADR-015 built the confinement-axis channel for exactly this shape, and the intent axis now has its concrete first consumer. The substrate extracts the channel; the shipped fragment is the single illustrative instance, kept minimal (principles-not-inventory).

**Why before the profile and the adopter's grants.** The composition order is the load-bearing safety choice. Placing capability fragments **before** the profile and `grants.yaml` means the operator's two sovereign levers — the active profile (their chosen posture) and their hand-authored grants — are unioned *later*, so on the allow side a capability can never narrow what the operator deliberately granted: the operator's grant is a peer that survives. The adopter-last placement is the same model-side anti-clobber property [ADR-005](ADR-005-permission-profile-surface.md) rule 2 established for the profile layer, extended one source earlier. A capability *deny* still survives via deny-wins regardless of order, which is correct: a capability that needs an agent denied a dangerous privilege should get that deny by default, and the operator overriding it must do so **explicitly** — an auditable `grant` in their own file — rather than the deny being silently absent. This keeps the deny default-on-but-overridable, the honest posture for a safety-narrowing default.

**Why the manifest walk, not a directory glob.** Reusing DEC-030's manifest-scoped walker is the difference between "installed capabilities contribute" and "any directory under `.pkit/capabilities/` contributes." DEC-030 chose the manifest walk precisely so orphan directories (botched uninstall, stash, rebase) cannot silently contribute settings; the identical hazard applies to intent grants — an orphan capability silently denying an agent a privilege would be a confusing, un-attributable model change. Install-state-as-gate keeps the contribution honest: present iff the capability is a registered component, gone the moment it is uninstalled.

**Why the same-code stdlib path is non-negotiable.** The PreToolUse hook runs `decide.py` with bare `python3` in the adopter tree, where `ruamel.yaml` may be absent (notably inside macOS Seatbelt, [ADR-014](ADR-014-macos-sandbox-platform-stance.md)) and the global `pkit` is never importable ([ADR-003](ADR-003-permission-core-code-home.md)). The fragment-merge must read fragments through the existing `load_yaml` fallback and the existing manifest read, with no new dependency — otherwise the hook and the CLI would parse fragments through different paths and could decide differently, breaking the [ADR-002](ADR-002-permission-realizer-ownership.md) same-code invariant the conformance fixtures exist to prove. Reusing the existing `grants.yaml` shape means the fallback already parses the fragment correctly (it is byte-for-byte the same node shape it parses for adopter grants today).

**Why DEC-030 confirms the channel rather than needing amendment.** DEC-030 already settled that capability permission contribution does **not** go through the adapter-overlay channel — it reserves `permissions` against overlays and defers it to the COR-028 realizer "as the sanctioned owner of that region." The model loader *is* that realizer-side surface. So this ADR is not carving a new exception to DEC-030; it is filling the slot DEC-030 left open. Amending DEC-030 would imply the overlay channel was a candidate; it was explicitly not.

### Alternatives considered

- **Mechanism 1 — the capability ships a permission *profile*.** Rejected. [ADR-005](ADR-005-permission-profile-surface.md) rule 1 pins **one active profile per project**; a capability shipping a profile would either contend for the single active slot (only one capability could ever contribute) or force a multi-profile model ADR-005 deliberately rejected. A profile is the *operator's* project-wide posture choice, not a per-capability contribution. The layered grant fragment adds a source without touching the one-active-profile invariant.
- **Mechanism 2 — seed the adopter's `grants.yaml` on capability install.** Rejected. This bars on two ADR-005/ADR-008 invariants at once. The adopter's `grants.yaml` is **adopter-sovereign and composed last** ([ADR-005](ADR-005-permission-profile-surface.md) rule 2) — a capability writing into it inverts the ownership (a capability mutating the operator's hand-authored file), and on uninstall would require strip-logic to distinguish the seeded grant from a manual edit, the exact silent-deletion / single-writer footgun [ADR-002](ADR-002-permission-realizer-ownership.md) and [ADR-008](ADR-008-confinement-allowances.md) rule 2 close. A capability-owned fragment read as its own layer keeps the single-writer property: the adopter owns `grants.yaml`, the capability owns its fragment, and uninstall removes the fragment with the capability — no strip-logic.
- **The narrow hardcoded floor (the direction the operator chose against).** Considered: hardcode the project-manager `issue-tracker-write` deny as a fixed floor (in `decide.py` or the backbone catalog), no general substrate. Rejected by the operator in favour of direction A: it couples the neutral core to one capability's policy, leaves every other capability's permission need homeless, and re-imports the orphan-contribution problem (the deny would outlive uninstall). The general substrate costs one layer in `load_model` and a manifest walk already proven by DEC-030; the hardcode saves that at the cost of relitigating the same problem for the second consumer.
- **Route the grant through the DEC-030 adapter-overlay channel.** Rejected — DEC-030 reserves the `permissions` key against overlays (silently overridden, "effectively a no-op") and defers permission contribution to the COR-028 realizer. An overlay-routed grant reaches `.claude/settings.json`, not the model loader, so it would be silently dropped. This is the authority that *confirms* the model-loader channel is the right one.

## Implications

- **Child of ADR-005 and ADR-003; sibling-in-spirit of ADR-015; no decision superseded, no DEC-030 amendment.** This extends `load_model`'s composition by one source ahead of the profile, preserving ADR-005's adopter-last anti-clobber property and ADR-003's same-code / dependency-direction invariants. It is the intent-axis analogue of ADR-015's confinement-axis capability contribution, but reuses none of ADR-015's machinery (no `confinement-toolkit`, no sandbox-block writer, no `detect:` glob). It fills the realizer-side slot DEC-030 reserved; DEC-030 stands unchanged.
- **New kit-owned file role: `.pkit/capabilities/<cap>/permissions/grants.yaml`.** Core-owned (propagated on capability install/sync; adopters do not edit it, edits are overwritten — the [ADR-003](ADR-003-permission-core-code-home.md) propagated-tree property). It reuses the existing `grants.yaml` schema shape, so it introduces **no new schema** and rides the existing JSON Schema companion for grant documents.
- **`load_model` and the read surface are implementation work (same issue #72), after this ADR in the same change.** The new layer in `load_model` (manifest-walked fragment composition through the stdlib-safe loader), the `pkit permissions overview` / `explain` attribution that surfaces "capability `<cap>` contributed this deny," the manifest-scoped fragment walker, and the relocation of the project-manager deny from `.pkit/permissions/project/grants.yaml` into the project-management capability's fragment all land in the implementation step. This ADR records only the decision.
- **Deferred seam — CA-1: relocate the privilege *definition* into a capability catalog fragment.** Today the `issue-tracker-write` privilege **definition** lives in the backbone catalog (neutral vocabulary) and the capability fragment references it by token. A future seam (CA-1) would let a capability ship its *own* privilege-catalog entries — relocating both the policy *and* the vocabulary into the capability. It is **deferred**, justified by [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)'s recurrence-not-yet-met: there is one consumer today, the privilege is genuinely backbone-neutral, and a capability-owned catalog fragment is speculative generality until a second capability needs a privilege the backbone catalog should not carry. Recorded here so the build does not over-reach into it, and so the recurrence is the trigger to revisit.
- **The narrowing-but-reported gloss is mandatory on the model read surface.** A capability deny narrows reach (safe to auto-apply on install per [ADR-008](ADR-008-confinement-allowances.md)) but MUST be attributed by `pkit permissions overview` / `explain`, mirroring [ADR-015](ADR-015-command-declared-network-egress.md)'s posture — the operator must always be able to see that a capability contributed a deny and which capability owns it. Silently changing an agent's reach without surfacing the source is the dishonesty [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) forbids.
- **COR-010 / versioning.** A **pure addition** for [COR-010](../../../.pkit/decisions/core/COR-010-resource-lifecycle.md): no file/directory rename or removal in a kit-owned tree, no `schema_version` bump (the fragment reuses the existing grants shape), no breaking CLI signature change, no capability-subtree restructure — so **no migration**. The implementation MUST run `pkit migrations check-diff --include-working-tree` to confirm COVERED. It **is** a surface change for versioning per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md) (a new contribution mechanism plus a new convention adopters follow), so `.pkit/VERSION` bumps in the implementing PR — that bump rides with the implementation commit, **not** this ADR.
