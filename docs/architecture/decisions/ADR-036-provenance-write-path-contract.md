## Summary

**In plain terms:** [project-management:DEC-041](../../../.pkit/capabilities/project-management/decisions/DEC-041-version-provenance-stamp.md) decided that every script-filed issue/PR carries a version-provenance footer at the foot of its body. This contract says every script that writes a body stops appending or replacing that footer by hand and instead calls **one shared module** that, on every write, **strips any provenance region out of the incoming body and reissues exactly one fresh footer**. Because each write strips-all-then-appends-one, the result is one correct footer no matter what state the incoming body was in — so a footer can never be doubled, orphaned, or stale-by-construction. A guard test then checks that no script writes an issue/PR body except through that module.

This is the provenance-side member of the capability's seam family — the write-side twin of the substrate-write contract in [ADR-031](ADR-031-substrate-write-path-contract.md), which is itself the non-label, write-side twin of the label read-path contract in [ADR-026](ADR-026-substrate-map-read-path-contract.md). It inherits their construction-point-plus-guard shape. What it adds — the reason it is a record of its own — is a boundary the other two do not face: **the managed region lives inside a document the agent edits.** The substrate writes ADR-031 governs are wholly script-owned (the agent never supplies the `gh` argv); the footer is a script-owned region embedded in an agent-authored body, because `edit-issue` hands the agent the entire body to rewrite. DEC-041 commits to the property — *agents never own the provenance bytes*; this ADR fixes the mechanism that makes that property hold even though the agent rewrites the surrounding document.

## Context

DEC-041 decided the *what*: a one-time immutable filing comment plus a self-replacing, versions-only footer, both written only by the capability's scripts, never typed by an agent. It deferred the *how* of keeping the footer correct to this record.

The footer is a methodology-managed region sitting at the end of an issue/PR body. Bodies are written today from several scattered scripts — `create-issue`, `edit-issue`, `open-pr`, `edit-pr`, `create-milestone`, the criterion-checkbox verbs (`check-criterion` / `uncheck-criterion`), `set-field`, and the doc-mapping check — each calling `gh issue edit --body-file` (or the PR equivalent) directly. This enumeration is as-of-authoring and serves only to size the convergence: it is a reading aid, not the contract's allow-list. The guard's authority (item 4) is the **scan-all over every pm script**, not this hand list — naming the concrete scripts is in scope for a project-kit capability-architecture record per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md), exactly as the sibling contracts enumerate their sites. There is no single body-write chokepoint today. This is the same pre-contract convergence surface ADR-026 faced for labels (~26 sites) and ADR-031 faced for non-label substrate writes (four sites): an enumerable set of construction points that must converge on one primitive for any whole-body invariant to hold.

The footer faces one hazard the substrate-write seam does not. `edit-issue` fetches the body, hands the **entire text to the agent** to produce a new version, and writes the result back. The agent therefore sees the raw region markers. The footer markers are the worst possible thing to ask a generative model to preserve byte-exact: they are semantically empty and invisible in rendered Markdown, so a reflow, a whitespace normalisation, or a "tidy" pass can drop or split one. A naive find-the-pair-and-replace helper then fails to match the damaged region and *appends a second footer*, leaving the orphaned remnant behind. The region is script-owned by intent but agent-reachable by mechanism — a boundary ADR-026/ADR-031 never had to reason about, because their constructed artifacts never pass through an agent's hands.

## Decision

1. **One sole-constructor seam for the footer.** A single `_lib` module (the provenance helper) is the only place a provenance footer is constructed onto a body. Every body-writing script obtains its written-back body by passing the new body text through this module, never by appending or splicing a footer inline. This is ADR-026's sole-constructor instinct (part i) and ADR-031's single-construction-point, applied to the footer region.

2. **The construction is strip-then-append-exactly-one.** On every invocation the module (a) removes *any* provenance region from the incoming body — complete, partial, orphaned, or doubled — by stripping from the provenance start-marker through end-of-document, then (b) appends exactly one freshly generated footer. Because the strip is unconditional and total and the append is singular, the operation is **idempotent regardless of the incoming body's state**: a clean footer, a mangled one, two footers, or none all produce one correct footer. Doubling is therefore structurally impossible rather than cleaned up after — there is no code path that yields two regions. This is the load-bearing invariant; the rest of the contract protects it.

3. **The agent never owns the footer bytes — enforced on the write side, reinforced on the read side.** The write-side strip (item 2) is the guarantee: whatever the agent did to the markers it was handed is erased and reissued, so the agent's handling is irrelevant by construction. As defence-in-depth, the agent's read path (`show-issue`, the methodology view it composes edits against) strips the footer from the body it emits, so the agent composes against footer-free text and has no markers to mishandle in the first place. The read-side strip lowers the rate at which damaged input reaches the seam; it is not what makes the property hold.

4. **A guard test proves no body write bypasses the seam.** A grep/AST guard over the body-writing scripts asserts that none constructs an issue/PR body write (`gh issue edit --body-file` / `gh issue create --body-file` and the PR equivalents) outside the provenance module's call path. This is the second half of the ADR-026/ADR-031 construction-test-plus-guard split: the construction test proves the module builds the region correctly; the guard proves the module is the *only* builder. Half (a) without half (b) is the documented failure mode — a stray inline body write that silently skips the seam — so both halves ship together.

5. **The footer's shape is schema-constrained so an un-stripped footer is inert.** The `provenance_marker` entry the schema skill adds to `body-format.yaml` constrains what the module may emit so that the footer contains nothing a body validator or a checkbox extractor could mis-scan — the entry itself carries the concrete token list, which is schema content, not pinned here. The constraint guarantees that even a footer that some future reader forgot to strip cannot collide with the body validators or the checkbox extractors. This demotes the read-side strip from a correctness precondition to defence-in-depth: readers that strip are tidy; readers that forget still cannot misread the region. It is the contents-level analogue of the construction-test/guard split — the schema says what a valid footer may contain, the module is the sole emitter that satisfies it.

6. **Failure posture is inherited, not redefined.** The module builds-and-runs the write and reports what happened; it takes no view on what to do when a write fails. The best-effort/report-and-continue posture lives with the caller, per DEC-041 — exactly as ADR-031's module delegates the skip/abort decision to its callers (the per-event hook and the bulk back-fill hold their own postures).

## Rationale

**Why strip-then-append rather than find-and-replace.** The brittle alternative — locate the existing `start…end` pair and substitute it — fails precisely in the case the agent-owned-document boundary makes common: a damaged marker makes the pair unmatchable, the replace silently degrades to an append, and the body ends with a remnant plus a new region. Strip-then-append-exactly-one removes that failure mode at the root by never depending on the incoming region being well-formed. The cost is trivial (the footer carries no per-edit date per DEC-041, so reissuing identical versions is a no-op diff), and the benefit is that idempotency holds against *arbitrary* prior state, including state produced by an agent that mangled the markers.

**Why this is a record distinct from ADR-031.** ADR-031's invariant ("every non-label substrate write goes through one constructor") could be stated without ever mentioning an agent, because those writes are wholly script-owned. The footer cannot: its whole difficulty is that a script owns a region inside a document an agent rewrites. Folding this into ADR-031 would bury the one architecturally novel boundary — script-owned region in an agent-owned document — under a contract that does not share it. Pinning it as the seam family's third member, citing the first two as the inherited pattern, keeps each record about one boundary. This mirrors how ADR-031 split from ADR-026 rather than extending it.

**Why a guard, not just review.** The convergence is only as strong as its weakest forgotten call site; a future body-writing script that calls `gh` directly reintroduces the hazard invisibly. The guard makes the invariant machine-checked at the same tier ADR-026 and ADR-031 chose, so the contract does not decay as scripts are added.

### Alternatives considered

- **Demote the footer to best-effort, no seam, no guard** (the lighter option weighed in review). Rejected once the strip-then-reissue mechanism made the strong guarantee cheap: with idempotent strip-on-write the convergence is a handful of enumerable sites, not the open-ended refactor that justified demotion, and the agent-owned-document hazard is removed rather than tolerated. DEC-041's filing comment remains the load-bearing record regardless; the footer being correct-by-construction is now low-cost enough to be worth it.
- **A read-time triage command instead of a written-in footer.** Already rejected in DEC-041's rationale (an issue's events are not git events, so timestamp-versus-history is a lossy heuristic). Out of scope here; this ADR governs only the write path of the chosen design.

## Implications

- **The provenance module is authored at implementation time**, once this record is accepted — the sole constructor of the footer region, exposing the strip-then-reissue write and the one-time filing-comment helper, reusing the comment-idempotency marker the lifecycle-hook engine already ships rather than reinventing it.
- **The body-writing scripts converge** on the module for their written-back body; the guard test lands with them. The enumerable site set is the convergence surface; adding a future body-writing script means routing it through the seam or failing the guard.
- **`show-issue` gains the read-side strip** so the agent composes edits against footer-free text.
- **The `provenance_marker` schema entry** (authored via the schema skill, citing DEC-041) carries the shape constraint this contract relies on; it is additive to `body-format.yaml` (no `schema_version` bump, no migration).
- **Accepted by the maintainer before the trunk implementation builds against it** (per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md)) — a forward design contract, not self-accepted. The provenance module, the converged body-writers, the guard test, and the `show-issue` strip land citing this record once it is accepted.
- **This ADR pins the contract, not the feature UX.** The footer's content, the recorded version set, and the accepted stamp-less gap are DEC-041's; this record must not accrete them. On-call/maintainer reference: if a body shows two footers or a stale one, the seam or the guard has been bypassed — that is the first thing to check.
