## Context

[ADR-004](ADR-004-autonomy-intent-confinement.md) pinned the *model* of agent autonomy — autonomy is **intent × confinement**, pkit owns intent and the enforcing-layer declaration and delegates confinement to the OS sandbox, confinement is session-wide (never per-agent), and a fail-closed invariant governs any confinement-conditioned prompt-suppression. ADR-004 **deferred the `pkit permissions profile` surface itself**, naming it demand-driven and flagging that it "lands once a real second consumer and the `apply` substrate exist," and that it "needs a fresh architect pass when proposed" (ADR-004 §63, §67). Issue #255's profile-surface design is that fresh pass.

The model is settled and the verbs a profile would compose already exist and are accepted: `grant`/`revoke` mutate the grant model, `mode` writes posture + ownership, `apply` additively realizes the model into `.claude/settings.json`, `enable`/`disable` toggle the live PreToolUse hook ([ADR-002](ADR-002-permission-realizer-ownership.md) §64, #247 — opt-in, orthogonal). The decision core's single `load_model` ([ADR-003](ADR-003-permission-core-code-home.md) same-code invariant; `.pkit/permissions/decide.py`) currently composes the effective model as `guardrail_denies(catalog) + adopter grants.yaml`, deny-wins and order-independent in `decide()`. The privilege catalog is a schema'd-data area (COR-018) and the `sandbox` block in the claude-code adapter settings is declared by ADR-004 but **not yet wired** (no writer exists).

The design was worked collaboratively across six questions (granularity, contents, location, verbs, shipped tiers, fail-closed activation) in the `2026-06-01-permission-profile-surface.md` scratchpad, and reshaped by a `critic` pass that caught two failure modes the first cut would have shipped: a **god-object** (one `apply` bundling posture + grants + sandbox + native-allow, re-breaking adopter sovereignty and the axis separation) and a **model-ownership footgun** (a profile owning `grants.yaml` wholesale would silently delete an adopter's manual grants — the model-side analogue of ADR-002's §52 Fork-1 silent-deletion). This ADR records the surface decision so a future build cannot relitigate the reshaped form back into either footgun. The proposed status is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md); as project-kit's own architecture record, harness specifics are in scope.

## Decision

**A permission profile is one per-project named binding of {a posture} × {a swappable, layered per-agent intent grant-source}, with confinement and the native-allow tier *referenced* rather than written. It composes the existing verbs; it introduces no new ownership tier and no second mutation engine.**

**1. Granularity — one active profile per project, carrying per-agent grant overrides.** A project has exactly one active profile. It sets the project-wide posture (the session-wide axis the sandbox forces per ADR-004 §41) and carries per-agent *intent grant* overrides inside it (e.g. `pm: +scripting`, `critic: read-only`). There are no independent per-agent profiles — ADR-004 §41 forbids per-agent confinement (the sandbox is session-wide), so the only per-agent lever a profile may carry is the tool-gating / intent sub-axis, expressed as per-subject grants. Project-wide posture + per-agent intent overrides, under one name.

**2. Contents — posture plus a *layered* grant-source; confinement and native-allow referenced, not owned.** A profile contributes a **named grant layer**, not ownership of `grants.yaml`. `load_model` composes the effective model as `guardrail_denies + active-profile-grants + adopter grants.yaml`, with the **adopter's `grants.yaml` last** so a manual `grant`/`revoke` is never clobbered by the active profile. Switching profiles swaps the active layer; `grants.yaml` is untouched. Confinement (the `sandbox` block) and the native-allow shell tier (#266, adopter-owned) are **referenced** by a profile — checked, glossed, depended upon for the fail-closed invariant — but **never written** by it: native-allow is adopter-sovereign per ADR-002's partition, and the `sandbox` block has no writer yet (deferred substrate, see §below). A profile is therefore composed only of surfaces that swap or overwrite cleanly: posture (overwrite) and the grant-layer pointer (swap).

**3. Location — a `permission-profile` schema'd YAML (COR-018).** A profile is data, not code: a `permission-profile` schema with its JSON Schema companion (`.pkit/schemas/`, per the area conventions). Backbone ships the named tiers (`read-only`, `non-destructive`, `autonomous`) in `.pkit/permissions/profiles/` (neutral, propagated, like the catalog); projects define or override in `.pkit/permissions/project/profiles/<name>.yaml` (a same-name project file overrides the shipped one). The active pointer is `active_profile: <name>` in `.pkit/permissions/project/config.yaml` — a new optional field on the existing `permission-config` schema, adopter-owned project state alongside `posture`/`ownership_mode`.

**4. Verbs — `profile list` / `show` / `activate`, composing existing mutations.** `profile list` (read-only) enumerates available profiles (shipped + project), marking the active one. `profile show [<name>]` (read-only) renders a profile's contents (defaults to active). `profile activate <name>` (mutating) sets the active profile: it writes `active_profile`, sets posture (reusing `set_mode`), activates the profile's grant-layer, **and runs `apply` by default** to realize the result; `--no-apply` defers the realize (model-only). It is idempotent. It does **NOT** auto-`enable` the live hook — enforcement is the orthogonal #247 opt-in (ADR-002 §64) — and instead detect-and-nudges ("enforcement is OFF — run `pkit permissions enable`") when the hook is not registered.

> **Reconciliation (2026-06-02):** the activation verb was originally shipped as `profile use`. It was renamed to `profile activate` (hard rename, no alias) so the verb agrees with the persisted state vocabulary (`active_profile`) and with the `enable`/`disable` pattern — per the verb↔state-agreement convention captured in `.pkit/scratchpad/active/2026-06-01-cli-design-conventions.md`. The decision below (compose verbs, don't auto-enable) is unchanged; only the verb's spelling changed. References to `use` further down read as `activate`.

**5. Shipped tiers nest; grants are over catalog privileges; shell autonomy rides the references.** Backbone ships `read-only` ⊂ `non-destructive` ⊂ `autonomous`. A profile's grants are over **catalog** privileges only; the shell tiers (text-processing / cleanup / scripting) are not catalog privileges, so a profile *references* the native-allow tier (#266) and the sandbox rather than granting them. The nesting:

| Profile | posture | per-agent grants (catalog) | references (not written) |
|---|---|---|---|
| `read-only` | strict | `repo-read` | — |
| `non-destructive` | lenient | `vcs`, `issue-tracker`, `kit`, `repo-read` | low-risk native-allow tier (#266) |
| `autonomous` | lenient | + `docker`, `web-fetch` | + prompt-free scripting via the **sandbox** (confinement-ref) |

**6. Fail-closed activation, and an operator-only, never-committable unconfined override.** Any profile that would enable prompt-free scripting (`autoAllowBashIfSandboxed`) MUST pair it with `failIfUnavailable: true` (and `allowUnsandboxedCommands: false`) per ADR-004 §39 — on a host without the OS sandbox the session refuses rather than running scripting unconfined-and-unprompted. This is recorded here as a constraint the future sandbox writer inherits; a profile cannot carry a fail-open variant. The escape hatch is **`--dangerously-allow-unconfined`** (name mirrors the harness's `dangerously*` convention) on `profile use`/`apply`. It is an **operator gesture at command time, NEVER a profile field**: it cannot be baked into a committed profile YAML (else a repo would silently ship fail-open for every checkout), it is loud and re-passed per invocation (the `--force` pattern, scarier-named), and it is auditable.

> **Reconciliation (2026-06-03, #274):** the sandbox writer landed as `pkit permissions sandbox enable | disable | status`, and building it against the live harness semantics (Claude Code v2.1.161) reconciled one clause above: **`allowUnsandboxedCommands: false` is NOT required for the fail-closed invariant.** Verified semantics: auto-allow applies only to commands that actually run *inside* the box; the unsandboxed *fail-over* path (a command failing under the sandbox, retried outside it) rides the normal permission flow — allowlist or prompt — and is **never auto-allowed**, so it cannot produce the "unconfined-and-unprompted" failure mode ADR-004 §39 forbids. Forcing `false` would meanwhile break legitimate fail-over (`git push` / `gh` need network/SSH reach the box blocks). The writer therefore leaves `allowUnsandboxedCommands` at the harness default and offers `--strict` as optional hardening; the invariant's load-bearing control is `failIfUnavailable: true`, which `enable` always writes. The `--dangerously-allow-unconfined` flag lives on `sandbox enable` (the surface that writes the block) rather than `profile use`/`apply`; its semantics are unchanged — per-invocation, loud, never persisted as a default (re-running `enable` without it restores the fail-closed floor). `enable` additionally writes a credential `denyRead` floor (`~/.ssh`, `~/.aws`, `~/.config/gh`, `~/.netrc`), which the default sandbox read policy would otherwise permit.

**Buildable scope.** The **minimal** profile — posture + the swappable grant-layer, with confinement and native-allow referenced/checked but not written — is **sanctioned to build now**. The confinement-writing and drift-healing parts are **deferred** to their substrate (see Implications).

## Rationale

**Why a layered grant-source rather than profile-owned `grants.yaml`.** A profile that regenerated the grant model wholesale would silently delete an adopter's hand-authored `grant`s on every switch — the model-side twin of ADR-002 §52's Fork-1 silent-deletion footgun, where wholesale region replacement at insufficient fidelity deletes content the adopter never asked to lose. Layering avoids it structurally: the profile is an *input* to `load_model`, unioned **before** the adopter's `grants.yaml`, so `decide()`'s deny-wins, order-independent composition (`.pkit/permissions/decide.py`) treats profile grants and manual grants as peers and the manual layer always survives. This is a sound extension because it adds **no new ownership tier** — `load_model` already composes `guardrail_denies + grants.yaml`; the profile inserts one more additive source in the middle. It is the additive merge philosophy ADR-002 makes the non-destructive default, applied to the model instead of to settings. Crucially it does **not** touch managed mode or wholesale region replacement, so it cannot reach the §52 Fork-1 surface at all.

**Why `use` composes verbs and does not auto-enable.** `profile use` reuses `set_mode` (posture) + a grant-layer swap + the existing additive `apply`; it builds no second mutation engine, so realization stays the one audited `project()`→union path (ADR-002 same-code; `apply` is set-union, idempotent). Auto-enabling the hook would re-fuse activation with enforcement, contradicting ADR-002 §64's settled #247 opt-in (enforcement is the adopter's explicit, separately-reversible choice). Detect-and-nudge keeps the two orthogonal while still steering the operator — the same posture `overview`'s enforcement banner already takes.

**Why the override is command-time-only and never a field.** ADR-004 §39 makes fail-closed an *invariant*, not a toggle: the unsafe variant (prompt-free + unconfined) is never the right committed default. A committable fail-open field would let one author silently lower the floor for every checkout of the repo — the exact "believes a boundary holds that does not" trap ADR-004 §49 names. Confining the override to a loud, per-invocation operator gesture means each operator accepts the risk on their own machine, the committed profile stays fail-closed, and the override is auditable. This keeps the fail-closed invariant intact while still offering an honest escape hatch (COR-028's honesty discipline).

**Why minimal-first rather than whole-thing-deferred (COR-007).** The recurrence test is met for the *intent* half: the pm-autonomy showcase plus the near-term adopters establish a real consumer for named posture + per-agent grant bundles, and every piece the minimal profile needs already exists (posture via `set_mode`, grants via the layered `load_model`, realization via additive `apply`, the catalog, the schema mechanism). Nothing in the minimal profile is blocked. The *confinement* half is genuinely blocked — the `sandbox` writer does not exist (ADR-004 §76 defers it, and managed-mode healing is #252-deferred per ADR-002 §61) — so building it now would be speculative generality against unbuilt substrate. Splitting at "writes cleanly-revertible surfaces only" lands the demand-backed value now and leaves the blocked surfaces referenced-but-unwritten until their substrate arrives, exactly COR-007's build-when-the-need-is-concrete discriminator.

### Alternatives considered

- **One profile `apply` bundling posture + grants + native-allow + sandbox (the god-object).** Rejected (critic). Writing native-allow re-breaks the adopter-sovereignty partition ADR-002 protects (#266 is adopter-owned, session-wide, no per-agent dimension); writing sandbox before its writer exists is speculative; fusing all four into one mutation rebuilds the second mutation engine ADR-002's `apply` already is. The reshaped profile composes existing verbs and writes only cleanly-revertible surfaces.

- **A profile that owns `grants.yaml` wholesale.** Rejected (critic) — the model-side §52 Fork-1 footgun: switching profiles would silently delete the adopter's manual grants. The layered grant-source preserves them by composition order.

- **Independent per-agent profiles.** Rejected — ADR-004 §41 forbids per-agent confinement (session-wide sandbox). Per-agent expression is carried as intent-grant overrides *inside* the one project profile, never as separate per-agent confinement profiles.

- **`profile use` auto-enables the hook.** Rejected — contradicts ADR-002 §64 (#247 opt-in). Activation and enforcement stay orthogonal; `use` nudges instead.

- **A committable `allow_unconfined` profile field instead of a command flag.** Rejected — would let a committed repo ship fail-open for everyone, violating ADR-004 §39's fail-closed invariant. The override is an operator-only, per-invocation gesture.

- **Defer the whole surface until the sandbox writer and managed `apply` exist (status quo from ADR-004 §63).** Rejected for the intent half (its substrate exists and a consumer is present — COR-007 recurrence met), retained for the confinement half (its substrate does not exist).

## Implications

- **No new ownership tier; child of ADR-004 and ADR-002.** The profile is an additive *input* layer to the existing `load_model`; it changes neither the intra-file ownership partition (ADR-002) nor the intent/confinement split (ADR-004). It rides the additive `apply` (#250), not managed-mode wholesale regeneration (#252, still deferred).

- **`load_model`'s same-code contract is preserved.** The profile layer is added inside the single `load_model` in `.pkit/permissions/decide.py`, so the CLI and the PreToolUse hook continue to decide and display from one byte-identical model (ADR-003). Because `decide()` is deny-wins and order-independent, inserting the profile layer as `guardrail_denies + profile-grants + grants.yaml` changes neither the precedence semantics nor the hook's contract — guardrail denies still win, manual grants still survive. The build MUST keep the composition in `load_model` only (no parallel reader).

- **New `permission-profile` schema (COR-018).** A new schema + JSON Schema companion under `.pkit/schemas/`, fields `{schema_version, description, posture, grants (per-subject)}` (the profile's *name* is its filename, not a field; a confinement field is intentionally omitted until the sandbox writer lands), consumed by the `profile` CLI. As a *pure addition* (new schema, new shipped data files, a new optional `active_profile` field on `permission-config`) this does not trip a COR-010 migration on its own; the build MUST run `pkit migrations check-diff --include-working-tree` to confirm, and it is a surface change for versioning (new CLI verbs + new schema + new principle), so it bumps the affected component per PRJ-002.

- **Backbone ships three profiles; projects override by same-name.** `.pkit/permissions/profiles/{read-only,non-destructive,autonomous}.yaml` are propagated, neutral, catalog-only-grants. `.pkit/permissions/project/profiles/<name>.yaml` is adopter-owned and overrides by name. The active pointer is `active_profile` in the project `config.yaml`.

- **The deferred-substrate boundary is explicit.** The minimal profile **references** the native-allow tier (#266) and the `sandbox` block but writes neither. When the sandbox writer lands (ADR-004 §76), it — not the profile — owns the whole `sandbox` block (clean revert), enforces the §39 fail-closed invariant (`failIfUnavailable: true` / `allowUnsandboxedCommands: false`), and only then does a profile's `autonomous` tier gain real prompt-free scripting. Healing additive-no-heal surfaces (native-allow, wholesale grants) remains blocked on #252. The `--dangerously-allow-unconfined` override is meaningful only once the sandbox writer exists; until then a profile cannot enable prompt-free scripting at all, so the flag is reserved, documented, and inert.

- **Build sanction: minimal-now, confinement-deferred.** This ADR **sanctions building** the minimal profile (posture + layered grant-source + `list`/`show`/`use` composing `set_mode` + grant-layer + additive `apply`, with detect-and-nudge on enforcement). It **defers** the sandbox-writing, prompt-free-scripting, and drift-healing parts to their substrate (ADR-004 §76 sandbox writer; #252 managed `apply`). The deferred parts are recorded-resolved here so the build cannot relitigate them into either footgun.

- **Issue #255 is resolved by ADR-004 (model) + this ADR (surface) + the minimal build.** ADR-004 was the model decision; this is the surface decision and a build sanction for its non-blocked half.
