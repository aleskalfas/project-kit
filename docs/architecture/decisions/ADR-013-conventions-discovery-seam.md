## Context

The `software-engineering` capability (Feature #337) ships the first *producer* agent in project-kit's review pipeline — `software-engineer`, which authors and edits code by conforming to the project's own coding conventions. The reviewer agents [COR-024](../../../.pkit/decisions/core/COR-024-critic-and-architect-agents.md) ships *check* work; this is the first agent that *produces* it, and producing to a project's opinionated standard requires the agent to read that project's conventions corpus.

That read raises a cross-cutting question the capability-scoped design cannot settle on its own: **how does an agent discover where a project's conventions live?** The conventions are adopter-owned prose that accretes over time — they are not the agent, not the capability, and not core. Three facts force the seam to be designed deliberately rather than as a one-off path the producer agent hardcodes:

- **The conventions must survive any single capability's uninstall.** Capability uninstall is `rm -rf` of the capability subtree ([COR-017](../../../.pkit/decisions/core/COR-017-capability-pattern.md)). A conventions corpus living at `.pkit/capabilities/software-engineering/project/conventions/` would be destroyed the moment that capability is removed — an adopter who uninstalls the producer capability would silently lose the conventions they had been accreting. That is a data-loss footgun, not an acceptable lifecycle.
- **The corpus will foreseeably have more than one consumer.** The producer agent reads it now; the reviewer stack ([COR-024](../../../.pkit/decisions/core/COR-024-critic-and-architect-agents.md)) is the obvious second consumer reading the same corpus to *check* conformance. *How an agent discovers conventions* is therefore a seam that outlives its first consumer.
- **The methodology already has a precedent for exactly this read.** The `architect` agent reaches adopter-specific documentation through overlay-resolved placeholder categories — `<architecture-docs>` and `<adr-records>` ([COR-024](../../../.pkit/decisions/core/COR-024-critic-and-architect-agents.md), built on [COR-013](../../../.pkit/decisions/core/COR-013-agent-architecture.md)'s adopter-overlay mechanism). The conventions read is the same shape: an agent reaching adopter-owned content the methodology cannot enumerate.

This ADR records the **cross-cutting contract** — the discovery seam itself, owned by no single capability. The capability-scoped application of that seam (the conventions-thin producer body, the producer/checker boundary, empty-tolerance, the adopter-setup gesture) is recorded in [software-engineering:DEC-001](../../../.pkit/capabilities/software-engineering/decisions/DEC-001-producer-agent-and-conventions-seam.md), which builds on this ADR. The producer-agent (#337) is authored against this seam, and this ADR pins the contract it stands on. As project-kit's own architecture record, concrete category names and overlay paths are in scope here (per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md)).

## Decision

**An agent discovers a project's conventions through an overlay-resolved `<project-conventions>` category — the same [COR-013](../../../.pkit/decisions/core/COR-013-agent-architecture.md) placeholder mechanism the `architect` uses for `<architecture-docs>`. The category is adopter-owned and resolves to wherever the adopter's overlay points; it is *not* a hard-coded path and *not* owned by any single capability. The consumer contract is the category name; what the category *resolves to* is the adopter's concern and may change over time without changing the consumer.**

1. **The discovery mechanism is the overlay placeholder, never a path.** A consuming agent declares `<project-conventions>` in its frontmatter `reads.patterns` and cites it in its body; the deploy-time overlay resolution ([COR-013](../../../.pkit/decisions/core/COR-013-agent-architecture.md) rule 5) substitutes the adopter's resolved value. The agent never names a filesystem path for conventions. This is the identical machinery `architect` rides for `<architecture-docs>` / `<adr-records>` ([COR-024](../../../.pkit/decisions/core/COR-024-critic-and-architect-agents.md)).

2. **The category is adopter-owned and capability-independent.** Conventions live wherever the adopter's `.pkit/agents/project/overlay.yaml` resolves `<project-conventions>` — adopter-owned content under the no-shared-files invariant. No capability owns the corpus; uninstalling any single capability (including the one that ships the first consumer) never touches it. The methodology does not seed a capability-specific category into the core overlay.

3. **The consumer contract is the category, not the source set.** A consumer agent reads "the project conventions" by reading what `<project-conventions>` resolves to. Today that is a single adopter-pointed location with one consumer; the contract is written so that when conventions later come from multiple sources, **only what the category resolves to changes** (single path → a union) — the consumer's declaration and read are unchanged.

4. **The aggregation mechanism is deferred, designed-for but not built ([COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)).** A consumer reading the *union* of conventions contributed across many installed capabilities — and the capability `depends_on` that would let one capability declare another's conventions as input — are deferred until a real second case appears. The seam is shaped now so the aggregator lands without changing the consumer contract; the aggregator itself is not part of this decision.

5. **Conventions content is out of scope.** What the corpus *says* (naming, modularity, DRY, comment discipline, …) is adopter-owned prose that accretes empirically. This ADR fixes only *how an agent discovers* the corpus, not what it contains.

## Rationale

**Why a named category and not a hard-coded path.** Two independent forces both point at the overlay category, and either alone would be sufficient. *No data-loss footgun:* a hard-coded `.pkit/capabilities/<X>/project/conventions/` couples the adopter's accreted conventions to one capability's lifetime, and [COR-017](../../../.pkit/decisions/core/COR-017-capability-pattern.md)'s `rm -rf` uninstall would destroy them — an adopter cannot be made to lose hand-authored conventions by removing a tool that merely *reads* them. The overlay-resolved category places the corpus wherever the adopter owns it, outside any capability subtree, so no uninstall reaches it. *Forward-compatibility:* expressing the read as a named category means the consumer contract is the *name*, not a source enumeration — so when conventions later arrive from multiple capabilities, the change is confined to *what the category resolves to* (single path → union), and no consumer agent is rewritten or re-deployed.

**Why reuse the COR-013 overlay mechanism rather than invent a conventions-specific lookup.** The mechanism already exists and is already proven on the `architect`'s `<architecture-docs>` / `<adr-records>` reads ([COR-024](../../../.pkit/decisions/core/COR-024-critic-and-architect-agents.md)). Reusing it costs no new machinery, gives adopters a single overlay vocabulary for "where my project-specific content lives," and keeps consistency across every agent that reaches adopter-owned content the methodology cannot enumerate. A bespoke conventions-discovery path would be a second mechanism for the same job — the divergence COR-013's placeholder design exists to prevent.

**Why the aggregator is deferred, not designed in.** There is exactly one consumer (the producer agent) and one source today. A union-across-capabilities reader and capability `depends_on` are real machinery with real questions (ordering, precedence, conflict between two capabilities' conventions) that cannot be answered well from a single case — answering them now would be speculative generality, the failure [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md) names. The discipline this ADR observes is the harder one: design the *seam* so the aggregator is a non-breaking later addition (the consumer contract is the category name, which already tolerates a one-to-many resolution), while building only the single-source case the first consumer needs. The aggregator is recorded here as a pinned future seam so its absence is a deliberate choice, not an oversight.

**Why an ADR and not a new COR.** This composes only existing principles — [COR-013](../../../.pkit/decisions/core/COR-013-agent-architecture.md)'s overlay-placeholder mechanism, [COR-024](../../../.pkit/decisions/core/COR-024-critic-and-architect-agents.md)'s `<architecture-docs>` precedent and reviewer-stack consumers, [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)'s defer-until-recurrence — into project-kit's architectural realization of a conventions-discovery seam. It introduces no new universal principle: "an agent reaches adopter-owned content through an overlay category" is already COR-013's principle; this ADR records *that an agent's conventions read is one such category, capability-independent by design*. It matches the ADR-002…012 hybrid.

### Alternatives considered

- **A hard-coded conventions path under the capability subtree** (`.pkit/capabilities/software-engineering/project/conventions/`). Rejected — couples the adopter's accreted conventions to one capability's lifetime; [COR-017](../../../.pkit/decisions/core/COR-017-capability-pattern.md)'s `rm -rf` uninstall destroys them. The data-loss footgun is disqualifying on its own.
- **A hard-coded path outside any capability** (a fixed repo-root `conventions/`, say). Rejected — survives uninstall but overfits the kit to one project's layout and breaks any adopter whose conventions live elsewhere; this is the exact overfit COR-024 rejected for `<architecture-docs>` when it chose the placeholder over a fixed `ARCHITECTURE.md`.
- **A conventions-specific discovery mechanism** (a dedicated lookup, registry, or config key distinct from the agent overlay). Rejected — a second mechanism for the job COR-013's placeholder already does; divergent vocabulary, more surface, no benefit over reusing the proven overlay category.
- **Build the cross-capability aggregator now.** Deferred ([COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)) — one consumer and one source do not justify union-resolution machinery and its ordering/precedence/conflict questions; the seam is designed so the aggregator lands later without changing the consumer contract.

## Implications

- **The seam is cross-cutting and capability-independent.** `<project-conventions>` is owned by no single capability; it is a category any agent (producer or reviewer) may consume. The first consumer ships in the `software-engineering` capability, but the category and its contract belong to the agent layer, not that capability.
- **The capability-scoped application is recorded separately.** [software-engineering:DEC-001](../../../.pkit/capabilities/software-engineering/decisions/DEC-001-producer-agent-and-conventions-seam.md) records the conventions-thin producer, the producer/checker boundary, and empty-tolerance — the capability's *use* of this seam. This ADR is the seam DEC-001 builds on; DEC-001 must not re-decide the discovery mechanism.
- **Adopter setup is an overlay edit, not a capability artifact.** The adopter defines `project-conventions` in `.pkit/agents/project/overlay.yaml` pointing at where they keep conventions (per [COR-013](../../../.pkit/decisions/core/COR-013-agent-architecture.md)). `pkit agents reconcile` surfaces the category when a consuming agent references it and the overlay does not yet define it. No capability seeds the category into the core overlay.
- **Empty / absent resolution is a normal state, not an error.** A `<project-conventions>` category that resolves to nothing (no overlay entry, or an empty corpus) is the expected early state; consumer-side handling of that (announce-and-proceed) is the consumer's contract, recorded for the producer in DEC-001.
- **The forward-compatibility property is recorded, not just coded.** "The consumer contract is the category name; multiplying the sources changes only the resolution" is pinned here so a later author building the aggregator extends the resolution rather than rewriting consumers — and cannot regress the seam into a per-consumer source enumeration.
- **Future seam noted.** Cross-capability convention *aggregation* (union-across-installed-capabilities resolution) and capability `depends_on` are the moves once a real second contributing source appears ([COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)) — pinned so the deferral is a recorded choice, not an oversight.
- **No COR-010 migration, no surface rename.** This ADR introduces no file rename / removal in a kit-owned tree, no `schema_version` bump, no CLI signature change — it records a contract the producer-agent design already realizes. Whether the producer-capability change-set bumps `.pkit/VERSION` per [PRJ-002](../../../.pkit/decisions/project/PRJ-002-version-bump-policy.md) follows from that change-set, not from this record.
- **Accepted** alongside the producer-capability change-set that realizes it (#337); `software-engineering:DEC-001` and the `software-engineer` agent build on the seam this record fixes.
