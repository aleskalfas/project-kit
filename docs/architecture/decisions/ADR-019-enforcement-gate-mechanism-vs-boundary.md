## Summary

A capability that wants to *enforce* a discipline — "this PR may not merge unless it satisfies rule X" — cannot, on its own, make merging impossible. The merge button lives in the adopter's forge (GitHub branch protection, required status checks), which the capability does not own and cannot assume exists ([COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md)). So a capability that runs its check inside its own PR-open flow and calls that "enforcement" is making a claim it cannot keep — the same overstated-fidelity failure [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) forbids for permissions, and [ADR-004](ADR-004-autonomy-intent-confinement.md) restated as "the merge layer is a speed-bump, not a boundary."

**Decision:** an enforcement gate decomposes into two parts that ship in two places, and the capability **declares the residual gap** between them rather than papering over it.

1. **Capability-owned mechanism** — a **read-only, verb-subject check script** (the `check-mesh.py` shape: surfaces findings, exits non-zero, *never* blocks a merge by itself) plus a **soft warning** emitted in the capability's PR-open / merge flow. This is what the capability can honestly ship to every adopter.
2. **Adopter-wired boundary** — the same script added by the adopter as a **required CI status check** with **branch protection** (the `scripts/check.sh` shape: project-kit's own CI/pre-push aggregator). This is the *only* place real enforcement lives, and it is the adopter's wiring, not the capability's.
3. **The residual gap is declared, not hidden.** The capability states plainly: "the PR-open warning is a speed-bump; real enforcement is your CI wiring of this script." It never claims the boundary it does not own.

Two consequences follow and are pinned here so the *next* capability that wants a gate does not relitigate them. First, the gate is **two-tier**: a soft warning at PR-open (what [project-management:DEC-015-doc-update-obligations] specifies today) plus a hard-reject in adopter-wired CI — mirroring [project-management:DEC-031-reject-unauthored-placeholder-bodies]'s established open=warn / hard-elsewhere house pattern. Second, the **override channel is the existing audit-trailed body section** (the `## Doc impact` section naming the code path and asserting no-doc-needed), at the `bypassable-with-audit` tier — never a bare `--skip` flag.

The motivating instance is the code→doc enforcement gate proposed in issue #117: an adopter wants to upgrade *their own already-opted-in* code-path→doc-path mapping from a warning to a hard reject. That specific upgrade sits **inside DEC-015's existing design space** and needs no DEC-015 amendment — see Context. This ADR records the general gate-decomposition principle the instance exposed; the detailed decision and the rejected alternatives follow.

## Context

The forcing question (issue #117, with a `critic`+`architect` review pass behind it) is whether the project-management capability can offer an *enforcing* code→doc gate — one that hard-rejects a PR which touches a mapped code path without touching the mapped doc path — and if so, where the enforcement actually lives and who owns it.

Two cases were on the table, and the review separated them sharply:

- **Case A (chosen) — opt-in, default-off.** An adopter who has *already* opted into a code-path→doc-path mapping (the optional mapping [project-management:DEC-015-doc-update-obligations] already defines) upgrades *that mapping's* severity from `warning` to hard-reject, for *their* project. This is **within DEC-015's existing design space**: DEC-015's severity table already tags the mapped-path rule `[validation-severity:warning]` with the explicit gloss **"(adopter may upgrade)"**, and the mapping itself is already optional and project-side. Case A exercises a latitude DEC-015 granted on day one. It is **not** a DEC-015 reversal and requires **no DEC-015 amendment**.
- **Case B (rejected) — mandatory / default-on.** The capability ships the code→doc hard-reject turned on for every adopter. This was rejected outright: it would **reverse** DEC-015 (whose whole rationale is that the mapping is optional because "not every adopter has stable enough docs to maintain mappings against") and would **fail [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md)** (it assumes every adopter has docs worth mapping and a CI surface to enforce against). Case B is named here only to record that it was considered and refused.

So the *severity-upgrade* question (Case A) was already answered by DEC-015. What issue #117 actually exposed — and what was *not* recorded anywhere — is the **architectural question underneath**: when a capability offers a gate that can be upgraded to "hard-reject," **where does the hard reject get enforced, and what may the capability honestly claim about it?**

The capability runs inside the PR-open / merge flow it ships (`open-pr` / `merge-pr`). That flow can *detect* the violation and *refuse to proceed with its own command*. But it cannot stop a human (or another tool) from merging via the GitHub UI, the API, or `gh pr merge` directly. The actual merge boundary is the forge's **branch protection + required status checks** — config the adopter owns, that the capability does not write, and that [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md) forbids the capability from assuming exists (an adopter may run on a forge with no branch protection, or no CI at all).

This is the same shape [ADR-004](ADR-004-autonomy-intent-confinement.md) already settled for confinement: pkit's hook can inspect a command but cannot confine a shell, so pkit **delegates** confinement to the OS sandbox and **declares** the residual gap rather than claiming an enforcement it cannot deliver. The "merge-pr layer is a speed-bump, not a boundary" framing is the direct analogue: the capability's PR-open flow is the speed-bump; the adopter's CI wiring is the boundary. [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md)'s honesty-about-gaps discipline — declare which layer enforces each dimension, report the residual gap, never overstate fidelity — is the governing obligation.

Two worked artifacts in this very repo already embody the two halves and serve as precedents:

- **`check-mesh.py`** (project-management capability) — a read-only, verb-subject diagnostic that "surfaces drift as warnings; never blocks." It is exactly the *mechanism* shape: a check a capability can ship honestly, because it makes no claim to enforce.
- **`scripts/check.sh`** (project-kit's own root tooling) — "the single source of truth for what must pass before this lands," run by both the local pre-push hook and CI. It is exactly the *boundary* shape: an adopter-wired aggregator that *is* the gate, owned by the project, not the capability.

The decomposition below is the generalisation of that already-existing split. As project-kit's own architecture-decision record, forge/CI/harness specifics are in scope here, unlike the harness-neutral COR. `status: proposed` is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md); accept before any implementation cites it.

## Decision

**An enforcement gate offered by a capability decomposes into a capability-owned *mechanism* and an adopter-wired *boundary*; the capability ships only the mechanism, declares the residual gap, and never claims the boundary it does not own. The decomposition is the contract; a capability that wants a gate conforms to it rather than re-deriving where enforcement lives.**

**1. The capability ships the mechanism, not the boundary.** The capability's deliverable is two things and only two:

- a **read-only, verb-subject check script** (the [`check-mesh.py`](../../../.pkit/capabilities/project-management/scripts/check-mesh.py) shape) — for the code→doc instance, a `check-doc-mapping` verb-subject script that reads the adopter's mapping config, inspects the PR diff, and **exits non-zero on a violation but blocks nothing by itself**;
- a **soft warning** emitted in the capability's own PR-open / merge flow (`open-pr` / `merge-pr`) when the script reports a violation.

The capability MUST NOT implement the hard reject *inside* `merge-pr` and present that as the enforcement boundary. Refusing to run its own command is a speed-bump; it is not a merge boundary, and calling it one is the overstatement [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) forbids.

**2. The adopter wires the boundary.** Real enforcement is the adopter adding the capability's check script as a **required CI status check** behind **branch protection** (the [`scripts/check.sh`](../../../scripts/check.sh) shape). This is the only place a merge is actually blocked. It is the adopter's wiring of config the adopter owns; the capability documents *how* to wire it but does not — and per [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md) cannot — write or assume it.

**3. The capability declares the residual gap.** Wherever the gate is surfaced, the capability states plainly: *the PR-open warning is a speed-bump; the real boundary is your CI wiring of this script; without that wiring there is no enforcement.* This is [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md)'s honesty-about-gaps obligation applied to the gate — the capability names which layer enforces (adopter CI) and reports the residual gap (an unwired adopter has detection only), exactly as [ADR-004](ADR-004-autonomy-intent-confinement.md) keeps `network-egress: enforcement: none` standing as the honest residual truth.

**4. The gate is two-tier.** Soft warning at PR-open (what [project-management:DEC-015-doc-update-obligations] specifies today) + hard-reject in the adopter-wired CI tier. This mirrors [project-management:DEC-031-reject-unauthored-placeholder-bodies]'s established house pattern — *warn at the open/create surface, hard-block at the surface where the harm actually lives* (for DEC-031 the first lifecycle transition; here the CI merge boundary). A capability gate adopts the same two-tier shape rather than inventing a new severity placement.

**5. Default-off, opt-in upgrade — within DEC-015, no amendment.** The hard-reject tier is **off by default** and is reached only when an adopter who has *already* opted into the optional code→doc mapping upgrades *their* mapping's severity from `warning` to hard-reject. This is the "(adopter may upgrade)" latitude [project-management:DEC-015-doc-update-obligations]'s severity table already grants; it requires **no DEC-015 amendment and is not a DEC-015 reversal.** The mandatory / default-on variant (Case B) is rejected — it would reverse DEC-015 and fail [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md).

**6. The override channel is audit-trailed, not a bare flag.** When an author legitimately needs to land a code change without the mapped doc change, the override is the **existing `## Doc impact` section** — a line naming the code path and asserting no-doc-needed, at the **`bypassable-with-audit`** tier. It is **not** a bare `--skip` flag. An override that leaves an audit trail in the artifact the reviewer reads preserves the gate's intent; a silent flag defeats it.

## Rationale

**Why decompose at all — why not just "enforce in the capability."** Because the capability literally cannot. The merge boundary is forge config the capability does not own and [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md) forbids it from assuming. A capability that runs its check in `merge-pr` and stops there has built a speed-bump that any direct merge (`gh pr merge`, the UI, the API) walks straight past. Naming that "enforcement" is precisely the overstated-fidelity failure [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) forbids and [ADR-004](ADR-004-autonomy-intent-confinement.md) restated for the merge layer. The honest decomposition — mechanism here, boundary there, gap declared — is the only stance that does not lie about where the gate holds.

**Why a read-only verb-subject script is the mechanism.** [`check-mesh.py`](../../../.pkit/capabilities/project-management/scripts/check-mesh.py) is the proof the shape works: a check that surfaces findings and exits non-zero, blocking nothing on its own, is *portable* — it runs identically in the PR-open flow (as a warning source) and in adopter CI (as a status check). A check that bakes in "and now block the merge" is not portable: it can only block the surface it is wired into, and it tempts the false enforcement claim. Keeping the script pure-diagnostic is what lets the *same* artifact serve both tiers, which is what makes the two-tier model cheap.

**Why the adopter — not the capability — wires CI.** [`scripts/check.sh`](../../../scripts/check.sh) is project-kit's own answer: the project owns its CI/pre-push aggregator and decides what must pass before a merge. Branch protection and required-status-check config are inherently project-and-forge-specific; the capability cannot write them without assuming a forge, a default branch, a CI provider, and a permission to mutate repo settings — every one of which [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md) forbids assuming. Documenting the wiring (so the adopter can do it in minutes) is the capability's job; doing the wiring is the adopter's.

**Why two-tier mirrors DEC-031 rather than inventing a placement.** [project-management:DEC-031-reject-unauthored-placeholder-bodies] already settled the house answer to "where does a soft signal become a hard block": warn at the open/create surface (visible, never silent), hard-block at the surface where the harm lives. For an unauthored body that surface is the first lifecycle transition; for a code→doc gate it is the CI merge boundary. Reusing that pattern keeps the methodology's severity placement coherent across capabilities — an author who has internalised DEC-031 already understands this gate's shape.

**Why the override is the audit-trailed body section, not a flag.** [project-management:DEC-015-doc-update-obligations]'s `## Doc impact` section already exists as the place an author records doc reasoning, including the Shape-B "no doc impact: <reason>" assertion the validator hard-rejects when empty. Routing the override through that section means a bypass is *recorded where the reviewer reads it* and forces a one-line justification — the `bypassable-with-audit` tier. A bare `--skip` flag bypasses silently, leaves no trace in the artifact, and turns the gate into a suggestion. The audit-trailed channel preserves the gate's intent (deliberate, visible consideration) while still letting a legitimate exception through.

**Why generalise now rather than solve only the code→doc instance.** [COR-007](../../../.pkit/decisions/core/COR-007-pattern-extraction.md)'s recurrence test: the mechanism-vs-boundary split is *already* instantiated twice in this repo (`check-mesh.py` as mechanism, `scripts/check.sh` as boundary), and the code→doc gate is a third consumer of the same shape. Three instances of one pattern is the point at which the abstraction has earned its keep. Pinning the principle as an ADR means the next capability that wants a gate — a lint gate, a coverage gate, a license-header gate — conforms to the decomposition instead of re-deriving (probably wrongly, as a `merge-pr` speed-bump claiming to be a boundary) where enforcement lives.

### Alternatives considered

- **Enforce in `merge-pr.py` and call it a boundary.** Rejected — the capability's PR-open / merge flow can refuse to run *its own* command, but cannot stop a direct merge via the UI, the API, or `gh pr merge`. Presenting that speed-bump as the merge boundary is the overstated-fidelity failure [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) forbids and [ADR-004](ADR-004-autonomy-intent-confinement.md) named for the merge layer. The flow is the speed-bump; the boundary is the adopter's branch protection.
- **Ship a CI step (a workflow / Action) from the capability itself.** Rejected — it assumes a forge, a CI provider, a default-branch name, and permission to write repo settings / workflow files, every one of which [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md) forbids a universal capability from assuming. The capability ships the *check script* (portable, forge-neutral); the adopter wires it into whatever CI they run, exactly as [`scripts/check.sh`](../../../scripts/check.sh) wires checks for project-kit itself.
- **Case B — make the code→doc hard-reject mandatory / default-on for every adopter.** Rejected — it reverses [project-management:DEC-015-doc-update-obligations] (whose rationale is that the mapping is optional because not every adopter has docs stable enough to map against) and fails [COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md) (it assumes universal docs and a universal CI surface). Default-off opt-in (Case A) is the only variant that stays inside DEC-015.
- **A bare `--skip` flag as the override channel.** Rejected — it bypasses the gate silently, leaving no trace in the artifact the reviewer reads. The audit-trailed `## Doc impact` line at the `bypassable-with-audit` tier preserves deliberate, visible consideration; a silent flag turns the gate into a suggestion.
- **Bake the block into the check script (a non-read-only check).** Rejected — a script that blocks rather than reports is not portable across the two tiers; it can only block the surface it is wired into and re-imports the false-boundary claim. Keeping the script pure-diagnostic ([`check-mesh.py`](../../../.pkit/capabilities/project-management/scripts/check-mesh.py)'s shape) is what lets one artifact serve both the PR-open warning and the CI status check.
- **Amend DEC-015 to add the enforcement tier.** Rejected as unnecessary — DEC-015's severity table already grants "(adopter may upgrade)" on the mapped-path rule and already makes the mapping optional and project-side. Case A exercises that existing latitude; the new content is the *architectural* decomposition (mechanism vs boundary, the residual-gap declaration), which is ADR-shaped, not a change to the capability discipline DEC-015 owns.

## Implications

- **The code→doc gate is buildable within DEC-015, no amendment.** The capability gains a read-only `check-doc-mapping` verb-subject script and a soft warning in `open-pr` / `merge-pr`; the hard-reject tier is the adopter wiring that script into required-status-check + branch-protection. The severity upgrade is the "(adopter may upgrade)" latitude [project-management:DEC-015-doc-update-obligations] already grants — no DEC-015 change.
- **The capability declares the residual gap on every surface.** The PR-open warning, the script's own output, and the capability docs each carry the honesty gloss: *this warning is a speed-bump; the real boundary is your CI wiring of this script; unwired, you have detection only.* This is [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md)'s discipline applied to gates, at the bar [ADR-004](ADR-004-autonomy-intent-confinement.md) set for the egress residual.
- **A general contract for the next gate.** Any future capability gate (lint, coverage, license-header, schema-drift) conforms to this decomposition: ship a read-only verb-subject mechanism + soft warning, document the adopter CI wiring as the boundary, declare the residual gap, place severity two-tier per [project-management:DEC-031-reject-unauthored-placeholder-bodies], and route overrides through an audit-trailed channel rather than a bare flag. The pattern does not get re-litigated per gate.
- **The override is the `## Doc impact` section at `bypassable-with-audit`.** No new flag, no new section. An author lands a code-only change by recording the no-doc-needed assertion in the existing section the reviewer already reads.
- **No migration.** No file/directory rename or removal in a kit-owned tree, no `schema_version` bump, no breaking CLI signature change, no capability-subtree restructure — `pkit migrations check-diff --include-working-tree --base main` is expected clean. This change authors the ADR file only.
- **No version bump in this change-set.** Like [ADR-017](ADR-017-confinement-allowance-cannot-override-harness-deny.md) and [ADR-018](ADR-018-capability-data-egress-axis.md), this authors the ADR file at `status: proposed`; the acceptance-gate flip and any version bump are a separate lifecycle gesture. The eventual `check-doc-mapping` script + warning is a capability surface change that bumps at *its* implementation time, not here.
- **Supersedes nothing.** This ADR sits beside [ADR-004](ADR-004-autonomy-intent-confinement.md)'s mechanism-vs-boundary / declare-the-gap stance and generalises it from permissions to enforcement gates; it changes no prior decision and leaves [project-management:DEC-015-doc-update-obligations] intact.
