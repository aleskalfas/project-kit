## Context

The forcing question (issue #24, under EPIC #18) was whether a capability could declare its *own* network egress — "the project-management capability needs to reach `api.github.com`, nothing else" — scoped to the commands that capability ships, so that installing a capability stands up exactly the egress it needs and no more. The intuitive shape is **command-declared, command-scoped** egress: each command carries the hosts it may reach, and the box enforces that command's egress against that command's declaration.

A spike against the live Claude Code harness established that this intuitive shape **is not expressible**, for two independent reasons rooted in already-accepted records:

1. **The sandbox network is session-wide and agent-blind.** [ADR-004](ADR-004-autonomy-intent-confinement.md) rule 2 and rule 5 pin confinement — filesystem *and* network reach — as a **session-wide** axis: the OS sandbox is configured once per session via the top-level `sandbox` key, subagents share the parent session's configuration, and there is no per-agent or per-command network differentiation the harness can honour. The sandbox cannot ask "which command is making this connection"; it applies one network posture to the whole session. So a per-command egress declaration has nowhere to land — the box has no command-granularity hook to enforce it against.

2. **The allowlist is not a security boundary.** [ADR-004](ADR-004-autonomy-intent-confinement.md) §61 and §71 record that the sandbox's network filtering is a **client-supplied-hostname allowlist with no TLS inspection** — it bounds well-behaved egress but does not stop IP-literal / DNS-rebind / domain-fronting exfil. `permission-enforcement.yaml` keeps `network-egress: enforcement: none` standing as the honest residual gap precisely because the allowlist is not an exfil control. Any egress surface pkit builds therefore cannot claim to *confine* egress as a security boundary; it can only declare which well-behaved hosts a session may reach.

What *is* expressible is a **session-wide host-allowlist** (`sandbox.network.allowedHosts`) that a capability declares as part of its confinement needs. This is the realization issue #24 actually asks for, once the per-command framing is corrected against the harness substrate. It must be described everywhere — `sandbox status`, `permissions overview`, the toolkit read surface, the schema docs — as **session-wide egress, and not a security boundary** (no TLS inspection; [ADR-004](ADR-004-autonomy-intent-confinement.md) §61). This is the [COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) honesty-about-gaps obligation applied to egress, at the same bar [ADR-014](ADR-014-macos-sandbox-platform-stance.md) set for the macOS exclusion: the gloss is mandatory and verbatim, never softened into "restricts egress" or "confines network."

This ADR resolves the six design forks the spike surfaced. The design is settled — spike findings plus operator sign-off on all six — and recorded `accepted` per the [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md) acceptance gate. As project-kit's own architecture-decision record, harness and platform specifics are in scope here, unlike the harness-neutral COR.

## Decision

**A capability declares its network egress as a session-wide host-allowlist through a confinement-toolkit `allow-host` allowance, routed through the existing single-writer + provenance machinery. Egress is never command-scoped, is always surfaced with the "session-wide; not a security boundary" gloss, and an unbounded `any` host is unambiguously widening.** The six forks resolve as follows.

**Fork — what is built (the realization).** Not command-scoped egress (not expressible — Context 1/2). The realization is a **session-wide host-allowlist**: a capability's declared hosts merge into `sandbox.network.allowedHosts`, applied once per session. There is no per-command and no per-agent egress differentiation; the surface never implies one.

**Fork 3 — carrier: a confinement-toolkit fragment with a new `allow-host` allowance kind.** Egress declarations are **not** a bespoke manifest field. A capability ships a **confinement-toolkit** fragment (the [ADR-008](ADR-008-confinement-allowances.md) rule 1 data carrier) carrying a new allowance `kind: allow-host`, whose values flow to `sandbox.network.allowedHosts`. This routes egress through the **single sandbox-block writer with provenance** ([ADR-008](ADR-008-confinement-allowances.md) rule 2) — the one place that knows the merge rule, the one place that can later remove only pkit-authored entries no longer claimed by an enabled toolkit. A bespoke manifest field would create a **second write path** into the `sandbox` block, bypassing the single writer and re-opening the ADR-002 §52 silent-deletion footgun ADR-008 rule 2 closes.

**Fork 1 — classification: narrowing-but-reported (a named third posture).** `allow-host` for **named, bounded hosts** (e.g. `api.github.com`) is **auto-applied on capability install** like other narrowing toolkits (it makes the box usable for the capability without carving the agent's reach out of the box), **but is always surfaced** by `sandbox status` / `permissions overview` with the "session-wide egress to X; not a security boundary" gloss. Because ADR-008's narrowing class does not *mandate* loud reporting (narrowing is the safe-to-auto-apply class), this posture is named explicitly: **ADR-008 is amended in place** to add **narrowing-but-reported** as a third posture beside narrowing and widening — auto-applied like narrowing, mandatorily-reported unlike ordinary narrowing. (See the amendment in ADR-008's body.)

**Fork 2 — realization mechanism: `allowedHosts` primary; whole-binary `exclude` only as the explicit fallback.** The primary mechanism is `sandbox.network.allowedHosts` (named hosts, session-wide). Where a tool cannot work through a host allowlist at all, the fallback is the whole-binary `excludedCommands` exclusion — but that is the [ADR-008](ADR-008-confinement-allowances.md) rule 4 **widening** path (loud, never-auto, never-committed, always-reported), reached explicitly, never as a silent substitute for `allowedHosts`.

**Fork 4 — trigger: install-state.** The allowance applies **when the owning capability is installed**. Egress stands up as a consequence of having the capability present (install-state), not gated on a separate per-run or detection step.

**Fork 6 — `any` is unambiguously widening.** `network: any` (→ `allowedHosts: ["*"]`) opens egress to every host. It is **always the loud, never-auto widening gesture** of [ADR-008](ADR-008-confinement-allowances.md) rule 4 — regardless of the narrowing-but-reported handling of named hosts. Named-and-bounded auto-applies-but-reports; `*` never auto-applies. This is pinned explicitly so a capability cannot obtain silent open egress by declaring `any`.

## Rationale

**Why session-wide and not command-scoped.** The per-command framing is the natural way to *ask* the question but it has no enforcement substrate: ADR-004 rules 2/5 make the sandbox network session-wide and agent-blind, so a per-command declaration has nothing to bind to. Building a command-scoped *surface* over a session-wide *enforcer* would be the dishonesty COR-028 and ADR-004 §49 forbid — the operator would believe egress is bounded per command when the box applies one posture to everything. The honest model is the one the harness can keep: session-wide, declared by the capability, reported as session-wide.

**Why the "not a security boundary" gloss is mandatory and verbatim.** The allowlist filters on client-supplied hostname without TLS inspection (ADR-004 §61/§71); it bounds well-behaved tools but does not stop a determined exfil. Softening the gloss to "restricts egress" would let the operator believe a boundary holds that does not — the exact failure ADR-004 rule 4 and ADR-008 rule 4 exist to prevent, and the COR-028 honesty obligation ADR-014 already met verbatim for the macOS exclusion. Egress gets the same bar.

**Why a confinement-toolkit fragment, not a manifest field (fork 3).** ADR-008 rule 2's entire safety property is *one* sandbox-block writer with provenance: that is what lets `disable` remove only pkit-authored entries and never an operator's hand-added host. A bespoke `manifest.network` field would mutate `allowedHosts` from outside that writer — a second write path with no provenance, which cannot be cleanly torn down and re-imports the ADR-002 §52 silent-deletion footgun. Routing egress through the existing `allow-host` allowance keeps it on the single-writer machinery that already solved this; the only new thing is the allowance *kind*, not a new write path.

**Why narrowing-but-reported rather than plain narrowing or plain widening (fork 1).** This is the load-bearing tension. Named egress is *narrowing-like*: it makes the box usable for the capability without carving a command out of the box, so forcing it onto the loud, never-auto widening path (like `excludedCommands`) would defeat the "no manual editing" win for the common, safe case — a capability that needs `api.github.com` should get it on install, not as a per-operator gesture. But it is *not* ordinary narrowing either: an `allowWrite` to `~/.cache/uv` enlarges nothing an operator needs to see, whereas an egress allowance — even named — touches a dimension pkit has *declared it does not confine* (`network-egress: none`), so leaving it silent would let the operator believe egress is unmanaged when in fact a host was opened. The honest resolution is the third posture: auto-apply it (it is safe to auto-apply, like narrowing) **and** always report it (it touches the declared residual gap, so silence would mislead). Burying it as an undocumented sub-case of narrowing was the alternative; naming it in ADR-008 is chosen so every command and the `detect` path enforce the mandatory-report mechanically, not by ad-hoc judgement — the same reason ADR-008 rule 1 encodes the narrowing/widening split in the data.

**Why `any` stays widening regardless (fork 6).** `allowedHosts: ["*"]` is open egress — it lowers the boundary to everything, which is definitionally the widening class. Letting it ride the narrowing-but-reported auto-apply path would let a capability obtain silent, unbounded egress merely by declaring `any`, the precise §49 trap. Named-and-bounded is auto-apply-but-report; unbounded is never-auto. The distinction is bounded-vs-unbounded, encoded in the data so it is mechanical.

### Alternatives considered

- **Command-scoped egress (the issue's literal framing).** Rejected on the spike — ADR-004 rules 2/5 make the sandbox network session-wide and agent-blind; a per-command declaration has no enforcement substrate, and a per-command *surface* over a session-wide enforcer would overstate fidelity (COR-028, ADR-004 §49).
- **A bespoke `manifest.network` field on the capability manifest.** Rejected — a second write path into the `sandbox` block, bypassing ADR-008 rule 2's single writer + provenance, which cannot be cleanly torn down and re-opens the silent-deletion footgun. The `allow-host` allowance routes egress through the existing single writer.
- **Classify named `allow-host` as plain narrowing (auto-apply, no mandatory report).** Rejected — egress touches the dimension pkit declares it does not confine (`network-egress: none`); silent application would let the operator believe egress is unmanaged. Hence narrowing-*but-reported*.
- **Classify all `allow-host` (named and `any`) as widening (always loud, never auto).** Rejected for *named* hosts — it forfeits the "no manual editing" win for the common safe case (a capability needing one named host on install). Retained for `any`, which is genuinely boundary-lowering.
- **A standalone `network-egress` schema separate from confinement-toolkits.** Rejected — egress is a confinement allowance like any other (it bounds reach); a separate schema would split the single-writer/provenance machinery and the narrowing/widening/reported classification across two data carriers. One carrier, one writer.

## Implications

- **ADR-008 is amended in place (third posture named), not superseded.** Its binary narrowing/widening split gains **narrowing-but-reported** as a documented third posture: auto-applied like narrowing, mandatorily-reported unlike ordinary narrowing, carrying the egress honesty gloss. The boundary-effect organising principle stands; the amendment qualifies the split to cover the egress case. See the amendment in ADR-008's Implications.
- **Child of ADR-004 and ADR-008; no decision superseded.** This resolves issue #24's egress question within ADR-004's session-wide confinement model and ADR-008's allowance surface. It adds no new ownership tier and no new write path; the `allow-host` kind is a new allowance value on the existing `confinement-toolkit` schema, routed through the existing single sandbox-block writer.
- **The `allow-host` kind, the reporting, and the `any` widening gate are implementation work (same issue #24), after this ADR in the same change.** Adding `kind: allow-host` to the `confinement-toolkit` schema (and its JSON Schema companion); flowing its named values to `sandbox.network.allowedHosts` through the provenance-tagged single writer; the mandatory "session-wide egress to X; not a security boundary" gloss on `sandbox status` / `permissions overview` / `toolkit show`; and the `network: any` → `allowedHosts: ["*"]` widening gate (loud, never-auto, riding ADR-008 rule 4) all land in the implementation step of this same issue. This ADR records only the decision.
- **The honesty gloss is verbatim, everywhere egress is surfaced.** "Session-wide; not a security boundary" (no TLS inspection; ADR-004 §61) is the COR-028-required residual-gap declaration for egress, at the ADR-014 bar. It is mandatory and not to be softened on `sandbox status`, `permissions overview`, the toolkit read surface, or the schema docs.
- **`network-egress: none` still stands.** This surface declares well-behaved hosts a session may reach; it does not relabel `network-egress` as enforced. The residual gap (IP-literal / DNS-rebind / domain-fronting exfil) remains the adopter's to close outside the harness, exactly as ADR-004 §61/§71 records.
- **The whole-binary exclusion fallback rides the widening path.** Where a tool cannot work through `allowedHosts`, the `excludedCommands` fallback is the ADR-008 rule 4 widening gesture — loud, never-auto, never-committed, always-reported — reached explicitly, never silently substituted for an allowlist.
- **No version bump.** This is a decision record; the observable behaviour (the `allow-host` allowance, the reporting, the `any` gate) ships with the implementation step of this same issue, so `.pkit/VERSION` is unchanged per PRJ-002.
