## Context

[COR-028](../../../.pkit/decisions/core/COR-028-permission-model-realization.md) establishes permissions as a harness-neutral model the methodology owns, realized into each harness's native enforcement by that harness's adapter. The design that produced it was settled across ten rounds in the `2026-05-29-permission-meta-model.md` scratchpad note (Q1–Q10). COR-028 is harness-neutral by construction and deliberately defers the concrete realization to adapter documentation. This ADR records the *architecturally-significant boundary change* that the first realizer — the claude-code adapter — forces, named in COR-028's own implications as ADR-worthy.

The problem is one of file ownership. The claude-code realizer enforces the model through four layers, two of which write adopter-owned files: agent frontmatter (`.claude/agents/<name>.md`), a PreToolUse hook, and `.claude/settings.json`. The settings file is the sharp case. It is adopter-owned: the adopter hand-edits it, the harness reads exactly that one file, and the no-shared-files invariant (every file has exactly one owner) has held it as a whole-file project-owned artifact reached only through COR-002's append-only merge. To let the model become the truth — so that `apply` can *heal* drift, removing realized rules the model no longer declares — the realizer must own and regenerate a *portion* of that file while leaving the rest under the adopter's hand. COR-002's whole-file ownership model has no vocabulary for "one owner for part of a file, another owner for the rest."

Two distinct ownership scopes result, and conflating them would mis-place the boundary:

- The **PreToolUse hook file** is *always* realizer-owned, in every mode. It reads the model live at decision time, so it carries no stale state and there is no adopter content to preserve within it — it is an ordinary propagated, pkit-owned artifact.
- The **`permissions` region of `settings.json`** is owned by the realizer *only in managed mode*, and shared at finer-than-file granularity even then: the region is realizer-owned, the rest of the file stays adopter-owned.

The proposed status is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md); COR-028 (the principle) is already accepted, and this ADR records how its first realizer lands against project-kit's own ownership invariants. Because this is project-kit's own architecture-decision record, harness specifics are in scope here — unlike the harness-neutral COR.

## Decision

The claude-code realizer partitions ownership *within* `.claude/settings.json` rather than at the whole-file boundary, and splits the two ownership scopes cleanly.

**Intra-file ownership partition.** In **managed** mode the realizer owns an *authoritative region* of `settings.json` — the `permissions` block (the full session-wide allow set plus guardrail denies; see the **#252 clarification** in Implications for exactly what composes it) — and regenerates it **wholesale** on each `apply`: the prior region content is replaced, not unioned. Everything outside the region remains adopter-owned and is never touched. This is exactly one owner per region — the realizer for its region, the adopter for the rest — which preserves the no-shared-files invariant at finer-than-file granularity. Because the region is replaced wholesale, healing drift is the natural behaviour (rules no longer in the model simply do not reappear) and an empty model projection regenerates an empty region on the next run, with no stale-content reconciliation and no in-file markers. "Vanish" here is precise: while the project is in managed mode the region is *unconditionally regenerated from the model's current realized composition every run* (what that composition is — narrow projection vs. the full additive-union-excluding-the-live-target — is the **#252 clarification** in Implications, which rules for the latter), so a region whose composition emits nothing heals to empty — the realizer never recovers or subtracts "what it previously owned," it always rewrites a region computed from scratch. This is distinct from turning managed mode *off* (managed→additive), which is a separate mode change governed by additive's "only adds" contract, not by this tier; and distinct from uninstalling the realizer entirely, an uninstall-path concern. The strip-free property is a property of *regenerate-from-scratch within managed state*, not of source removal.

In **additive** mode (the default) the realizer only *adds* to `settings.json` through COR-002's existing append/union merge — guardrail denies plus the hook registration — and never removes or replaces. No region is owned; the file stays whole-file adopter-owned exactly as today.

**The hook/region boundary.** The PreToolUse hook file is realizer-owned in *both* modes — it is the live authority that reads the model at decision time, so it is always current and always pkit-managed, independent of how much of `settings.json` the realizer touches. The mode gates only the `settings.json` `permissions` region. The hook is registered through the top-level `hooks` key, which merges normally and is not part of the owned region. This separation is load-bearing: the always-owned hook lets the default (`additive`) mode rely on a live authority without owning any region of the adopter's settings, so adopting the model is non-destructive while still enforcing the nuanced per-agent and per-directory rules.

**Fail-open / double-lock.** The hook fails *open* — on error, timeout, or malformed payload the call proceeds. Therefore non-negotiable denies (`sudo`, `rm -rf`, `git push --force`) are realized on **both** layers: the fail-open hook *and* the fail-closed native `settings.json` deny. The nuanced, evolving rules ride the hook (where fail-open is tolerable); the catastrophic denies are double-locked, so a hook bug, timeout, or version skew can never bypass them. The native deny stands on its own regardless of the hook's health.

**Deny-wins composition.** Where an allow and a deny conflict — across the baseline-for-all and per-agent grants, or across layers — deny prevails. This is the model's composition rule (per COR-028) and it realizes cleanly because it matches Claude Code's own precedence: a hook `deny` overrides a settings `allow`, and a hook `allow` cannot override a settings `deny`. The double-lock above is the same rule read from the failure direction.

## Rationale

The partition is the only ownership stance that lets the model heal drift in `settings.json` without violating no-shared-files. The whole-file alternatives all fail:

- **Markers in the file** to delimit the realizer's content. Rejected for the same reason COR-002 rejected markers for fixed-path content: the no-shared-files invariant forbids two owners sharing a file mediated by in-file delimiters. Wholesale region replacement achieves the same "realizer owns this content" outcome without markers — the region is defined by its key (`permissions`), not by sentinels the adopter could disturb.
- **A separate realizer-owned file pulled in via include.** Rejected because the harness reads exactly one `settings.json`; there is no include mechanism to split a generated region into a sibling file the adopter also owns. (This is the same constraint COR-002 cites for why merge exists at all.)
- **Full-file generation** (the realizer owns all of `settings.json`). Rejected because it destroys adopter sovereignty over a file the adopter legitimately hand-edits — every adopter customisation outside permissions would be lost on `apply`. The partition confines realizer ownership to exactly the region the model is authoritative over.

**Why additive is the default and managed is opt-in.** Additive rides COR-002's existing append/union merge and owns no region, so introducing the realizer into an installed adopter removes nothing and changes nothing the adopter relied on — adoption is non-destructive, and the always-owned hook still provides the live authority. Managed mode is the stronger promise ("I never hand-edit settings; the model is the whole truth") and the one that requires the new ownership tier; it is therefore opt-in, chosen per project. Defaulting to managed would silently take ownership of a region of a file the adopter has been hand-editing — exactly the sovereignty loss the partition exists to bound.

## Implications

- **COR-002 authoritative-region tier is a hard dependency of managed mode.** The wholesale-replace region behaviour above is the *authoritative-region* delivery tier added to COR-002 by COR-028's refinement; this ADR's managed mode is its first concrete consumer. The additive default rides COR-002's pre-existing append behaviour unchanged and depends on no new tier.
- **DEC-030's strip-logic dissolves under managed mode.** [project-management:DEC-030](../../../.pkit/capabilities/project-management/decisions/DEC-030-capability-contributed-adapter-overlays.md) carried explicit `disable` strip-logic to remove a capability overlay's contributed keys, because the append-only merge could not distinguish a stale source contribution from a manual adopter edit. For the `permissions` region under a managed realizer that problem does not arise: while in managed mode the region is regenerated wholesale from the current model projection every run, so a projection that emits nothing heals the region to empty on the next `apply` — no strip-logic, no per-source attribution, no markers. This is strip-free precisely because the realizer never reads-back-and-subtracts a previously-owned region; it rewrites a region computed from scratch each run. (It is therefore *not* structurally identical to DEC-030's `disable` strip-logic, which operates on survivor top-level keys the merge primitive carries forward — `permissions` is not a survivor key in the merge, it is recomputed from scratch every run, so no key-subtraction step exists or is needed. Per the **#252 clarification** below, the managed composition recomputes the region as the additive union of all merge sources *excluding the live target* — dropping the live target from the inputs is what heals drift, and it remains strip-free in the sense above: still no read-back-and-subtract of a previously-owned region, still no per-source key attribution. The mechanism is "regenerate this whole region from the projection," at the granularity of the single `permissions` top-level key.) Stopping ownership of the region — managed→additive — is a separate mode change, not this heal; uninstalling the realizer is an uninstall-path concern where the same lingering-key reality DEC-030 documents applies. DEC-030 already defers its reserved-`permissions`-key rule to COR-028 as the sanctioned owner of that region; the additive default contributes nothing to `permissions` beyond the baseline denies and skill grants DEC-030 already permits, so that rule stands unchanged for the default and for any adopter not running a managed realizer.
- **COR-010 migration is additive and a no-op by default.** Introducing the realizer ships a migration per COR-010 that deploys and registers the hook (top-level `hooks` key) and sets the project to `additive` mode, leaving existing `settings.json` content untouched. With an empty model the hook abstains on unmodeled commands in lenient mode, deferring to the harness's normal flow — a behavioural no-op on day one. The migration is idempotent (re-running detects the hook and mode already present and exits clean). It does **not** auto-import existing grants into the model, which would launder possibly-over-broad live grants as declared intent; `import` remains an explicit, reviewed bootstrap path.

  > **Update (#252) — what the managed `permissions` region contains; the narrow-vs-subsume fork; deferred.** Planning managed-mode `apply` (issue #252) surfaced a contradiction §26 did not visibly grapple with. Wholesale replacement means the region file *is* the entire `.permissions` — so the question "what composes it?" has a load-bearing answer, and two candidates pulled in opposite directions:
  >
  > - **Fork 1 — narrow region.** The region is *only the model's projection*: `projection.project().settings.allow` (the handful of session-wide `Bash(<cmd>:*)` / tool allows the model declares) plus the guardrail denies (`_core_settings_denies`). This reads §26's parenthetical ("coarse session-wide allows plus guardrail denies") literally, leaves the #251 merge mechanism untouched, needs no new strip-logic, and does **not** trip the escalation flag. Its cost is **fatal at the current model fidelity**: the projection is *structurally incapable* of reproducing the native baseline. There is no `Skill` privilege and no `Write`/`Edit`/`jq`/`ssh`/`mv`/`cp`/… privilege in the catalog at all, and `collect_skill_grants` (the deploy-time skill-trust signal) lives only in `merge-settings.sh`, not in `project()`. So Fork 1 does not merely "drop deploy-time skill trust" — flipping an adopter to managed mode would wholesale-delete the working-developer baseline (`gh`/`git`/`Read`/`Write`/`Skill(...)`/…) from native settings, leaving the fail-**open** hook as the sole live authority for everything the model happens not to project. That directly contradicts the deploy-is-the-trust-signal principle the additive path depends on, and silently revokes skill trust the adopter never asked to lose.
  > - **Fork 2 — subsume.** The region must *union in* the native baseline allows and the computed `Skill(...)` grants before replacing, so managed mode loses nothing additive provided. But skill grants and the baseline union are computed in `merge-settings.sh`, not in `project()`; making the region "union all sources **except** the live target, then replace" is a **change to the #251 region mechanism** and touches the merge primitive — it trips this ADR's architect escalation, and the critic's reading is that "union-except-target" is structurally strip-logic's cousin (it must reason about what each source contributed), in tension with §49's "regenerate-from-scratch, no read-back-and-subtract" claim.
  >
  > **Ruling (architect, escalated per #252's own flag + the foundational-decision / merge-primitive triggers):**
  >
  > 1. **Ruling E is not overridden by §26; it forces Fork 2 *when the surface is built*.** §26's "coarse session-wide allows" was written before the projection's actual fidelity was known. It described an *aspiration* (the region is the model's truth), not a measured claim that the projection reproduces the baseline — which it cannot. Reading §26 to sanction Fork 1's silent baseline-deletion would elevate a parenthetical over the load-bearing don't-silently-revoke-deploy-time-trust principle. The narrow region is therefore **rejected as the managed posture** at current fidelity. The correct managed region is the *full* session-wide allow set the additive path produces (baseline ∪ skill-grants ∪ capability-overlay allows ∪ projected model allows) plus the guardrail denies — i.e. Fork 2's composition.
  > 2. **This is a mechanism change, and it is reconciled with §49 rather than contradicting it.** Composing the region as "union all merge sources *except* the live target, then assign wholesale" is a genuine change to the #251 mechanism (the gate flips the *source* of the recomputed region from "additive union including the live target" to "additive union excluding the live target"). It is **not** DEC-030-style strip-logic: there is still no read-back-and-subtract of a previously-owned region and no per-source key attribution — the region is recomputed from scratch every run; the only difference from additive is that the **live target is dropped from the union inputs**, which is what makes drift heal (a grant the adopter hand-added to live settings is not an input, so it does not survive). §49's strip-free property holds; the §49 DEC-030 bullet is updated in place to name "exclude-the-live-target-from-the-union" as the managed composition, distinct from both additive (includes the live target) and DEC-030's key-subtraction.
  > 3. **Build deferred; recorded as a resolved decision, not an open question.** Per #252's own deferral (critic Y-2) and ADR-004's confirmation that the deferred `pkit permissions profile` surface — managed mode's only concrete near-term consumer — is itself blocked and demand-driven: **#252 stays deferred.** Neither near-term adopter (`example-brownfield`, `example-greenfield`) needs wholesale ownership; additive (#250) is the shipped non-destructive default; the `apply` managed guard in `permissions.py` correctly refuses managed mode with a "not yet implemented (#252)" error. The fork is **resolved on paper** (Fork 2 is the sanctioned composition) so that whoever builds it cannot relitigate it into Fork 1's silent-deletion footgun, but no code lands until a concrete adopter needs managed mode. When it is built it MUST: (a) compose the region per Ruling 1 (Fork 2); (b) implement the mechanism change per Ruling 2 in `merge-settings.sh` (or its COR-004 successor merge runtime) — i.e. the managed branch unions baseline ∪ skill-grants ∪ overlay-allows ∪ `project()` allows, *excluding the live target*, rather than swapping in the bare projection; (c) re-run `pkit migrations check-diff` — the merge-primitive change is a surface change and may itself require a migration; (d) ship the heal/idempotency tests #252 names.
  > 4. **No ADR boundary change.** The ownership *partition* (one owner per region; managed owns `.permissions`, adopter owns the rest) is unchanged. This update fixes only *what the owned region contains* and *how it is composed* — a refinement within the existing boundary, not a new tier.
  >
  > **Update (#247) — hook registration is opt-in; no migration deploys or registers the hook.** The implemented design (issue #247, "Option B") makes live enforcement *opt-in* rather than on-by-default, mirroring the [DEC-030](../../../.pkit/capabilities/project-management/decisions/DEC-030-capability-contributed-adapter-overlays.md) default-agent toggle precedent. The hook *script* is a propagated adapter file whose lifecycle `pkit sync` owns; the PreToolUse hook is **registered only when the adopter runs `pkit permissions enable`** (and stripped by `pkit permissions disable`), never auto-registered. Consequently the "ships a migration per COR-010 that deploys and registers the hook (top-level `hooks` key)" claim above is **superseded**: introducing the realizer into an installed adopter is a pure addition with no day-one state change — nothing is auto-registered and nothing in `settings.json` changes until the adopter opts in. `pkit migrations check-diff` reports COVERED with **no migration script**, so COR-010 mandates none for this arc. The rest of this bullet stands unchanged: `additive` remains the default mode, the idempotency reasoning carries over to the `enable`/`disable` gestures, and the no-auto-import-of-grants discipline is unaffected.
- **The concrete realization lives in the claude-code adapter docs, authored with the implementation.** Which model dimension each of the four layers carries (tool on/off → frontmatter; coarse session-wide allow + guardrail deny + domain rules → native settings; per-agent / per-directory / conditional → hook; unenforceable intent → out-of-harness report), the routing rule that picks the most-native faithful layer, the hook's abstain contract in lenient versus strict posture, and the per-adapter enforcement-capability declaration are adapter-documentation concerns, not this ADR's. This ADR fixes only the ownership boundary; the realization detail follows the implementation.
