## Context

[ADR-008](ADR-008-confinement-allowances.md) gave pkit confinement *accommodations* (narrowing sandbox allowances) and *exclusions* (widening), with `accommodate --detect` auto-applying narrowing it finds in the repo and recording the choice in a committed `confinement_accommodations` list (rule 3). `setup autonomy` ([ADR-007](ADR-007-setup-command-class.md)) auto-accommodates uv/node when it detects lockfiles. But the tools that actually generate per-session permission friction — the **SSH agent socket** (`git push`) and the **commit-signing socket** (`op-ssh-sign`/gpg-agent on `git commit`) — are **not repo facts**. They are host-environment facts: a socket path in `$SSH_AUTH_SOCK`, a path that exists on this machine. ADR-008's `detect` vocabulary is repo-relative by construction, so `setup autonomy` cannot serve them, and the operator was left hand-authoring a project toolkit and running `accommodate` manually — not "one command" autonomy.

A `critic` + `architect` consult on a first proposal (extend the toolkit `detect` to machine signals and auto-apply) surfaced four findings this record is built to honour: (1) the shipped `ssh-agent` toolkit's socket value (`~/.ssh/agent-socket`) was **wrong** — the real socket is `$SSH_AUTH_SOCK`, a per-session path on macOS — so a static toolkit value structurally cannot serve the generic agent; (2) auto-applying a *machine* fact drops the property that made ADR-008 rule 3 safe — the signal being repo-local and committed — so it is a genuinely **new concern** (pkit reading host state), not a clean carry-over, and needs its own decision; (3) auto-seeding a *committed* `confinement_accommodations` from one machine's state pollutes other checkouts and, because the macOS socket path varies per session, accretes dead entries — breaking `setup autonomy`'s idempotency contract; (4) shipping a vendor toolkit (`onepassword`) is inventory that fails universal applicability ([COR-014](../../../.pkit/decisions/core/COR-014-universal-applicability.md)).

The product owner's direction is decisive: the operator should **not** have to run the accommodation by hand for the common case — `setup autonomy` should *run* it, not merely nudge. This record makes that safe rather than rejecting it: it admits host-environment detection under constraints that neutralise each finding above. Proposed status is the acceptance-gate gesture per [PRJ-005](../../../.pkit/decisions/project/PRJ-005-adopt-adrs.md).

## Decision

**`setup autonomy` (and a manual `accommodate --socket` lever) may resolve a bounded, vendor-neutral set of host-environment signals to the *real* socket/path and auto-apply it as a narrowing accommodation — written to per-machine settings only, never to committed config, recomputed-and-replaced each run, and honest about liveness. Host-environment detection is a second, distinct detection-source axis from ADR-008's repo-local one, and the two route to different destinations.**

1. **The detection-source axis.** ADR-008 detection is *repo-local* (a lockfile in the tree): deterministic, reviewable, identical for every checkout → it may seed the **committed** `confinement_accommodations`. Host-environment detection (this record) reads *machine* state (`$SSH_AUTH_SOCK`, a socket path's existence): non-deterministic across machines, invisible in the repo → it routes **only** to per-machine live settings, never to committed config. Same narrowing safety (no reach increase); different provenance, different destination.

2. **Real-path resolution, never a static guess.** Host detection resolves the *actual* value (`$SSH_AUTH_SOCK`'s current path) and allows exactly that. It never writes a guessed path. This is correctness-by-construction and is the fix for the broken static `ssh-agent` toolkit value: the system supplies the path the operator otherwise types.

3. **Per-machine, never committed.** A host-detected (or `--socket`) accommodation writes the socket/path into the live per-machine sandbox block (`.claude/settings.json`) under the synthetic `_manual` provenance — the narrowing twin of how `sandbox exclude` already handles per-machine widening. It is **never** recorded in the committed `confinement_accommodations`, because the signal that justified it does not exist on another machine. *(Correction — [ADR-032](ADR-032-per-machine-activation-routing-axis.md), post-[ADR-029](ADR-029-move-mutable-sandbox-state-out-of-tracked-settings.md): the file reference is stale. Post-ADR-029 `.claude/settings.json` is the **committed floor** and the per-machine file is the gitignored `.claude/settings.local.json`; the host socket resolves into `settings.local.json`, never the committed file. ADR-032 realizes this rule's never-committed guarantee **by construction** via a value-provenance routing axis — ADR-029's effect-only routing had regressed it by sending the narrowing socket committed. The guarantee is unchanged; only the destination filename is corrected.)*

4. **Recompute-and-replace each run, keyed by logical source + provenance.** A host-detected accommodation is keyed by its source (e.g. `ssh-agent`), and each `setup autonomy` run *re-resolves and replaces* that source's prior value rather than unioning. Replacement is scoped by the ADR-008 rule 2 **provenance tag** (pkit-authored, this-source) — it removes only the entry *pkit itself* wrote for that source, never an operator's hand-added socket that happens to share the path (the ADR-002 §52 silent-deletion footgun stays closed). So a per-session-varying socket path leaves no stale entries — `setup autonomy` stays idempotent and "the live system is the checkpoint" (ADR-007 rule 3) holds against varying host facts.

5. **Liveness-honest.** Detection that finds a signal but cannot confirm a working endpoint (an env var set but the socket dead/unconnectable) **does not claim "applied"** — it reports the honest state and, where it cannot safely auto-resolve, falls back to a nudge naming the exact `accommodate --socket` command. Never assert a boundary works when it doesn't (COR-028 / ADR-004 §49, the usability inversion).

6. **Vendor-neutral signals only; narrowing only; no schema change.** The built-in host-detection set stays universal — protocol/standard env-var signals (`$SSH_AUTH_SOCK`), never a named commercial product. Vendor-specific or long-tail sockets (a separate 1Password signing socket, gpg-agent, a corporate helper) are served by the manual **`accommodate --socket <path> [--name <id>]`** lever and/or an adopter's *project* toolkit — pkit ships no vendor toolkit (COR-014 / principles-not-inventory). Host detection auto-applies **narrowing only**; widening (`gh`) stays the explicit gesture (ADR-008 rule 4, unchanged). The universal detection is a small built-in in `setup`, **not** a per-toolkit schema field, so the `confinement-toolkit` schema is unchanged — no `schema_version` bump, no project-data migration.

7. **Credential-floor collision → never silently auto-apply.** A detected socket whose resolved path falls *under* the credential `denyRead` floor (`SANDBOX_CREDENTIAL_DENY_READ` — `~/.ssh`, `~/.aws`, …; some agents put their socket there) creates a same-block contradiction: a deny-read directory plus an allow-socket inside it. Resolving it silently is the trap either way — if deny-read wins, setup would falsely report "applied" for a socket the box still blocks (the ADR-004 §49 believed-but-absent boundary); if the socket-allow wins, it punches a hole through the credential floor the operator believes is solid (a silent floor reduction — widening through the narrowing door). So `setup autonomy` **must not auto-apply a socket inside the floor**: it reports the collision honestly and falls back to the nudge (rule 5), naming the explicit `accommodate --socket` gesture so the operator decides the tradeoff. Auto-apply proceeds only for resolved paths *outside* the floor. Liveness (rule 5) is necessary but not sufficient here — a socket can be live *and* inside the floor — so the floor check is a distinct gate.

## Rationale

**Why auto-run, not nudge-only.** Autonomy's promise is "one command." For the universal case the operator holds no fact the system cannot read itself — `$SSH_AUTH_SOCK` is right there in the environment. A nudge that asks the operator to run a command the system could run is exactly the friction this arc exists to remove. The consult's reflex toward nudge-only was the safe default *absent constraints*; rules 1–6 supply the constraints that make auto-run safe, so the safe default moves to auto-run for the universal narrowing case.

**Why the per-machine/never-committed split dissolves the architect's objection.** The architect's escalation was that auto-applying host facts into committed state pollutes other checkouts and rides ADR-008's repo-local safety argument illegitimately. Routing host-detected accommodations to per-machine live settings only (rule 3) means no host fact ever enters a shared file — the objection's object simply doesn't exist. The committed path stays exclusively for repo-local signals, preserving ADR-008's rule-3 second leg intact (and pinning it, which ADR-008 left implicit).

**Why recompute-replace dissolves the idempotency objection.** The critic showed a per-session-varying socket path would accrete dead `allowUnixSockets` entries under a union-write. Keying by logical source and replacing (rule 4) makes each run reconcile to the *current* host state — idempotent by reconciliation, not by union. This is the same "live system is the checkpoint" stance `setup autonomy` already takes.

**Why no vendor toolkit and no schema change.** Shipping `onepassword` fails COR-014 (a Linux/gpg-agent/`secretive`/corporate-helper adopter carries dead data) and crosses ADR-008's own principles-not-inventory line. A universal env-var detector plus the general `--socket` lever covers the same ground without naming a product, and keeps the shipped set neutral. Building the universal detection as a `setup` built-in rather than a toolkit-schema `detect` extension avoids reshaping a typed field that adopters have already overridden in project files — sidestepping a project-data migration entirely.

### Alternatives considered

- **Nudge-only (the consult's default).** Rejected per product-owner direction — it leaves the universal case a manual command, defeating the one-command goal. Retained only as the *fallback* for cases that can't be safely auto-resolved (rule 5).
- **Extend the toolkit `detect` to machine signals + auto-seed committed `confinement_accommodations`.** Rejected — cross-machine pollution + per-session accretion (idempotency break) + a `schema_version` bump forcing a project-data migration. The per-machine/non-committed built-in (rules 3–4, 6) delivers the value without any of these.
- **Ship an `onepassword` (or other vendor) toolkit.** Rejected — COR-014 / inventory creep. Vendor sockets → project toolkit or `--socket`.
- **Static `ssh-agent` toolkit value.** Rejected — the agent socket path is dynamic; a static value is wrong by construction (the shipped one was). Real-path resolution (rule 2) replaces it.

## Implications

- **Child of ADR-008 + a reconciliation of its rule-3 safety argument (no supersession).** Adds the host-environment detection-source axis alongside ADR-008's repo-local one; changes neither the narrowing/widening split nor ADR-008's committed-accommodation path, and replaces no rule. But it does *make explicit a premise ADR-008 left latent*: ADR-008 rule 3 stated auto-apply is safe "because narrowing-only," resting the argument on the boundary-effect split alone — while its detection was repo-local-and-committed by construction (`--detect` scans lockfiles). ADR-010 extends auto-apply to host-environment sources, which the narrowing-only argument **does not by itself license**: host-env auto-apply is safe *only because* of the per-machine/never-committed routing (rule 3) + the floor-collision gate (rule 7). Recorded here so a future reader does not read ADR-008 rule 3 as already sanctioning host-env→committed auto-apply (it does not). ADR-007's `setup autonomy` contract is unchanged — it gains a host-resolution step whose effects are per-machine and idempotent (rules 3–4).
- **New `accommodate --socket <path> [--name <id>]` CLI** — a one-off narrowing socket/path allowance, per-machine, `_manual` provenance, never committed; the manual lever for the long tail and the writer `setup` reuses for host-detected accommodations.
- **`setup autonomy` gains universal host detection** — resolves `$SSH_AUTH_SOCK` (live-checked), auto-applies it, recompute-replace; nudges with the exact `--socket` command where it can't safely resolve. Auto-applies narrowing only.
- **Fix the broken shipped `ssh-agent` toolkit** — its static `~/.ssh/agent-socket` value is wrong (no such standard path; the real socket is `$SSH_AUTH_SOCK`). The fix is to **drop the shipped `ssh-agent` toolkit entirely** (host resolution replaces it), not neuter it — removing a shipped illustrative entry is an addition-class content edit (no adopter has a reason to override a broken entry; confirm via `pkit migrations check-diff`), and a static value can never serve a dynamic socket. SSH-agent reach is henceforth the host-resolved accommodation (`$SSH_AUTH_SOCK`) + the `--socket` lever.
- **Bounded, documented host-probing.** `setup`/`accommodate --socket` reads only the named universal signals (`$SSH_AUTH_SOCK`) and a socket's existence/liveness — documented so the operator knows what the command inspects. No process listing, no broad env scan.
- **No schema change / no migration; surface change → version bump (PRJ-002).** The `confinement-toolkit` schema is untouched. New CLI verb + `setup` behaviour is a surface change → bump. `pkit migrations check-diff` confirms no trigger.
- **Build sanction on acceptance.** Accepting sanctions building: `accommodate --socket`, the `setup autonomy` host-resolution step (provenance-scoped recompute-replace, liveness-honest, floor-collision gate, nudge fallback), and dropping the broken `ssh-agent` toolkit — with tests, README rows, and a version bump.
