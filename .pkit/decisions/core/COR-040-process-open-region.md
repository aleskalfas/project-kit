A process can now declare a state as an **open region**: a phase with no pre-wired order of work inside it, where progress accumulates in any order — looping and feeding back — until an explicit **exit gate** lets the subject leave. It is the substrate's escape hatch for genuinely open-ended work that a fixed sequence would misdescribe. The engine stays a deterministic validator throughout: inside the region it reduces to one definite question, "may this subject leave yet?". This fulfils the open region COR-033 named as a deferred extension point (P6) and un-defers COR-035's per-state (`applies_to`) invariant scoping — both additively, leaving COR-035's report-only default untouched.

## Context

COR-033 (P5/P6) named an *open region* as a deferred extension point — a state "bounded only by invariants + an exit gate", the escape hatch for genuinely open-ended work whose steps have no fixed order. COR-035 shipped invariants report-only and process-wide, and **explicitly deferred** both per-state (`applies_to`) scoping and any boundary-enforcing posture, stating they would "land together with the open-region slot when a binding demands it" and that the additive path was forward-compatible.

A binding now demands it. The trip-planning adopter's `build` phase (its DEC-014) is genuinely open-ended: discovery, route, logistics, and safety accumulate in **no fixed order and loop** — discovery reshapes the route, the route triggers more discovery, a booking change bounces back to the itinerary. A static edge graph misrepresents this, and a linear chain forces a false order that breaks for route-first work. This is the un-defer trigger (COR-033 P5 / COR-007's recurrence test); this record ships the slot, minimally.

## Decision

**A process may declare a state as an open region: a single state with no internal edges, carrying region-scoped invariants and exactly one guarded outgoing transition — its exit gate.** Inside the region the engine reduces to a deterministic boundary check ("may this subject leave yet?").

- **An open region is one state, not a container of sub-states.** It has no internal edges because it has no internal states to wire. Progress inside it is the *subject's reality changing* — data accumulating via the binding's commands — not the engine moving between positions. Work that is itself a staged, looping process (its own lifecycle plus a readiness outcome the region must read) is modelled as a **separate composed process** (COR-036) whose terminal outcome or cascade fold (COR-037) the exit gate reads — it is not an internal edge of the region. So the substrate gains **no new node kind**: an open region is a normal state distinguished only by (a) region-scoped invariants and (b) an exit gate as its sole outgoing transition.
- **The exit gate is an ordinary checkable gate (COR-033 P4).** Leaving the region is a move, so it is a gate's job, not an invariant's: a deterministic predicate the engine evaluates, and/or a recorded cross-authority authorisation the engine confirms exists (produced by a different authority than the acting agent). A judgment exit — "a person declares the work ready" — leaves a cross-authority trace the engine *confirms*, never re-judges.
- **Invariants may be scoped to a region (`applies_to`), additively.** A COR-035 invariant holds process-wide; an invariant may now name the state it applies to. **Absent scope means process-wide** (COR-035 unchanged). A region-scoped invariant is surfaced on the status view while the subject sits in that region, so status can explain *why the exit gate is shut*.
- **Boundary-enforcement is gate composition, not a new engine coupling.** An open region that "must hold its invariants to be left" expresses that as its **exit-gate predicate referencing those conditions** — exactly COR-035's own guidance that a hard always-gate is expressed as a gate predicate. The engine still never blocks a move *because of an invariant*; the exit gate is a predicate, and a predicate may check any condition. So COR-035's report-only decision is preserved verbatim — invariants report; the exit gate (a gate) checks — and no per-invariant "enforce" flag is introduced.
- **Fail-closed at the moment of mutation is the binding's responsibility, not the engine's.** The engine is a validator, not a write-path interceptor; it cannot block an arbitrary edit. A binding that needs a hard invariant to fail-closed the instant it would be violated enforces that in its own commands (COR-035's shipped model: the violation surfaces on status and the binding's command declines downstream). The region's exit gate is the *structural* backstop — the subject cannot leave while the condition is unmet — not a replacement for the command-level guard.
- **A region-scoped invariant does not reach into a composed inner process.** Invariants are single-subject (COR-032); an invariant scoped to a region governs the region's *own* subject, not the separate subject of a process composed inside it — the inner process declares its own invariants.
- **Liveness is the binding's problem; determinism is the engine's guarantee.** The engine gives a definite answer to "where is this subject, may it leave" at every check; it does not guarantee an open region ever reaches its exit. An open region whose gate never opens is a stuck *binding*, not a non-terminating engine computation — and it is **visible**: status shows the subject parked in the region with its exit gate shut and the unmet conditions named.

**Ship narrow.** This un-defers exactly the open-region state type + `applies_to` scoping — the pair COR-035 said would land together — and nothing more. Invariant severity / auto-remediation and cross-subject always-checks stay deferred. The slot ships with its demanding binding (trip-planning `build`) so it is never inert.

## Rationale

- **Why one state, not a sub-state container.** The open-endedness *is* the absence of a pre-wired order among work items, which a single state whose internal progress is data accumulation and whose only structured move is its exit captures fully. A sub-state-and-edge sublanguage would add machinery the demand does not need and would reintroduce the very ordering the open region exists to avoid; where a work item genuinely is a staged process, composition (COR-036) already models it as a separate process the region reads an *outcome* from.
- **Why the exit is a gate, not a special invariant.** The substrate draws a clean line — invariants answer "is this always true?", gates answer "may this move happen?". Leaving the region is a move. Expressing the exit as a gate keeps that line intact and reuses P4's checkability unchanged; a special "exitable" invariant would blur the two and duplicate the gate-checkability rule.
- **Why boundary-enforcement is gate composition, not move-blocking invariants.** COR-035 deliberately kept invariants report-only and rejected coupling them to move-blocking. Delivering the region's enforcement through the exit-gate *predicate* honours that: the engine gains no "an invariant blocks a move" behaviour; a gate predicate simply checks conditions that may also be reported as invariants. This is the smallest change that makes the region enforceable, and it makes this record *fulfil* COR-035's deferral additively rather than revise it.
- **Why mutation enforcement stays with the binding.** The engine never sees an arbitrary edit; only the binding's commands do. Putting fail-closed-at-mutation in the engine would require intercepting every write, contradicting its validator role. The binding's command reading the surfaced violation and declining is the honest home for it; the exit gate is the structural backstop.
- **Why liveness is out of scope.** The engine's guarantee is a definite answer over current reality, not a promise of progress; a termination bound is a different tool with no demanding binding. Surfacing non-progress on status is enough for an operator to see a stuck region.

### Alternatives considered

- **Model the region as a container of sub-states with free movement.** Rejected — adds a node/edge sublanguage the demand doesn't need and reintroduces intra-region ordering; composition already handles genuinely-staged inner work as a separate process.
- **Express the exit as a special "region is exitable" invariant.** Rejected — leaving is a move, a gate's job; overloading invariants with a movement role blurs the invariant/gate line and duplicates gate-checkability.
- **Couple region-scoped invariants to move-blocking in the engine.** Rejected — that is the move-blocking coupling COR-035 explicitly deferred; the same enforcement is achieved additively by the exit-gate predicate referencing the conditions, leaving the engine's report-only invariant behaviour untouched.
- **Ship `applies_to` scoping or boundary-enforcement on their own.** Rejected — neither has a standalone consumer (COR-035 already refused to ship scoping alone); their only consumer is the open region, so they land together with it and only it, exactly as COR-035 scheduled.
- **Put fail-closed-at-mutation in the engine.** Rejected — the engine is a validator, not a write-path interceptor; mutation-time enforcement belongs in the binding's commands.
- **Give the substrate a termination / bounded-iteration guarantee for open regions.** Rejected — no binding needs it; determinism holds without it and non-progress is already visible on status.

## Implications

- **Shape contract.** A state may be marked an open region; an invariant gains an optional `applies_to` scope (absent = process-wide). Both additive — existing processes declare neither and validate byte-unchanged — so no migration is owed (COR-010). The exact field layout lives in the process-area shape reference, not here.
- **Engine.** Inside an open region, move-legality reduces to the exit gate (predicate and/or cross-authority artifact), and the status view surfaces region-scoped invariant violations as the explanation for a shut exit gate. Position stays inferred; the validator stays deterministic; no cross-subject read. Standing this up is an acceptance-gated downstream task; the engine strategy (how scoped invariants are evaluated against the resolved position, and the failure semantics of a shut exit) is recorded in a sibling engine-strategy ADR at implementation time (ADR custody per COR-025), per the substrate's per-slot ADR precedent.
- **Acceptance sign-off bar.** This record ships under a **single-maintainer** acceptance sign-off: it is additive (report-only preserved, no migration) and introduces no cross-subject read, so it does not need the heavier acceptance bar COR-037 set for a slot that crossed the never-enumerate line (COR-032). The choice is recorded here deliberately.
- **Grounding binding.** The slot lands with the trip-planning `build` region (its DEC-014, proposed): `build`'s evidence-backed / commands-only invariants scope to the region and surface on status, and its "ready-to-realize" exit gate is a deterministic predicate over each track's recorded readiness plus a cross-authority sign-off. The core slot and the binding accept in sequence (the binding cites this record).
- **One open item left to implementation** (not decided here): whether a single exit gate can express a compound "(predicate) AND (cross-authority artifact)" directly, or whether the adopter folds the artifact's existence into the predicate while preserving the engine's authorship check. The engine task settles it.
- **Docs.** On acceptance, the process-area reference flips its open-region and `applies_to` / boundary-enforcement rows from deferred to core (mirroring how COR-035's acceptance flipped the invariants / validate rows).
- **Relationship to records.** Fulfils the open region named in COR-033 (P5/P6) and the per-state-scoping + boundary-enforcement deferrals COR-035 scheduled to "land together" with it; reuses COR-033-P4 gate-checkability and COR-036 / COR-037 composition and cascade for genuinely-staged inner work; applies name-broad / ship-narrow (COR-016) and extract-on-recurrence (COR-007). **Supersedes nothing** — un-deferring named extension points is fulfilment, not revision (mirrors COR-032, COR-034, COR-035), and COR-035's report-only default posture is preserved verbatim.
