title: Stamp each issue/PR with the methodology version it was filed and last touched under

When a bug surfaces in an issue or PR workflow, the first question is *which version of the methodology was in force when this was filed* — did the bug predate an upgrade or arrive with it. This record decides that the capability records that version provenance directly on the issue, in two complementary places: a **one-time, immutable filing comment** posted at creation (the load-bearing record that answers the before/after-upgrade question), and a **self-replacing footer** at the foot of the body that always reflects the version of the last script-mediated touch. Both are written only by the capability's scripts, never typed by an agent or a human. This record fixes *what* gets recorded and *why*; the structural contract that makes the footer tamper-proof — how every body write strips and reissues it — is pinned in a companion ADR (authored next).

## Context

An adopter who works across several projects pinned to different methodology versions cannot, today, tell from an issue whether a bug it describes predates or postdates a given upgrade. The creation timestamp is on the GitHub object, but the version that was active then is not — and it cannot be reconstructed reliably after the fact. A GitHub issue is not a git object: issues are filed and edited with no commit nearby, from uncommitted working trees, from other branches or clones, or from the web UI. So correlating an issue's timestamp against the version file's git history is a lossy heuristic, not ground truth. The only moment at which the version in force for an operation can be *observed* is the moment the operation runs. To have the fact later, the capability must record it then.

The capability already routes every issue/PR mutation through its own scripts — the executable-command surface per [project-management:DEC-020-methodology-as-executable-commands] — so there is a script boundary at which to capture it. The body already carries one methodology-managed region — the `Integration:` marker defined in the body-structure rules per [project-management:DEC-010-issue-body-minimum-structure] — so a managed region in the body is established precedent. And the capability already has a failure posture for non-essential side effects — the report-and-continue stance of the lifecycle-hook engine per [project-management:DEC-024-lifecycle-hooks] — that a best-effort provenance write should inherit.

## Decision

1. **Record provenance in two places with two distinct jobs.** Provenance is written by two classes of script, by class not by roster: the **creating scripts** — those that bring a new issue or PR into being — post a **single immutable comment** recording the filed-under versions and the calendar date, never edited afterward, so it preserves the birth version even after later edits; and the **body-writing scripts** maintain a **footer** at the end of the body carrying the versions of the *current* touch. Which concrete commands populate each class is owned by the command surface per [project-management:DEC-020-methodology-as-executable-commands], not pinned here. The comment answers "what version was this filed under?" (the stated goal); the footer answers "what version is on it now?" (at-a-glance, and the drift signal when current and filed-under differ). The comment is the load-bearing record; the footer is a convenience that the comment can always backstop.

2. **The footer carries versions only; the date lives in the comment.** A methodology version changes on upgrade, not on every edit. By keeping the date out of the footer, re-writing the footer on any body write is idempotent — the same versions in, the same footer out — so a version-irrelevant edit (ticking a checkbox) never churns it. The single dated record is the immutable filing comment.

3. **Two version axes, named for their two purposes.** Provenance records the **backbone version** and the **capability version** — these answer the stated goal, *which methodology version governed this issue*, and either an engine upgrade or a capability upgrade can be the one that introduced a bug. It additionally records the **installed-CLI version** under a distinct label as *actor-tooling provenance* — which tool wrote this — a different question from which methodology version governed the issue. Naming the two purposes is the boundary that keeps the recorded set from growing without discipline: a candidate version earns a place only by answering one of these two questions, not by being available to read.

4. **Both records are written only by scripts, never by an agent.** The provenance is injected at the script boundary, so neither an agent nor a human composing a body authors or alters it. The footer in particular is a script-owned region living inside an otherwise agent-editable document; keeping it correct under agent edits is a structural problem deferred to the companion ADR (the strip-and-reissue write seam, the read-side strip, and the guard that proves no script writes a body around the seam). This record commits to the *property* — agents never own the provenance bytes — and leaves the mechanism to that ADR.

5. **Best-effort, report-and-continue.** A failed provenance write never blocks the underlying operation. If the filing comment cannot be posted, issue creation still succeeds; the issue simply joins the set of stamp-less issues. This inherits the report-and-continue posture of the lifecycle-hook engine per [project-management:DEC-024-lifecycle-hooks].

6. **Stamp-less issues are an accepted gap.** Issues created before the capability adopted this convention, and issues created by hand in the GitHub web UI, carry neither comment nor footer. There is no back-fill obligation. Triage of such an issue falls back to the lossy timestamp-versus-history heuristic; the gap is accepted rather than papered over, because the alternative — a guaranteed stamp on every issue regardless of origin — is unattainable for objects the capability's scripts did not create.

## Rationale

**Why a written-in record over a read-time triage command.** The discarded alternative was a read-only command reconstructing the filed-under version from the creation timestamp and the version file's git history. It was rejected because that correlation is unsound: an issue's events are not git events (see Context), so the command would confidently report the version committed near a date, which need not be the version actually in force when the issue was touched. Capturing at write time records ground truth the heuristic cannot recover. The command is not forbidden — it remains a sensible *reader* of stamps and a fallback for the stamp-less gap — but it is deferred until cross-issue triage ("which open issues were filed before version X") actually recurs, rather than built speculatively for a single-issue question the comment already answers.

**Why two records rather than one.** A single self-replacing footer would lose the birth version on the first edit under a newer version — exactly the fact the goal needs. A single immutable comment would not surface the current version at a glance, nor the drift between filed-under and current. The two carry different, both-wanted facts: the comment is immutable history, the footer is mutable present. The cost of the footer is bounded because it carries no date (item 2), so maintaining it is idempotent.

**Why the installed-CLI version is labelled, not merged.** The drift the adopter also wants to see — the installed CLI diverging from the synced project tree — is a tooling fact, not a methodology-governance fact. Recording it unlabelled alongside the methodology versions would invite the next reach (adapter version, harness version) with no stated discriminator. Pinning the two purposes (item 3) gives the discriminator: methodology-governance versions answer the filing goal; actor-tooling versions answer "which tool wrote this." Anything that answers neither does not belong.

**Why best-effort rather than blocking.** Provenance is an audit convenience, not a correctness invariant of the issue. Blocking issue creation on a failed comment write would trade a real capability (filing the issue) for a nice-to-have (the stamp). Item 6 already concedes stamp-less issues exist; item 5 simply makes a transient write failure one more way into that accepted set rather than a hard stop.

## Implications

- **A companion ADR** — [ADR-036](../../../../docs/architecture/decisions/ADR-036-provenance-write-path-contract.md) — pins the structural write contract this record depends on: the strip-then-reissue footer seam, the read-side strip so an agent never sees the footer bytes, and the guard that proves no body write bypasses the seam. It joins the capability's existing seam-family ADRs as their provenance-side member. This DEC stays at the what/why level and must not be treated as the structural spec.
- **A new `provenance_marker` entry** in the body-structure schema carries the validation exemption: the footer region is methodology-managed, stripped before the body validators and checkbox extractors scan, and constrained in shape so an un-stripped footer cannot collide with a scanned construct. It is authored via the schema skill once this record is accepted, mirroring the existing `Integration:` marker exemption in the body-structure rules per [project-management:DEC-010-issue-body-minimum-structure].
- **The two script classes from item 1** take on the new behaviour: the creating scripts gain the one-time filing comment, the body-writing scripts gain the footer via the seam. The reusable comment-idempotency marker the lifecycle-hook engine already ships is reused rather than reinvented.
- **A surface change.** A new convention adopters observe on every script-filed issue is a surface change, so it bumps the capability version. The schema addition is additive (no `schema_version` bump, no migration); retrofitting footers onto existing issues, were it ever wanted, is a back-fill data concern, not a schema migration.
- **No new hard-reject.** Provenance never gates a transition or a close. Its failure mode is absence (items 5 and 6), surfaced as the stamp-less gap, not a validation finding — except a defensive warning when a provenance region appears anywhere but its sanctioned trailing position.
