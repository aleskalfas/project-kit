## Context

The project-management capability assumes it can **create its own substrate**: `bootstrap` creates the `type:*` / `priority:*` / `workstream:*` / `state:*` labels (and optionally a Projects v2 board), and `pre-check` **hard-refuses** when they are absent. That is a *greenfield* assumption — fine when the adopter controls repo settings.

Many real repos are **brownfield**: labels/fields/board are fixed by org policy or permission, hundreds of issues already follow a native convention, and the team *cannot create labels at all*. The grounding case is `agentic-user-journey` (AUJ): no `type:*`/`workstream:*`/`state:*` labels and none creatable; priority via native `P0`/`P1`/`P2`; type via `[Task]`/`[Epic]` title-prefix; state implicit (open/closed + a `Blocked` label, no board); a **flat** tracker with no machine-checkable parent-refs. Vendored there, the capability is unusable end-to-end — `pre-check` fails, `create-issue` refuses a parent the repo can't express. The design input is the `pm-brownfield-adoption` scratchpad note; its operational-friction companion is the `pm-capability-dogfooding-findings` note.

The need: the capability should **adapt to an existing, immutable substrate** — remap each conceptual axis onto what exists, degrade feature-by-feature where a substrate is truly absent, and tolerate attributes it does not model — so a brownfield adopter keeps as much of the methodology as the substrate allows instead of being blocked wholesale. **Greenfield stays the default, easy path; brownfield is a supported mode, not a wall.** This record settles the **substrate-axis binding and degradation model**; its Wave-2 Features (EPIC #217) depend on it.

## Decision

**In plain terms:** the capability stops assuming its own labels are the substrate. An adopter may declare, per conceptual axis (type, priority, workstream, lifecycle-state, hierarchy), **how that axis is encoded in their existing tracker** — or that it isn't, in which case the feature needing it degrades to advisory rather than refusing. Greenfield (no such declaration) reads exactly as today.

1. **A dedicated `substrate-map.yaml` schema** (per the architect steer — not per-schema `binds_to` fields). Each methodology axis binds to one of: `label: <existing-set>` with a value→value remap (e.g. `priority: {High: P0, Medium: P1, Low: P2}`); `title-prefix`; `derive` (e.g. state from open/closed + a `Blocked` label); or `unsupported` (the feature needing it degrades). Consumers resolve **through** the map; the existing schemas (`classification.yaml`, `issue-types.yaml`, `git-conventions.yaml`) keep their shape, read via this indirection.

2. **Activation is emergent, not a global `mode` flag** (EPIC #217 constraint 4) — and per-axis resolution is **ternary, not binary**. No `substrate-map.yaml` at all → every axis reads the kit's own labels directly = greenfield (byte-unchanged). With a map present, each axis is one of three: **bound** (resolve through the binding); **`unsupported`** (the feature needing it degrades — see point 3); or **absent from the map** → treated as **`unsupported`, NOT greenfield**. This last rule is load-bearing: a brownfield adopter who simply omits an axis they can't serve must get degradation, not a silent fall-back to greenfield refusal on labels they cannot create. One uniform consumer contract (presence of a map flips the default for unlisted axes from greenfield to degrade); no second source of truth that can disagree.

3. **Degradation reuses the existing severity vocabulary** ([project-management:DEC-014-validation-severity-model]), not a new parallel mechanism — *where a severity knob exists to flip*. When an axis is `unsupported`, the rules needing its substrate soften through DEC-014 tokens via the existing per-axis `*_severity` fields ([project-management:DEC-012] / [project-management:DEC-019]): a substrate-absent `hard-reject` becomes `warning` (advisory); a *cheap, substrate-free* rule (e.g. a declared title-prefix) stays as authored. **Honest gap:** not every hard-reject has a `*_severity` field today — the containment invariants (`issue-types.yaml`) and the parent-ref hard-rejects (`body-format.yaml`) are authored in prose with no schema knob. Where a rule must be softenable but has no severity field, **adding that field is in-scope schema work** (Implications), not a claim the model already expresses it. Either way `pre-check` **degrades, not refuses**: it reports a capability matrix (served / advisory / disabled) and runs the rest.

4. **Load-bearing vs degradable, settled per axis:**
   - **priority / workstream / type** — *bindable*; serve via the adopter's labels / title-prefix, or `unsupported` (classification goes partial, advisory).
   - **lifecycle-state** — *degradable, via a detection-predicate remap* (see the dedicated paragraph below — this is the subtlest reconciliation and the one with a real state-set consequence).
   - **hierarchy — two distinct rules, degraded differently.** Relax only the **parent-requiredness** rules (`parent_ref_optional` per type + the `body-format.yaml` parent-ref hard-rejects): a flat repo files children with a **body-text-only, ungated** parent-ref, so `create-issue` never demands a parent the repo can't express. The **containment invariants stay hard** (Feature-in-Feature, EPIC-in-EPIC, etc. remain rejected) — a flat repo can still malform a nesting, and EPIC #217 constraint 3 (native conventions read-only / degrade only what the substrate can't express) does not license loosening a rule whose substrate *is* present. Softening the parent-requiredness rules requires a severity field on those entries (the point-3 gap).
   - **extras** — *tolerated*; unknown labels/attributes pass through untouched, never clobbered, never tripping validation.

### The lifecycle-state remap (detection-predicate, with a state-set consequence)

Lifecycle is a **detection-predicate** remap, not a label remap — faithful to [project-management:DEC-033-rebind-issue-lifecycle-onto-process-substrate], whose engine resolves position from a predicate over reality (first matching detection wins). Brownfield supplies a predicate over **open/closed + a blocked label** instead of over `state:*`. The engine contract is untouched — it never cared whether the predicate read a `state:*` label or open/closed.

The real consequence the binding must own: a derive-from-open/closed predicate resolves a **reduced state set** — it cannot distinguish the greenfield "open" states (Todo / Backlog / In-progress all read as `open`). So brownfield **collapses** those to a single open-ish state (plus `blocked` from the label, plus `done` from `closed`); transitions *between* collapsed states degrade to **no-ops** (not errors). [project-management:DEC-034-cascade-slot-binding]'s closure cascade reads this derived state: `done` derives from `closed`, and since GitHub `closed` doesn't distinguish completed from won't-do, DEC-034's "won't-do counts toward closure" is automatically satisfied (both are `closed`) — the fold still reads a well-defined terminal. The binding declares which states survive the collapse.

5. **Never write an unmanaged label** (EPIC #217 constraint 1). In brownfield mode the capability never creates / renames / deletes any label or field the adopter owns; it only reads and maps. `bootstrap`'s additive label-creation is suppressed per-axis wherever a binding exists. The adopter's native conventions are **authoritative and read-only** (constraint 3) — methodology validation degrades to advisory where the substrate can't express a rule, while [project-management:DEC-003-github-bound-substrate]'s GitHub-as-the-bound-tracker binding stays intact.

### Relationship to DEC-003 (partial supersession)

This **partially supersedes [project-management:DEC-003-github-bound-substrate]**. DEC-003's anti-mapping stance had two legs: (a) *cross-tracker mapping notes decay* (the rejected "GitHub-bound with Linear/Jira mapping notes" alternative), and (b) *concrete-is-sharper — no indirection layer*. This record overturns **only leg (b)** (a brownfield adopter resolves axes *through* `substrate-map.yaml`) and **does not reopen leg (a)**: the remap is **intra-GitHub** (GitHub labels → other GitHub labels, title-prefixes, open/closed state), not a tracker-neutral abstraction, and it is grounded in a real adopter (AUJ) rather than speculative — so DEC-003's actual fear (a decaying cross-tracker translation layer) is *not* reintroduced. What is **preserved**: GitHub remains the only bound tracker; Issues / sub-issues / Milestones / Projects v2 / GraphQL remain the substrate vocabulary. DEC-003 stays `accepted` and receives a partial-supersession *note* in the same change-set (the lighter reciprocal-note gesture, per the [project-management:DEC-034]↔DEC-033 precedent — not a `superseded` status flip). Moving this record `proposed → accepted` requires explicit maintainer sign-off on the DEC-003 supersession.

### Scope

This record settles the **label/axis substrate** axis only. The companion *issue-less PR* rigidity (every PR must close a numbered issue — decision-acceptance / scratchpad / docs-only PRs close none) is the **same class** of greenfield assumption but is tracked separately as **EPIC #218**; cross-referenced, not folded in here.

## Rationale

Refusal-wholesale is the wrong default for an immutable substrate: the methodology's *concepts* (type, priority, workstream, state, hierarchy) are separable from the *substrate* that encodes them (labels, prefixes, board columns, body refs), so binding each concept independently is what lets an adopter keep the served axes while only the truly-unsupported ones degrade. A dedicated `substrate-map.yaml` (over per-schema `binds_to` fields) keeps the indirection in one auditable place and leaves each schema's own shape unpolluted — the architect steer. Emergent activation (over a `mode` flag) gives one consumer contract: there is no second switch that can disagree with the presence of a binding. Reusing DEC-014's severity tokens for degradation avoids a parallel softening mechanism — "soften when the substrate is absent" is exactly a severity decision the model already expresses.

**On generalisation (the carrier discipline).** This is a pm-capability decision and stays pm-local. The "remap conceptual axes onto an immutable substrate / greenfield-default + brownfield-adopt-existing" shape *looks* general, but the **governing precedent is [project-management:DEC-022-methodology-mesh]'s "capability-local at v1, COR-promote on the second instance"**: defer naming until a second capability binds the same shape. So this record does **not** name a generic two-mode-bootstrap shape; it binds pm's instance narrowly and flags the generalisation as **COR-promotable once living-docs (EPIC #234) is the second instance**. The COR-033↔DEC-033 *name-broad-at-core / bind-narrow-in-capability* pattern is the **complementary** discipline that applies *when the shape is eventually named*: it lands at core (a COR), never in a capability DEC. Naming it here would both jump DEC-022's defer-until-second-instance order and mis-place the carrier (over-committing pm to a shape the docs instance may diverge from).

### Alternatives considered

- **Per-schema `binds_to` fields** instead of a dedicated `substrate-map.yaml`. Rejected (architect steer) — scatters the indirection across every schema and entangles each schema's shape with adopter-substrate concerns; one dedicated map is auditable and keeps schemas clean.
- **A global `mode: brownfield` flag.** Rejected — a second source of truth that can disagree with the actual presence of bindings; emergent activation is one contract.
- **A new parallel "degraded severity" mechanism.** Rejected — DEC-014 tokens + the existing `*_severity` fields express "soften this rule" where a knob exists; the fix for rules lacking one is to *add* the field (in-scope schema work), not invent a parallel vocabulary.
- **Name the generic two-mode-bootstrap shape here.** Rejected — jumps DEC-022's defer-until-second-instance order and mis-places the carrier (a cross-capability shape belongs at core); defer to a COR extracted once living-docs is the second instance.
- **Keep refusing wholesale (greenfield-only).** Rejected — blocks real brownfield adopters (AUJ) from any of the methodology; the whole point is graceful partial adoption.
- **Hand-authored map only (no scaffolding).** The `adopt-existing` inventory tool — scan the live repo's labels/prefixes and *scaffold a draft* `substrate-map.yaml` for the adopter to confirm (the brownfield analogue of `bootstrap`, scratchpad candidate 3) — is **deferred to a Wave-2 Feature**, not settled here. v1 is a hand-authored map; this record settles the map's *binding model*, not the *authoring UX*.

## Implications

- **New `substrate-map.yaml` schema** + a read-path indirection the axis consumers resolve through; greenfield (no map) is byte-unchanged. Trunk Wave-2 Feature (untestable in isolation — lands with its first consumer).
- **Severity-field coverage is schema work, not just reuse.** The rules brownfield must be able to soften but that carry **no severity field today** — the `issue-types.yaml` containment invariants and the `body-format.yaml` parent-ref hard-rejects — need a per-rule `*_severity` field *added* before the DEC-014 mechanism can flip them. (Note: the containment invariants stay hard regardless — see D4 hierarchy; the field is added for the *parent-requiredness* rules that do degrade.)
- **`pre-check` degrades** to a capability matrix (served / advisory / disabled) rather than hard-failing; **`create-issue`/`move-issue`** tolerate flat hierarchy + derived (reduced) state; a **write-side guard** enforces "never write an unmanaged label" across every mutating script — and the read-path **fail-closes** (an omitted/`unsupported` axis degrades; it never falls through to writing the kit's own label). The four EPIC-#217 constraints are carried in.
- **No `workflow.yaml` schema change.** The lifecycle remap is a *detector*-swap (a new pm detector command + a `substrate-map.yaml` binding), not a `workflow.yaml` `schema_version` bump — so it must NOT re-trigger DEC-033/DEC-034's warn-on-override migration. Confirm at impl.
- **DEC-003 partial-supersession note** lands in the same change-set; **DEC-033/DEC-034** reconciled (lifecycle = detection-predicate remap → reduced state set; cascade reads derived state). **Acceptance requires maintainer sign-off on the DEC-003 supersession.**
- **ADR follow-up:** the read-path contract over schema consumers (how the indirection resolves; the fail-closed posture that must never fall through to a greenfield write) is capability-architecture worth an ADR at implementation time — the architect offered to author it once this DEC settles.
- **Generalisation deferred:** the shared greenfield+brownfield bootstrap shape is COR-promotable on the **second** instance (living-docs #234), extracted *from both* into a new core COR — not named here (DEC-022's defer-until-second-instance order).
- **Scratchpad retirement on acceptance:** `pkit scratchpad done pm-brownfield-adoption --produced DEC-036`.
- Relationships to records are stated inline above and in the *Relationship to DEC-003* subsection; companion to EPIC #218 (issue-less PR mode).
