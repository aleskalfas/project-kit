## Tool hygiene

1. **Run `pkit` from the project root.** The dispatcher resolves the project root via `git rev-parse --show-toplevel` (with a CWD-walk fallback); staying at root avoids ambiguity in path resolution. Specific to pkit's CWD-resolution behaviour, so this rule lives here rather than in core.

2. **Do not disable the sandbox for allowlisted `gh` / `git` / `pkit` commands.** The default sandbox already permits the network egress these commands need — GitHub over HTTPS (`github.com`, `api.github.com`) — and writes under `/tmp`. Verified empirically: inside the sandbox with no override, `gh` reads, the network-touching `pkit project-management` scripts, `git ls-remote` and `push` over the HTTPS transport, `/tmp` body-file writes, and a real `create-issue` mutation all succeed (host-based egress does not distinguish reads from writes). Meanwhile `.claude/settings.json` auto-approves `Bash(gh:*)`, `Bash(git:*)`, `Bash(pkit:*)`, `Bash(uv:*)`, `Edit`, `Read`, and `Write` — silently, no prompt. Attaching the harness's "disable sandbox" flag to such a call forces a manual operator confirmation *by design* and overrides that auto-approval; doing it reflexively on routine work produced roughly forty needless confirmations in a single session. So: leave the sandbox on. Reserve the disable-sandbox flag for the rare command that genuinely must escape it, not as a default for anything touching the network. This is tied to project-kit's sandbox configuration and the Claude Code harness, so it lives here rather than in the universal `core.md` (per COR-014's universal-applicability split).
