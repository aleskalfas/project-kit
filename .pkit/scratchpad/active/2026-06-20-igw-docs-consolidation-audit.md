# IGW docs consolidation — audit log (spike golden baseline)

Companion to `2026-06-20-documents-registry-and-onboarding.md`. A per-change record of the **manual** docs-consolidation spike on IGW's `spike/docs-consolidation` branch. Three roles: (1) **complete diagnosis** (what was wrong, nothing forgotten); (2) the **golden baseline** the crystallised tooling's from-fresh re-run is diffed against; (3) the de-facto **spec for the tool's audit report**.

Each entry: id · docs touched · convention rule violated (1–6 from the design note) · what was wrong · what was done · resulting registry entry (`path` + `intent` + `audience` + `status` + `canonical_owner` + `provenance`).

Convention rules (from the design note): **R1** one canonical owner per topic · **R2** registered with intent+audience+status · **R3** living+currency-gated · **R4** audience-led taxonomy · **R5** product ≠ internal notes · **R6** provenance marked.

## Summary (spike pass 1 complete — 2026-06-20/21)

**Failure types seen → convention rule motivated:**

| Failure type | Count (changes) | Rule it motivates |
|---|---|---|
| Same topic split across docs / verbatim re-copy of an owner's content | C1, C2, C4a | **R1** one canonical owner per topic |
| Stale / contradictory / self-inconsistent content (drift) | C3, C4a, C4b, C5a–d | **R3** living + currency-gated |
| Contributor note inside the product docs set | C6 | **R5** product ≠ internal |

(No R2/R4/R6 violations surfaced in this set — the doc set was already well-indexed via `docs/README.md`, which is why the work was de-dup + de-stale, not re-taxonomy. The registry/intent (R2), audience taxonomy (R4), and provenance (R6) are being *captured* here for the first time rather than *repaired*.)

**Moves applied:** C1 (merge SIMPLIFIED → canonical variant; 8 links repointed), C2 (INTEGRATION ↔ MCP_TOOLS de-dup), C3 (tool-name drift, with false-positive caught), C4a/b ("four modes" + `panes:` contradiction, resolved against source), C5a–d (SMOKE_TEST Path collision + version pins; CHANGELOG release-history; WORKFLOW changelog line), C6 (mermaid-style relocation).

**Three plan assumptions were wrong — the audit caught each (this is the headline finding for the tool):**
1. `mcp__igw__* → mcp__operator-*` was *mostly a false positive* — double-underscore refs are valid standalone-server names; the real drift was the informal single-underscore notation (C3).
2. mermaid-style "cites non-existent lint tooling" — `scripts/lint-mermaid.py` **exists** (C6).
3. The CHANGELOG `0.3.1` content was placed by **archaeology** (`git tag --contains` proved it shipped in 0.4.0), not by the plan's implied dating (C5c).
→ **Tool implication:** the consolidation tool must *propose* and *cite evidence*, never auto-apply blind renames/dates. The human-review gate is load-bearing. Two flagged-for-author residuals: the missing CHANGELOG `0.3.0` section, and SMOKE_TEST's frozen version-pinned sample outputs (a "rendered-example" currency convention candidate).

**Registry shape learned (which fields proved load-bearing):**
- `path`, `intent`, `canonical_owner` — load-bearing for **R1** (the owner field is what lets a doc legitimately *defer* a topic to another, e.g. INTEGRATION → MCP_TOOLS; "defers X → Y" emerged as a needed sub-field).
- `status` — load-bearing for **R3**; needs richer values than current/retired (e.g. "current except <gap>", "not-yet-revalidated") — staleness is rarely binary.
- `audience` — load-bearing for **R5** (the contributor/internal vs product distinction *is* the relocation trigger).
- `provenance` — lightly used here (all hand-authored); will matter once generated docs (`docs/cli/`) enter the registry.
- **New field discovered:** `defers_to` (topic → owning doc) — the cross-link that makes R1 enforceable without merging everything into one file.
